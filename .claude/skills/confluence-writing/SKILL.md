# Confluence Writing Skill

사내 Confluence(ljindong.atlassian.net) 의 TECH 스페이스 페이지를 작성·수정할 때 적용하는 **글쓰기 톤·구조·서식 규약**.
콘텐츠는 매번 다르지만 **톤앤매너와 구조는 고정**한다.
이 문서는 콘텐츠 구성 가이드가 아니라 **스타일 시스템 규약**이다.

---

## 1. 글쓰기 철학

- **독자는 개발자다.** 기술 용어를 풀어쓰지 않고 자연스럽게 사용한다. CLI 명령, 파일 경로, 환경 변수, API 호출 형식, 콜백 함수, 비동기 처리 같은 용어를 일일이 정의하지 않는다. 글로 길게 설명하기보다 코드 한 줄·명령어 한 줄로 보여주는 것이 더 명확할 때는 코드를 우선 보여준다. 동작 원리·트레이드오프·한계·엣지 케이스를 정직하게 다루며, "쉽다", "강력하다" 같은 마케팅 톤을 사용하지 않는다.
- **분야 적응 (Domain Adaptation)**: 페이지 주제 분야를 먼저 판단하고, 그 분야에 맞는 예시·비유·코드 샘플을 사용한다. 임베디드 C/C++ 가이드면 차량용/AUTOSAR/MISRA 비유, 웹 프론트엔드면 React/CSS 예시, Python 자동화면 데이터 처리 예시, 인프라면 컨테이너·네트워크 예시. 분야가 다른데 차량용 비유를 끼워 넣지 않는다 (자세한 분야별 가이드는 13장 참조).
- **결론 먼저, 근거는 나중에** (PREP 패턴). 첫 줄에 답을 주고 이어서 풀어 쓴다.
- **개념 → 비유 → 시각화 → 표 → 결론** 5단계 구조를 기본으로 한다. 추상 개념은 그 분야 독자가 친숙한 비유로 한 번 더 설명하고, 가능하면 다이어그램으로 시각화한다 (ASCII 코드블록 즉시 포함 또는 SVG 파일 첨부 자리 표시).
- **시각화는 1순위 표현 수단이다.** 시스템 구조 / 연결성 / 흐름 / 계층 / 데이터 관계가 등장하는 모든 페이지에는 다이어그램을 적극 사용한다. 자동 가능한 시각화(ASCII 다이어그램, 매트릭스 표)와 수동 마무리가 필요한 시각화(SVG 파일 첨부)를 분담해 사용한다 (자세한 워크플로우는 7장 참조).
- **2레벨 트리 구조**: 모든 가이드는 "대표 페이지 1개 + 하위 페이지 N개" 의 2레벨까지를 기본으로 사용한다. **사용자 요청 시에 한해 3레벨(Lv4 손자) 허용** — Claude 가 임의 결정 금지.
- **Part 그룹화 사용 안 함**: "Part 1 / Part 2 / Part 3" 같은 중간 묶음 페이지를 만들지 않는다. 목차의 각 항목이 곧 하나의 하위 페이지로 1:1 매핑된다.
- **페이지 간 연결 최소화**: 내비게이션 요소("다음 단계", "이전/다음", "대표로 돌아가기"), "이 페이지의 위치" 안내, 형제·부모 하드 링크를 **기본적으로 넣지 않는다.** 자식 트리 탐색은 Confluence 사이드바에 맡기고 본문은 자기완결적 내용에 집중한다. 하드 링크는 사용자가 명시 요청할 때만 (자세한 정책은 §3.5 참조).
- **본문 헤더는 H3부터 시작**: H1은 Confluence가 title을 자동 렌더링하므로 중복 방지, H2는 본문에 시각적으로 너무 크다. 본문의 가장 큰 섹션 헤더는 H3, 그 아래는 H4, H5 (자세한 가이드는 4.5 참조).
- **수동적 안내 X, 능동적 추적 O**. 사용자가 같은 의문을 다시 가지지 않도록 깊이 들어간다.
- **솔직함**. 모르거나 부정확하면 그렇다고 적는다 ("부분적으로 맞고 부분적으로 더 신중해야 한다" 같은 표현 권장).
- **분야별 책임 의식**: 임베디드/차량용은 보안·컴플라이언스·계약, 웹/백엔드는 보안 취약점·확장성, 데이터/AI는 데이터 프라이버시·모델 신뢰성 — 각 분야의 책임 영역을 항상 의식한다.

## 2. 페이지 구조 템플릿

### 2.1 ⭐ 전체 트리 구조 (2레벨까지)

```
[Level 1] 대표 페이지 (목차 페이지)
   │
   ├── [Level 2] 하위 페이지 1  ← 목차 항목 1
   ├── [Level 2] 하위 페이지 2  ← 목차 항목 2
   ├── [Level 2] 하위 페이지 3  ← 목차 항목 3
   └── [Level 2] 하위 페이지 N  ← 목차 항목 N
```

**규칙**:
- 기본은 Level 3 이상 금지. 더 세분화가 필요하면 같은 Level 2에 형제로 배치한다. **사용자 요청 시에 한해 Lv4 허용** — §6.3.1 참조.
- Part / Section / Group 같은 중간 묶음 페이지 사용 안 함. 대표 → 하위 직접 연결.
- 하위 페이지는 5~12개 권장. 13개 이상이면 가이드를 두 개로 분리하거나 항목을 통합한다.

### 2.2 ⭐ 대표 페이지(목차) 표준 구조

대표 페이지는 항상 **3블록 구조**:

```
[블록 A: 간단한 개념]
  - 이 가이드가 무엇인지 1~2 문단
  - 분야에 맞는 비유 1회 권장

[블록 B: 시스템 다이어그램]
  - 전체 시스템 구조 / 연결성 / 흐름을 한눈에
  - 형식: ASCII 다이어그램 (즉시 포함) 또는 SVG 파일 첨부 자리 표시 (사용자 수동 드래그앤드롭)
  - 색상 의미 / 화살표 의미 / 핵심 통찰 블록 동반

[블록 C: 하위 페이지 목차]
  - 각 항목이 무엇을 다루는지 설명하는 표 (제목·요약)
  - 학습 순서 / 우선순위 표 (선택)
  - 꼭 읽어야 하는 항목 ⭐ 표시
  - 페이지 간 하드 링크는 기본적으로 걸지 않는다 (§3.5 연결 정책)
```

순서 고정 (개념 → 다이어그램 → 목차).

### 2.3 하위 페이지 표준 구조

각 하위 페이지(목차의 한 항목)도 자체 완결 구조:

```
[리드 문단]              ─ 이 페이지가 무엇을 다루는지 1~2줄
[다이어그램]             ─ 가능하면. 이 항목 영역의 시각화 (ASCII 또는 SVG 파일 첨부)
[> ⚠️ blockquote]       ─ 분야별 사전 경고 (해당 시)

---                     ─ HR 구분선

[본문 ─ 의미 단위 H3/H4]
```

하위 페이지는 그 자체로 한 편의 완결된 글이어야 한다. 대표 페이지로 다시 가지 않아도 그 항목을 이해할 수 있어야 한다.

> 📌 **연결·내비게이션 안내 (기본 미사용)**
>
> 본 스킬은 **페이지 간 내비게이션 요소를 기본적으로 넣지 않는다.** "대표 페이지로 돌아가기", "이전/다음 페이지", "다음 단계 →" 같은 안내 블록, "이 항목의 위치 / 이 페이지의 위치" 안내, 형제 페이지 하드 링크는 작성하지 않는다. "전체 구조는 대표 페이지 참조" 같은 위치 안내 한 줄도 기본 생략한다. 자식 트리는 Confluence 좌측 사이드바가 자동으로 보여주므로 본문에서 중복할 필요가 없다. 하드 링크는 **사용자가 명시적으로 요청할 때만** 건다. 자세한 정책은 §3.5 참조.

### 2.4 표준 페이지 골격 (Markdown 흐름)

```
[제목]                  ─ Confluence가 title 파라미터로 자동 H1 렌더링
                          본문에서는 H1·H2를 쓰지 않음 ⚠️

[리드 문단]              ─ 일반 단락 또는 *...* 강조 단락부터 시작
[다이어그램]             ─ ASCII 코드블록 또는 SVG 파일 첨부 자리 표시
[> 📌 blockquote]       ─ 추가 자료 / 참고 페이지 링크 (panel-info 대체)
[> ⚠️ blockquote]       ─ 분야별 사전 경고 (panel-warning 대체, 해당 시)
[이 페이지의 목차]       ─ 대표 페이지일 경우 필수

---                     ─ HR 구분선

### 1. ... / 2. ...      ─ H3 (본문의 가장 큰 헤더, 시작점)

#### 1.1 / 1.2 ...       ─ H4 (절 제목)

##### A / B / C          ─ H5 (소절)

[본문]
[표]
[코드 블록]
[blockquote 박스 강조]
```

### 2.5 페이지 종류별 구조

| 종류 | 위치 | 권장 구조 | 다이어그램 |
| --- | --- | --- | --- |
| **대표 페이지** | Level 1 | 개념 → 다이어그램 → 하위 페이지 목차 | **필수** ⭐ |
| **개념 설명 하위 페이지** | Level 2 | 리드 → 다이어그램 → 본문 | **권장** ⭐ |
| **Q&A 형식 하위 페이지** | Level 2 | 리드 → 다이어그램(가능 시) → Q1~Qn → 추적 흐름 요약 | 권장 |
| **튜토리얼/설치 하위 페이지** | Level 2 | 리드 → 사전 확인 → 단계 흐름도 → 트러블슈팅 | 권장 |
| **시스템 구성도 하위 페이지** | Level 2 | 다이어그램 → 색상/화살표 의미 → 핵심 통찰 | **주인공** |

## 3. 톤앤매너 규약

### 3.1 기본 어투

- **존댓말 + 평어 혼용**. 설명문은 평어("~다", "~한다"), 안내·요약은 존댓말("~합니다", "~하세요").
- **단정적 표현 권장**. "~할 수도 있을 것 같습니다" 보다 "~합니다" / "~하세요". 모호한 표현은 약함의 신호.
- **불확실한 부분은 명시적으로**. "정확히는 ~", "부분적으로 ~", "다만 ~" 같은 단서를 명확히.
- **2인칭 "사용자님"** 사용. "당신"이나 "유저"는 쓰지 않음.
- **이모지는 제목 강조용으로만**. 본문에 이모지 남발하지 않음. 허용되는 위치: H3 앞 (📌, ⭐, 🎯), panel 내 키워드(⭐, ⚠️, 💡).

### 3.2 개발자 독자 전제

본 스킬에서 작성하는 모든 문서의 독자는 개발자다. 다음 규칙을 따른다:

- **기술 용어를 풀어쓰지 않는다**: "API 엔드포인트", "콜백", "비동기 처리", "레지스터 매핑", "디펜던시 인젝션" 같은 용어를 자연스럽게 사용. 어린이 책 어조 ❌
- **코드가 글보다 명확할 때는 코드를 먼저** 보여준다. 동작을 설명하기 전에 코드 한 줄로 보여주고, 그 아래에서 풀어 쓴다.
- **CLI 명령·환경 변수·파일 경로**를 자유롭게 인용. `$HOME/.config/`, `NODE_ENV=production`, `git rebase -i HEAD~3` 같은 표현 그대로 사용.
- **버전·플랫폼 의존성 명시**: "Python 3.10+ 필요", "POSIX 환경 가정", "AUTOSAR 4.4 기준".
- **동작 원리 / 내부 구조까지** 들어간다. 단순 사용법만 다루지 않고 "왜 이렇게 동작하는지" 설명한다.
- **트레이드오프·한계·엣지 케이스 명시**: 개발자는 의사결정에 이 정보가 필요하다.
- **검증·디버깅·로깅 관점 포함**: "이게 안 될 때 어떻게 추적하나" 가 중요한 가치.
- **에러 메시지·스택 트레이스·로그 포맷**은 그대로 인용.

피해야 할 톤 (일반 사용자용 어조):

```
❌ "패키지 매니저는 프로그램을 쉽게 설치하고 관리할 수 있게 해주는 편리한 도구입니다."
❌ "먼저 ~란 무엇인지부터 알아볼까요?"
❌ "걱정 마세요, 어렵지 않습니다."
❌ "이 강력한 기능을 활용하면..."
```

권장 톤 (개발자용):

```
⭕ "npm 의존성 격리는 프로젝트 단위로 동작한다 (워크스페이스 모드 제외). 
   peer dependency 충돌 시 npm 7+ 부터는 자동 설치, 6 이하는 수동 해결."
⭕ "이 함수는 O(n log n). n=10^6 부근에서 GC pressure가 두드러지므로 
   배치 처리 권장."
```

### 3.3 전문 용어 처리

- **첫 등장 시 짧게 풀이가 필요한 경우만** 풀이: "MISRA-C(자동차 임베디드 C 코딩 표준)". 단, 그 분야 개발자에게 자명한 용어는 풀이하지 않는다 (예: 웹 개발자에게 "HTTP", 백엔드 개발자에게 "ORM" 풀이 불필요).
- **두 번째부터는 약어만**.
- **혼용 금지**: 한 페이지 내에서 같은 개념의 다른 표기를 섞지 않음.

### 3.4 분야별 비유 사용 규칙

- **분야가 임베디드 C/C++ 인 경우**: 차량용 비유(ECU, BSW, ARXML, CAN 통신, AUTOSAR 레이어) 1순위.
- **분야가 웹 프론트엔드 인 경우**: 컴포넌트·렌더링·상태 관리 비유 (예: "Redux store는 글로벌 단일 진실 공급원", "VDOM diff는 변경 감지 알고리즘").
- **분야가 백엔드 / API 인 경우**: 요청-응답·미들웨어·트랜잭션 비유.
- **분야가 Python 자동화 / 데이터 인 경우**: 파이프라인·스트리밍·배치 비유.
- **분야가 인프라 / DevOps 인 경우**: 컨테이너·오케스트레이션·네트워크 토폴로지 비유.
- **분야가 AI / ML 인 경우**: 모델·임베딩·파인튜닝·추론 비유.

비유 블록은 다음 형식으로 표시:

```
### 비유

[추상 개념 A]    = [그 분야의 친숙한 매핑 A]
[추상 개념 B]    = [그 분야의 친숙한 매핑 B]
[추상 개념 C]    = [그 분야의 친숙한 매핑 C]

→ [매핑이 어떻게 의미를 만드는지 한 줄 요약]
```

분야 판단 기준은 13장 참조.

### 3.5 페이지 간 연결·내비게이션 정책

본 스킬은 **페이지 간 연결을 최소화**한다. 과거 기본값(대표↔하위 링크 + "다음 단계" 안내)은 자식이 늘수록 교차 참조가 촘촘해져 번호·구조 변경 시 본문 수정 부담을 키웠다. 그래서 다음을 **기본 규칙**으로 한다.

| 요소 | 기본 방침 | 비고 |
| --- | --- | --- |
| **"다음 단계 / 다음 페이지" 안내 블록** | ❌ 넣지 않음 | 페이지 끝 학습 흐름 안내 자체를 작성하지 않음 |
| **"이전 / 다음 / 대표로 돌아가기" 내비게이션** | ❌ 넣지 않음 | 위치 안내 blockquote·섹션도 생략 |
| **"이 항목의 위치 / 이 페이지의 위치" 안내** | ❌ 넣지 않음 | 현재 페이지가 트리 어디에 있는지 알리는 안내 자체를 작성하지 않음 |
| **형제·부모 페이지 하드 링크** | ❌ 기본 미사용 | 사용자가 명시적으로 요청할 때만 건다 |
| **페이지 위치 안내 (텍스트)** | ❌ 넣지 않음 | "전체 구조는 대표 페이지 참조" 같은 한 줄도 기본 생략 |
| **하위 페이지 목록** | 사이드바에 위임 | 본문에 목록을 중복 생성하지 않음. 대표 페이지에는 "각 항목이 무엇을 다루는지" 설명 표만 (링크 없이) |
| **본문 내 형제 주제 언급** | △ 링크 아닌 텍스트 | "8.3에서 다룬다" 처럼 번호·제목만 텍스트로, `<a>` 링크는 걸지 않음 |

> 💡 **핵심**
>
> 자식 트리 탐색은 Confluence 좌측 사이드바가 자동으로 해준다. 본문은 **자기완결적 내용**에 집중하고, 내비게이션은 도구(사이드바)에 맡긴다. 링크를 거는 순간 그 페이지는 대상 페이지의 위치·번호 변경에 종속되므로, 링크를 줄이는 것이 곧 유지보수 부담을 줄이는 것이다.

> 📌 **예외**
>
> 사용자가 "이 페이지들을 링크로 연결해줘" 라고 명시적으로 요청하면 그때만 하드 링크를 건다. 그 경우에도 양방향 자동 연결(대표↔하위 모두)이 아니라 사용자가 지정한 방향만 건다.

## 4. Confluence Markdown 서식 규약

### 4.0 ⚠️⚠️⚠️ contentFormat — 신규 작성은 "markdown", 첨부 보존 수정은 "html"

Atlassian MCP (`mcp.atlassian.com/v1/mcp`) 의 `createConfluencePage` / `updateConfluencePage` / `getConfluencePage` 호출 시 가장 먼저 정할 것이 `contentFormat` 이다. **작업 종류에 따라 다르게 쓴다.**

```
contentFormat 의 enum: markdown / adf / html

[신규 페이지 작성 — createConfluencePage]
  ⭕ "markdown"   ← 기본값. 본문을 GFM 으로 작성하면 서버가 ADF 로 변환·렌더링
  △ "html"       ← 동작하지만 신규 작성에는 markdown 이 더 간결
  ❌ "adf"        ← JSON 구조체 (Claude가 손으로 작성하기 비현실적)
  ❌ 생략         ← default 가 ADF 로 해석돼 HTML/markdown body 보내면 파싱 실패 가능

[첨부(SVG·이미지) 있는 페이지 수정 — getConfluencePage → updateConfluencePage]
  ⭐ "html"       ← 반드시 이것. storage XHTML 을 받아 첨부 참조를 보존하며 수정 (§8.3.3)
  ❌ "markdown"   ← 받으면 <figure> media 참조가 텍스트화/소실 → update 시 첨부 사라짐
  ❌ "adf"        ← JSON 노드 트리라 손으로 텍스트 치환 비현실적
```

> ⚠️ **2026년 5월 재검증 — 과거 기록 정정**
>
> 본 스킬 초판은 "`html` 은 enum 위반으로 거부된다"고 기록했으나, **이후 재검증 결과 `getConfluencePage` / `updateConfluencePage` 모두 `contentFormat: "html"` 이 정상 동작**한다 (사내 ljindong 인스턴스, 2026-05 확인). MCP 서버 버전업으로 동작이 바뀌었거나 초판 검증이 부정확했던 것으로 보인다. 따라서 "html 무조건 금지"는 더 이상 맞지 않으며, **첨부 보존 수정에서는 html 이 유일하게 올바른 경로**다. 다만 신규 작성은 여전히 markdown 이 더 간결하므로 기본값으로 둔다.

**결론**:
- **신규 페이지 작성** → `contentFormat: "markdown"`, body 는 CommonMark + GFM.
- **첨부가 있는 기존 페이지 수정** → `contentFormat: "html"`, storage XHTML 을 fetch 해서 그 위에서 텍스트만 교체 (§8.3.3 워크플로우 필수).
- **첨부가 없는 단순 페이지 수정** → markdown / html 어느 쪽이든 가능하나, fetch 한 형식과 send 형식을 일치시킨다.

**markdown 본문에 HTML 인라인 시도 금지**: `contentFormat: "markdown"` 으로 보내는 body 안에 `<div data-type="panel-info">` 같은 태그를 인라인으로 넣으면 보존되지 않는다 — escape 되어 `&lt;div&gt;` 로 노출될 수 있다. markdown 모드에서 표현 불가한 ADF 요소(panel, status lozenge, layout, expand 등)는 **markdown 의 다른 수단으로 우회**한다 (4.1 참조). 단, 위 표대로 `contentFormat: "html"` 모드로 보낼 때는 storage XHTML 태그(`<table>`, `<figure data-type="media-single">`, `<ac:*>` 등)를 그대로 쓸 수 있다.

### 4.1 박스 강조 — blockquote + 이모지 헤더로 대체

ADF 의 panel-info / panel-warning / panel-note / panel-success / panel-error 같은 색상 박스는 markdown 에서 직접 표현할 수 없다. 본 스킬은 **blockquote + 이모지 헤더 + 굵은글씨** 조합으로 대체한다. Confluence 에서 blockquote 는 회색 좌측 막대 + 들여쓰기로 렌더링되므로 시각적 강조 효과가 충분하다.

| 기존 ADF panel | markdown 대체 패턴 | 용도 |
| --- | --- | --- |
| `panel-info` (파랑) | `> 📌 **제목**` blockquote | 참고 자료, 추가 정보, 무난한 안내 |
| `panel-warning` (노랑) | `> ⚠️ **주의**` blockquote | 보안·컴플라이언스 주의, 사전 경고 |
| `panel-note` (회색) | `> 💡 **핵심**` blockquote | 핵심 통찰, 꼭 정독해야 하는 강조 |
| `panel-success` (초록) | `> ✅ **권장**` blockquote | 권장 패턴, 성공 사례 |
| `panel-error` (빨강) | `> 🚫 **절대 금지**` blockquote | 절대 금지 사항, 사고 위험 |

**작성 예시**:

```markdown
> 📌 **참고 자료**
>
> 같은 가이드의 형제 페이지 **"[관련 페이지 제목]"** 도 함께 정독하기를 권장합니다.

> ⚠️ **사전에 인지해야 할 것**
>
> - **[분야별 위험 영역 1]**: [구체 설명]
> - **[분야별 위험 영역 2]**: [구체 설명]

> 💡 **이 페이지의 핵심**
>
> [한 문장 결론]

> 🚫 **절대 금지**
>
> [사고 위험 사항]
```

**규칙**:
- 박스 강조 (blockquote) 는 **남용 금지**. 페이지당 총 4개 이내.
- 첫 줄은 항상 `> [이모지] **헤더**` 형식. 본문은 두 번째 줄부터 (빈 `>` 한 줄로 구분).
- blockquote 안에서도 리스트·코드·강조 사용 가능 — 각 줄 앞에 `> ` 만 붙이면 됨.
- 절대 금지 사항이 아닌데 🚫 사용하지 말 것 (panel-error 와 동일 규칙).

### 4.2 표 (table) 작성 규약

기술 가이드의 핵심 정보 전달 도구. **본 스킬에서 가장 자주 쓰는 컴포넌트.** GFM 표 syntax 를 그대로 사용한다.

```markdown
| 항목 | 설명 | 비유 / 매핑 |
| --- | --- | --- |
| **[항목명]** | [설명] | [그 분야의 비유] |
```

**분야별 표 예시**:

임베디드 C/C++:
```markdown
| **PduR 모듈** | PDU 라우팅 담당 | BSW 통신 스택의 중앙 라우터 |
```

웹 프론트엔드:
```markdown
| **useReducer** | 복잡한 상태 로직 | 작은 Redux의 컴포넌트 로컬 버전 |
```

백엔드:
```markdown
| **Connection Pool** | DB 연결 재사용 | 일회용 컵 대신 머그컵 |
```

**규칙**:
- **헤더 행(첫 번째 행)은 모든 셀을 `**...**` 굵게**. 키 컬럼이 굵은 것과 겹쳐도 그대로 둔다 (단순 일관성 우선). 예: `| **항목** | **설명** | **비유 / 매핑** |`.
- **첫 번째 컬럼은 항상 키 항목** (이름, 단계, 카테고리). `**...**` 굵게 강조.
- **3~4 컬럼 권장**. 5 컬럼 이상은 가로 스크롤 발생 → 분할.
- **비유 / 매핑 컬럼**을 추가하면 학습 효과 ↑.
- **상태 표기 ⭕ ❌ ⚠️ ⭐**: 텍스트 대신 시각적 마커로 빠른 스캔 가능하게.
- 표 셀 안에 줄바꿈이 필요하면 `<br>` 사용 가능 (GFM 일부 셀에서 동작) — 안 되면 짧게 끊거나 표를 분할.

| 마커 | 의미 |
| --- | --- |
| ⭕ | 권장, 가능, 적합 |
| ❌ | 금지, 불가, 부적합 |
| ⚠️ | 주의, 신중 |
| ⭐ | 핵심, 강조 |
| ◯ | 선택적 |
| △ | 보류, 조건부 |

### 4.3 상태 표시 (Status Lozenge 대체)

ADF 의 status lozenge (`<span data-type="status" data-color="...">`) 도 markdown 에서 표현 불가. 다음 두 가지 중 택일:

| 기존 ADF | markdown 대체 | 예시 |
| --- | --- | --- |
| green "완료" | 이모지 + 텍스트 | ✅ 완료, ✅ **DONE** |
| red "차단" | 이모지 + 굵게 | 🚫 차단, ❌ **BLOCKED** |
| yellow "진행 중" | 이모지 + 텍스트 | 🟡 진행 중, ⚠️ **WIP** |
| blue "정보" | 이모지 + 텍스트 | 🔵 정보, ℹ️ INFO |
| purple "검토" | 이모지 + 텍스트 | 🟣 검토, 👀 REVIEW |

표 안에서 상태 컬럼이 필요할 때 자주 쓴다.

### 4.5 인라인 강조

| 사용처 | markdown | 예시 |
| --- | --- | --- |
| 핵심 키워드 | `**...**` | **BSW 레이어**, **useReducer 훅** |
| 파일 경로, 식별자 | `` `...` `` | `Com_RxIndication()`, `~/.bashrc` |
| 강조 요점 | `> **결론**: ...` | > **결론**: ~ |
| 부드러운 강조 | `*...*` 또는 `_..._` | *약결합* 구조 |

**굵게 표기 `**...**` 가 그대로 노출될 때 — 원인과 회피**:

본문에서 `**XXXX**` 가 굵게 처리되지 않고 마크다운 기호 그대로 보이는 경우가 간헐적으로 발생한다. 대부분 다음 두 케이스다.

- **`**` 안쪽에 공백이 들어감** (CommonMark spec 위반): `** 비동기 **` 처럼 여는 `**` 직후·닫는 `**` 직전에 공백이 있으면 굵게 토큰으로 인정되지 않는다. 양 끝은 공백 없이 붙여 쓴다.
  - ⭕ `이 함수는 **비동기**로 동작한다.`
  - ❌ `이 함수는 ** 비동기 ** 로 동작한다.`
- **한글·CJK 문자 양옆에 `**` 가 공백 없이 붙음** (Confluence 파서의 word boundary 판정 한계): CommonMark 는 CJK 문자도 단어 문자로 다루지만, Confluence 의 markdown→ADF 변환기는 ASCII 영문 기준으로 boundary 를 판정해 한글 사이에 끼인 `**` 를 토큰으로 못 자르는 경우가 있다. **단어 한가운데를 굵게 표시하지 말고 단어 통째로 굵게 처리**하며, 굵게 직전·직후에 공백·구두점·줄바꿈 중 하나를 둔다.
  - ⭕ `이 **비동기** 함수는 ...`
  - ⭕ `이 함수는 **비동기**다.` (닫는 ** 뒤에 구두점 OK)
  - ❌ `이것은**비동기**함수다.` (앞뒤 한글에 공백 없이 붙음)

추가 회피 규칙:

- `**` 사이에 **빈 줄(단락 경계)** 을 끼우지 않는다. 한 단락 안에서 닫는다. 줄바꿈 한 번도 일부 파서에서 깨뜨리므로 한 줄로 유지 권장.
- 한 줄 안에서 `**` 는 **짝수 개**. 홀수 개면 마지막 `**` 가 그대로 노출된다.
- `**` 안에 백틱·이탤릭(`*...*`, `_..._`) 같은 다른 마크다운을 중첩하지 않는다 (`**굵은 *이탤릭** 닫고*` 처럼 토큰 매칭이 꼬인다). 굵게+코드가 필요하면 굵게는 포기하고 코드만 사용.
- 이전 텍스트에서 복붙 시 백슬래시 이스케이프(`\**`) 가 묻어 들어오지 않았는지 확인.

**금지**:
- 색상 직접 지정 시도 (`<span style="color:red">`) 금지 — markdown 으로 표현 불가하며 인라인 HTML 도 escape 됨.
- 본문 중간에 H1 / H2 임의 사용 금지 (4.6 참조).

### 4.6 ⚠️ 헤딩 위계 — 본문 H1·H2 사용 금지

Confluence 는 페이지 생성·수정 시 받은 `title` 파라미터를 **자동으로 페이지 상단에 H1 으로 렌더링**한다. 따라서:

- **H1 (`# ...`) 금지**: 본문에 또 H1 을 쓰면 같은 제목이 두 번 표시됨.
- **H2 (`## ...`) 금지**: 본문에 H2 는 시각적으로 너무 크다. 본 스킬의 일관 정책으로 사용하지 않음.
- **본문의 가장 큰 헤더는 H3 (`### ...`) 부터 시작**.

```
[헤더 위계 정책 — markdown 표기]
# ...      →   ❌ 본문 사용 금지 (Confluence title 자동 렌더링과 중복)
## ...     →   ❌ 본문 사용 금지 (시각적으로 너무 큼)
### ...    →   ⭕ 본문의 가장 큰 섹션 헤더 (시작점)
#### ...   →   ⭕ 절 단위
##### ...  →   ⭕ 소절 단위
###### ... →   ⭕ 더 깊은 단위 (필요 시, 권장 안 함)
```

**올바른 패턴**:

```markdown
<!-- title 파라미터: "PduR 라우팅 동작 원리" -->

*이 페이지는 PduR 이 어떻게 PDU 를 라우팅하는지 설명합니다.*

### 1. PduR 의 역할

...

#### 1.1 라우팅 테이블 구조

...

### 2. 라우팅 알고리즘

...
```

**잘못된 패턴 (절대 사용하지 말 것)**:

```markdown
# PduR 라우팅 동작 원리   ← ❌ title 과 중복
## 1. PduR 의 역할         ← ❌ 본문에 H2 금지
```

**규칙**:
- 본문 첫 요소는 항상 일반 단락 또는 `*...*` 강조 단락 (리드 문단).
- 본문 안에서 사용 가능한 헤더: **H3, H4, H5, H6** (즉 `###`, `####`, `#####`, `######`).
- 가장 큰 섹션 구분은 **H3 (`###`) 부터** 시작.

이 규칙은 모든 페이지 종류에 예외 없이 적용된다.

### 4.7 markdown 에서 표현 불가한 ADF 요소 — 대체 전략 요약

| 잃는 ADF 요소 | 본 스킬의 markdown 대체 |
| --- | --- |
| panel-info / warning / note / success / error | blockquote (`>`) + 이모지 헤더 + 굵은글씨 (4.1) |
| status lozenge | 이모지 + 굵은글씨 텍스트 (4.3) |
| 2/3 컬럼 레이아웃 (`layout-two-equal`) | 표로 대체 또는 두 개의 H3 섹션으로 분할 |
| expand / collapse (`<details>`) | `<details><summary>` 도 escape 됨. 헤더 + 본문으로 평탄화 |
| 인라인 카드 (`data-card-appearance="inline"`) | 일반 markdown 링크 `[제목](URL)` 로 사용 (호버 시 일부 자동 미리보기) |
| 사용자 멘션 (`<span data-type="mention">`) | 텍스트로 `@이름` 표기 (자동 멘션 변환 안 됨) |
| 날짜 위젯 (`<time datetime="...">`) | 텍스트로 `2026-05-11` 표기 |
| `<svg>` 인라인 | 일관되게 무시됨. ASCII 다이어그램 또는 SVG 파일 첨부 (사용자 드래그앤드롭) — §7 참조 |

이 표는 markdown 작성 중 "이 요소가 안 된다는 걸 깜빡했다" 싶을 때 즉시 참조하는 용도다.

추적형 학습 자료를 작성할 때.

### 5.1 Q&A 페이지 구조

```
[리드]
[> 📌 blockquote: 페이지 가치 한 줄]
[다이어그램 — Q&A가 다루는 영역 시각화]
[Q 흐름 목록]

---

### Q1. [질문]
**답**: [한 줄 결론]
[본문 풀이 — 2~5 문단]
[표 또는 코드 예시]
[비유 또는 핵심 통찰]

---

### Q2. ...
... (반복)

---

### 📋 종합 정리
[Tier 분리 / 권장 운영 정책 / 의사결정 매트릭스]

### 🎯 추적 흐름 요약
Q1.  [한 줄]          → [한 줄 답]
Q2.  [한 줄]          → [한 줄 답]
...

### 💡 이 페이지가 다루지 못한 영역
[향후 보강 예정 항목]
```

### 5.2 Q&A 한 항목의 권장 길이

- **짧은 Q**: 답 한 줄 + 본문 1 문단 + 표 또는 비유 1개
- **중간 Q**: 답 한 줄 + 본문 3~5 문단 + 표 + 비유
- **긴 Q (⭐)**: 답 한 줄 + 본문 5+ 문단 + 표 2개 + Tier 분리 + 비유 + 통찰

### 5.3 추적 흐름 요약 블록

페이지 끝에 모든 Q를 한 줄씩 압축한 흐름 요약을 둔다 — Q&A 페이지의 식별 시그니처.

```
Q1.  [질문 압축]                          → [답 압축]
Q2.  [질문 압축]                          → [답 압축]
...
```

## 6. 페이지 메타데이터 규약

### 6.1 페이지 제목 명명 (분야별 예시)

```
[Level 1: 대표 페이지 — 분야별 예시]

임베디드 C/C++:
  "AUTOSAR BSW 통합 가이드"
  "CAN 통신 스택 학습 노트"
  "MISRA-C 검증 워크플로우"

웹 프론트엔드:
  "React 상태 관리 패턴 가이드"
  "Next.js App Router 마이그레이션"

백엔드 / API:
  "FastAPI 비동기 처리 가이드"
  "GraphQL 스키마 설계 노트"

Python 자동화 / 데이터:
  "ETL 파이프라인 구축 가이드"
  "Pandas 성능 최적화 노트"

인프라 / DevOps:
  "Kubernetes 배포 전략 가이드"
  "Docker 멀티스테이지 빌드 패턴"

→ 간결한 명사구. "가이드" / "정리" / "노트" / "워크플로우" 같은 종결어 권장.

[Level 2: 하위 페이지 — 개념 설명]
"[항목 명]"
→ 한국어 명사구 또는 동명사구.
   "Part 1. ...", "1장. ..." 같은 번호/Part 접두 사용 안 함.

[Level 2: 하위 페이지 — Q&A]
"[주제] — N단계 Q&A"
→ 주제 + em-dash(—) + 부제 (Q&A 페이지 시그니처)

[Level 2: 하위 페이지 — 시스템 구성도]
"[대상] 아키텍처" 또는 "[대상] 구성도"

[Level 2: 하위 페이지 — 튜토리얼/설치]
"[대상] [동작]" — 예: "Tessy 설치 및 초기 설정", "Webpack 설정 마이그레이션"
```

**금지 사항**:
- ❌ "Part 1. ..." 같은 Part 번호 접두
- ❌ "1장. ..." 같은 장 번호 접두
- ❌ Title Case 영문 제목 (한국어 + 필요 시 괄호 영어 부제)

### 6.2 versionMessage (수정 시)

페이지 수정 시 `versionMessage` 에 변경 내용 요약을 남긴다. 후일 추적 가능하게.

분야별 예시:
```
임베디드: "PduR 라우팅 테이블 섹션 보강, MISRA-C 검증 절차 추가"
웹:     "useReducer 패턴 예시 추가, Concurrent Mode 주의사항 명시"
백엔드: "트랜잭션 격리 수준 비교 표 추가"
인프라: "롤백 전략 섹션 신규 추가"
```

### 6.3 부모-자식 관계 (기본 2레벨 트리)

```
[기본 깊이] 2레벨

[Level 1] 대표 페이지
   ├── [Level 2] 하위 페이지 1
   ├── [Level 2] 하위 페이지 2
   ├── [Level 2] 하위 페이지 3
   └── [Level 2] 하위 페이지 N (5~12개)
```

**규칙**:
- 기본은 2레벨까지 사용. 항목별 세분화가 필요하면 같은 Level 2 자리에 형제로 배치.
- Confluence MCP 호출 시 모든 하위 페이지의 `parentId` 는 동일한 대표 페이지 ID로 지정.
- **사용자 요청 시 §6.3.1 의 예외 적용** (Lv4 손자 허용).

### 6.3.1 예외 — 사용자 요청 시 3레벨 허용

**조건**: 사용자가 명시적으로 손자 페이지(Lv4) 가 필요하다고 요청한 경우에만 허용한다. Claude 가 임의로 결정하지 않는다.

명시 요청의 예시:
- "이 하위 페이지는 손자 페이지를 두는 게 좋겠어"
- "3레벨까지 가자"
- "이 항목은 너무 방대해서 손자가 필요해"

```
[예외 깊이] 3레벨

[Level 1] 대표 페이지
   ├── [Level 2] 하위 페이지 1
   ├── [Level 2] 하위 페이지 2  ← 사용자 요청으로 손자 보유
   │      ├── [Level 3] 손자 페이지 2.1
   │      ├── [Level 3] 손자 페이지 2.2
   │      └── [Level 3] 손자 페이지 2.N
   └── [Level 2] 하위 페이지 N
```

손자 페이지를 만들 때:
- 손자의 `parentId` 는 해당 Level 2 하위 페이지 ID 로 지정.
- 손자 페이지 제목은 "2.1 …", "2.2 …" 형식 권장 (위계가 한눈에 보이도록).
- Level 2 하위 페이지(손자의 부모) 는 손자들의 카탈로그 역할을 하도록 — 손자 목차 표를 포함.
- "Part A / Section B" 같은 중간 묶음 표지 페이지 명명은 손자에서도 금지 (§11 참조).

## 7. 시각화 (다이어그램) ⭐⭐⭐

본 스킬에서 **시각화는 1순위 표현 수단**이다. 가능한 한 모든 문서에 다이어그램을 적극 사용한다.

### 7.1 ⚠️ Confluence MCP 자동화 한계 (먼저 인지)

본 스킬은 Confluence MCP 의 다음 한계 위에서 동작한다 (2026년 5월 검증).

**완전 차단된 경로** (시도 자체 금지):
- ❌ **SVG 인라인 삽입**: markdown body 안에 `<svg>...</svg>` 코드를 넣으면 텍스트로 그대로 노출됨 (검증됨).
- ❌ **data URI 이미지**: `![](data:image/svg+xml;base64,...)` 는 ADF 파싱 단계에서 사라짐 (검증됨).
- ❌ **HTML img 태그 + data URI**: `<img src="data:...">` 는 HTML 그대로 텍스트로 노출 (검증됨).
- ❌ **MCP 첨부파일 업로드 도구**: Atlassian MCP 에는 페이지에 이미지/파일을 자동 첨부하는 도구가 아예 없다. `attachmentId` 파라미터는 이미 존재하는 첨부 ID 를 참조하는 용도일 뿐.

**조건부 가능 경로**:
- ⚠️ **외부 URL 이미지** (`![](https://[외부 URL])`): Atlassian media 가 자동으로 자기 미디어 객체로 수입해 페이지에 인라인 렌더링한다 (검증됨). 단 **외부에서 접근 가능한 공개 URL** 이 필요 — 사내 비공개 환경에서는 사용 불가.

**채택된 경로**:
- ⭐ **자동으로 페이지에 들어가는 시각화**: ASCII 다이어그램 (코드 블록), GFM 표
- ⭐ **수동 마무리가 필요한 시각화**: **SVG 파일을 사용자가 편집 모드에서 드래그 앤 드롭으로 첨부** (Confluence 가 SVG 를 그대로 렌더링 — PNG 변환 불필요, §7.6 참조)

이 한계를 극복하지 않고, **자동 가능한 시각화를 1순위, SVG 파일 첨부를 2순위**로 운용한다.

### 7.2 다이어그램 우선 원칙 (모든 문서 대상)

다음 중 하나라도 등장하는 페이지는 **반드시 다이어그램을 포함**:

- 시스템 구성 / 아키텍처 / 컴포넌트 배치
- 컴포넌트 간 연결성 / 의존성 / 인터페이스
- 데이터 / 명령 / 신호의 흐름
- 계층 / 레이어 / 트리 구조
- 상태 전이 / 라이프사이클
- 추상 개념의 직관적 이해 (비유적 시각화)
- 분류 / 카테고리 매트릭스 (단순 표보다 시각화가 효과적인 경우)

**한 가이드 시리즈 안에서 다이어그램 사용 빈도**:
- 대표 페이지: **시스템 전체 다이어그램 1개 필수**
- 하위 페이지: 가능한 한 페이지당 **1~2개 다이어그램** 동반
- 단순 명령 나열·짧은 노트만 다이어그램 생략 가능

### 7.3 다이어그램 종류와 선택

| 표현하고 싶은 것 | 다이어그램 종류 | 자동 가능 형식 |
| --- | --- | --- |
| 시스템 전체 구조 / 컴포넌트 배치 | 구조도 (structural) | ASCII, SVG 파일 |
| 컴포넌트 간 연결 / 의존성 | 연결성 다이어그램 (connectivity) | ASCII, SVG 파일 |
| 데이터/요청의 흐름 | 흐름도 (flowchart) | ASCII, SVG 파일 |
| 시간순 단계 | 타임라인 / 시퀀스도 | ASCII, SVG 파일 |
| 상태 전이 | 상태도 (state diagram) | SVG 파일 (ASCII 한계) |
| 추상 개념의 직관적 이해 | 일러스트 (illustrative) | SVG 파일 |
| 데이터·상태의 분류 | 매트릭스 / 4분면도 | GFM 표, SVG 파일 |

### 7.4 ⭐ 시각화 작성 흐름 (자동/수동 분담)

```
[페이지 작성 시작]
       ↓
[다이어그램 복잡도 판단]
       ↓
   ┌───┴────────────────────────┐
   │                            │
[단순/중간 — 자동]      [복잡/색상 중요 — 수동 마무리]
   │                            │
   ▼                            ▼
ASCII 다이어그램            SVG 파일 작성
또는 GFM 표                 (Code Execution 환경)
   │                            │
   ▼                            ▼
페이지 본문에                페이지 본문에 blockquote(📌)로
즉시 포함                   "SVG 첨부 필요" 자리 표시 + 발행
   │                            │
   ▼                            ▼
발행 (사용자 작업 0)        사용자: SVG 다운로드 →
                            편집 모드 → 드래그 앤 드롭 →
                            (선택) 안내 blockquote 삭제 → 게시
```

### 7.5 자동 가능한 시각화 (1순위 — 사용자 작업 없음)

#### 방법 A: ASCII 다이어그램

코드 블록 안의 텍스트로 그린 다이어그램. 단순한 구조·계층·흐름에 적합.

````markdown
```
┌────────────────────┐
│   Application      │
└─────────┬──────────┘
          │
          ├─→ Cache (Redis)
          ├─→ Database (PostgreSQL)
          └─→ Message Queue (RabbitMQ)
```
````

장점: 페이지에 즉시 포함, 깨질 일 없음, Git diff 가능.
단점: 색상 사용 불가, 6개 이상 박스 또는 곡선 화살표 표현 한계.

#### 방법 B: GFM 표 (매트릭스 시각화)

분류·비교가 핵심인 시각화는 표로 충분히 대체 가능.

```markdown
| 구분 | 특성 A | 특성 B | 비고 |
| --- | --- | --- | --- |
| **옵션 1** | ⭕ | ❌ | ... |
| **옵션 2** | ⚠️ | ⭕ | ... |
```

#### 방법 C: 외부 호스팅 이미지 링크 (사내 정책 허용 시)

```markdown
![architecture](https://[사내 이미지 서버]/diagrams/architecture.png)
```

사내 정책 확인 후 사용. 링크 깨질 위험 있으므로 안정적인 호스팅(사내 위키 첨부, GitHub Enterprise, S3 등)에서만.

### 7.6 수동 마무리가 필요한 시각화 (2순위 — 사용자 작업 1분 이내)

복잡한 구조·여러 색상·일러스트 같은 표현이 필요할 때.

#### SVG 파일 직접 첨부 워크플로우 ⭐ (PNG 변환 없음)

**핵심 발견 (2026년 5월 검증)**: Confluence 는 페이지 편집 모드에 SVG 파일을 드래그 앤 드롭으로 첨부하면 그대로 렌더링한다. **PNG 변환 단계는 불필요**하다.

다른 자동 삽입 경로 (MCP 를 통한 자동 첨부) 는 다음 이유로 사용 불가:
- ❌ markdown body 안에 `<svg>...</svg>` 인라인: 텍스트로 노출됨 (검증됨, atlassian/atlassian-mcp-server issue #53 계열)
- ❌ data URI `![](data:image/svg+xml;base64,...)`: ADF 파싱 단계에서 사라짐 (검증됨)
- ❌ `<img src="data:...">` HTML 인라인: HTML 그대로 텍스트 노출 (검증됨)
- ⚠️ `![](https://[외부 URL])` markdown 이미지: Atlassian media 가 자동 수입해서 렌더링되긴 하나, 외부 공개 URL 호스팅 필요 — 사내 비공개 환경에서는 불가
- ❌ Atlassian MCP 도구에는 페이지 첨부파일 업로드 도구가 없음 (`attachmentId` 파라미터는 이미 존재하는 첨부 ID 참조만)

따라서 본 스킬의 채택 경로는 **Claude 가 SVG 파일을 생성·제공 + 사용자가 편집 모드에서 드래그 앤 드롭** 이다.

**Claude 작업** (Code Execution 환경):

SVG 코드를 작성하기 전에 **`svg-creation` 스킬을 먼저 활성화**한다. 그 스킬이 정의한 폰트 스택과 `viewBox`-only 규칙을 따른 뒤, 본 스킬 §7.6 의 Confluence 다이어그램 고유 규칙 (viewBox 680 폭, 색상 의미, 분야별 라벨 등) 을 추가로 적용한다.

```python
# SVG 코드를 그대로 파일로 저장 (PNG 변환 없음)
# 폰트 스택과 viewBox 규칙은 svg-creation 스킬 참조
svg_code = '''<svg xmlns="http://www.w3.org/2000/svg" viewBox="0 0 680 320"
     font-family="'Apple SD Gothic Neo','Malgun Gothic','Noto Sans CJK KR','Helvetica Neue',Helvetica,Arial,sans-serif">
  ...
</svg>'''

with open('/mnt/user-data/outputs/[가이드명]-[다이어그램명].svg', 'w', encoding='utf-8') as f:
    f.write(svg_code)
```

그 다음 `present_files` 로 사용자에게 다운로드 링크 제공.

**페이지 본문에 이미지 자리 표시** (markdown):

```markdown
### 📐 전체 구조

> 📌 **SVG 첨부 필요**
>
> 이 자리에 SVG 파일 `[파일명].svg` 을 드래그 앤 드롭으로 업로드해 주세요. Confluence 가 SVG 를 그대로 렌더링합니다. 업로드 후 본 안내 박스는 삭제 가능합니다.

#### 다이어그램 보는 법

- **회색** — [의미]
- ...

#### 핵심 통찰 ⭐

1. [통찰 1]
2. ...
```

**사용자 작업**:
1. SVG 파일 다운로드 (Claude 가 `present_files` 로 제공한 링크)
2. 페이지 편집 모드 진입
3. blockquote 안내 위치에 SVG 드래그 앤 드롭
4. (선택) 안내 blockquote 삭제
5. 게시

페이지당 작업 시간: **30초 이내**.

#### SVG 작성 원칙

**먼저 `svg-creation` 스킬을 참조하라.** SVG 파일을 새로 생성하거나 수정할 때는 본 스킬과 별개로 항상 `svg-creation` 스킬을 먼저 활성화한다. 그 스킬이 정의하는 두 가지 핵심 규칙 — (1) 루트 `<svg>` 에 `viewBox` 만 지정하고 `width`/`height` 는 생략, (2) 한글 폰트 스택 `'Apple SD Gothic Neo','Malgun Gothic','Noto Sans CJK KR','Helvetica Neue',Helvetica,Arial,sans-serif` 사용 — 은 Confluence 첨부 SVG 에도 그대로 적용된다. 본 스킬은 그 위에 **Confluence 다이어그램 고유 규칙만** 얹는다:

- **viewBox 는 680 폭 기준**으로 작성 (Confluence 페이지 본문 폭과 비율이 잘 맞음). 예: `viewBox="0 0 680 400"`. 실제 표시 크기는 svg-creation 규칙대로 `width`/`height` 없이 컨테이너에 맡긴다.
- **색상은 의미를 인코딩**: 회색=중립/구조, 파랑=실행 주체, 노랑=설정/저장소, 청록=외부 도구, 보라=클라우드/원격, 산호=보조 도구
- **2~3개 색상으로 제한** (5개 이상은 시각적 노이즈)
- **글자는 14px 이상**
- **분야별 라벨 사용**: 박스 옆에 "(= [그 분야의 친숙한 개념])" 같은 매핑 라벨
- **색상 범례 + 화살표 범례를 SVG 안에 함께 포함**: 본문 텍스트 설명과 별개로 다이어그램 자체로도 의미가 전달되게.
- **`xmlns="http://www.w3.org/2000/svg"` 필수**: 독립 `.svg` 파일은 이 선언이 없으면 파싱 실패.

**피하는 것**:
- 그라데이션, 광택, 그림자 효과 (시각적 노이즈, 분야 일관성 깨짐)
- 6개 이상의 박스를 한 줄에 배치
- 색깔만으로 의미 표현 (라벨/형태도 함께)
- `width="100%"` 또는 `width="680"` 같은 폭 지정 (svg-creation 규칙 위반 — 일부 Confluence 매크로/컨테이너에서 0px 로 계산되어 SVG 가 사라질 수 있음. `viewBox` 만 둔다.)
- font-family 를 `Arial`, `sans-serif` 같은 영문 전용 스택으로 지정 (svg-creation 규칙 위반 — 한글이 시스템 기본체로 폴백됨)

#### SVG 파일 보존

페이지에 첨부된 SVG 는 Confluence 첨부파일로 저장되어 추후 다운로드 가능하지만, **수정 편의를 위해 Claude 측 작업물도 함께 보존** 권장:

```
[가이드명]/
├── architecture.svg       ← 원본 (수정 시 이 파일을 Claude 에게 다시 줘서 편집)
├── data-flow.svg
└── state-machine.svg
```

수정 시 흐름: 사용자가 Claude 에게 SVG 코드 (또는 파일) 를 다시 주고 수정 요청 → Claude 가 수정본 SVG 파일 제공 → 사용자가 Confluence 페이지 편집 모드에서 기존 첨부를 새 SVG 로 교체.

### 7.7 Mermaid / draw.io 매크로 (사내 플러그인 사용 시)

사내 Confluence 에 Mermaid 또는 draw.io 매크로 플러그인이 설치되어 있다면, 텍스트 기반으로 다이어그램을 페이지 안에서 직접 작성·편집 가능. 단, 본 스킬은 플러그인 설치를 가정하지 않는다 — 설치 여부는 사내 환경별로 별도 확인. 또한 markdown 모드에서 `{macro:mermaid}` 같은 wiki markup 도 지원되지 않으므로 (markdown→ADF 변환에서 사라짐), 플러그인 활용은 사용자가 편집 모드에서 매크로 메뉴로 직접 삽입해야 한다.

### 7.8 페이지별 다이어그램 선택 가이드

| 페이지 위치 | 권장 다이어그램 | 빈도 | 추천 형식 |
| --- | --- | --- | --- |
| 대표 페이지 (Level 1) | 시스템 전체 구조도 1개 | **필수** ⭐ | SVG 파일 (사용자 첨부) |
| 개념 설명 하위 페이지 | 부분 구조도 / 연결 다이어그램 | **권장** ⭐ | ASCII 또는 SVG 파일 |
| Q&A 형식 하위 페이지 | 추적 흐름 / 영역 분류 | 권장 | GFM 표 또는 ASCII |
| 튜토리얼/설치 하위 페이지 | 단계 흐름도 / 의존성 트리 | 선택 | ASCII |
| 시스템 구성도 하위 페이지 | 메인 구조도 + 보조 흐름도 | **주인공** | SVG 파일 (사용자 첨부) |

### 7.9 다이어그램 + 설명 블록 표준 패턴

다이어그램(자동/수동 무관) 옆에 항상 다음 3가지 설명 블록을 함께 배치한다:

```
[다이어그램 영역 — ASCII 코드블록 / GFM 표 / 이미지]

#### 색상의 의미
- 회색  — [의미]
- 파란색 — [의미]
- ...

#### 화살표가 말하는 것
- 위에서 아래로 — [의미]
- 점선          — [의미]

#### 이 그림의 핵심 통찰 ⭐
1. [통찰 1]
2. [통찰 2]
3. [통찰 3]
```

**핵심 통찰 블록**은 단순 그림 설명을 넘어, **"이 그림에서 무엇을 깨달아야 하는가"** 를 1~3개의 한 줄 문장으로 정리. 차량용 / 웹 / 백엔드 등 분야별 활용 관점도 여기에 반영.

이 설명 블록 덕분에 **이미지가 깨지거나 사용자가 SVG 첨부를 잠시 미뤄도** 본문 텍스트만으로 핵심이 전달된다.

## 8. 작업 순서

### 8.1 신규 가이드 생성 시 (대표 페이지부터)

```
1. 부모 페이지 ID 확인 (스페이스 루트 또는 상위 가이드)
2. spaceId 확인
3. 페이지 분야 자동 판단 (13장 참조) → 예시·비유 결정
4. 가이드 전체 항목 목록 작성 (5~12개)
5. ⭐ 대표 페이지 작성 (Level 1)
   ├── 5-1. 간단한 개념 한 단락 (분야에 맞는 비유 포함)
   ├── 5-2. 시스템 전체 다이어그램
   │       (ASCII 또는 SVG 파일 첨부, 색상/화살표/핵심 통찰 함께)
   └── 5-3. 하위 페이지 목차 표 + 학습 순서
6. createConfluencePage 호출 → 대표 페이지 ID 획득
7. ⭐ 하위 페이지 항목별 N개 작성 (Level 2)
   - parentId: 6번에서 받은 대표 페이지 ID
   - 각각 자체 완결 구조
8. 모든 하위 페이지 만들어진 후 대표 페이지의 목차에 실제 URL 채워서 update
```

### 8.2 단일 하위 페이지 추가 시

```
1. 대표 페이지 ID 확인
2. 동일 spaceId, parentId=대표페이지ID 로 createConfluencePage 호출
3. 대표 페이지의 하위 페이지 목차 표에 새 항목 추가 (update)
```

### 8.3 ⭐⭐ 기존 페이지 수정 워크플로우 (반드시 따르라)

**Confluence MCP는 부분 수정을 지원하지 않는다.** `Atlassian:updateConfluencePage` 의 `body` 파라미터는 **페이지 전체 본문을 통째로 받는다**. 일부분만 보내면 나머지가 사라진다.

이 제약 때문에 기존 페이지 수정에는 **사용자 편집을 보존하기 위한 엄격한 절차**가 필요하다.

> 🚫 **절대 금지 — 사고 위험**
>
> 다음 두 행위는 **절대 하지 않는다**. 어느 하나라도 어기면 사용자가 페이지에서 한 편집(SVG 첨부, 본문 미세 수정, 댓글로 받은 피드백 반영 등)이 통째로 사라진다.
>
> - **사용자 허락 없이 `updateConfluencePage` 호출** — 변경이 작아 보여도, 표 한 줄·오타 한 글자만 바꾸는 경우에도 금지
> - **`updateConfluencePage` 직전에 `getConfluencePage` 로 최신 본문을 fetch 하지 않고 호출** — Claude 의 메모리에 있는 본문은 최신이 아닐 가능성이 높음 (사용자가 직접 페이지를 편집했을 수 있음)

**기존 페이지 수정의 정식 절차 (5단계)**:

```
1. 사용자에게 무엇을 갱신할지 미리 제시 → 명시적 허락 받기
   (예: "§3.2 표에 행 하나를 추가해도 될까요?")

2. 허락 받은 직후 Atlassian:getConfluencePage 로 최신 본문 fetch
   (Claude 의 메모리 본문이 stale 일 수 있으므로 매번 새로 받기)

3. fetch 한 본문 위에서 필요한 부분만 교체
   (사용자가 한 편집 — SVG 첨부, 본문 정리, blockquote 삭제 등 — 은 그대로 유지)

4. 수정한 전체 본문을 updateConfluencePage 의 body 로 전송

5. versionMessage 에 "[구체적 위치] 수정" 명시
   (예: "§3.2 표에 행 추가")
```

#### 8.3.1 변경 규모별 세부 패턴

위 5단계는 변경 규모와 무관하게 동일하다. 변경 규모에 따라 달라지는 것은 **3단계(교체) 의 범위**뿐이다.

```
[패턴 A: 작은 부분만 변경 — 가장 흔함]
3-A. 한 줄 또는 한 표만 교체. 다른 부분은 fetch 한 본문 그대로.
   → versionMessage 예: "§3.2 표 한 행 수정"

[패턴 B: 섹션 단위 재작성]
3-B. 변경할 섹션을 새로 작성. 다른 섹션은 fetch 한 본문 그대로.
   → versionMessage 예: "§3 BSW 카탈로그 절 재작성"

[패턴 C: 전면 개편]
3-C. 새 본문을 통째로 작성. (이 경우만 fetch 결과를 거의 안 씀)
   → versionMessage 예: "전면 개편: 목차 트리 변경"
   → 단, 패턴 C 도 사용자 허락 + fetch 단계는 동일하게 수행 — 사용자가 fetch 결과를 보고 어떤 편집이 사라지는지 알 수 있어야 함
```

#### 8.3.2 새 페이지 생성과의 차이

| 작업 | 사용자 허락 | fetch 단계 |
|---|---|---|
| 새 페이지 생성 (`createConfluencePage`) | 작업 자체가 사용자 요청이면 OK (예: "3번 페이지 가자") | 불필요 |
| **기존 페이지 수정 (`updateConfluencePage`)** | **항상 별도 허락 필수** (작업 흐름상 자연스러워 보여도 별도 확인) | **항상 직전 fetch 필수** |

새 페이지 생성의 결과로 기존 페이지에 영향이 가는 경우 (예: 새 페이지 만들었으니 부모 페이지 목차에 링크 추가 필요) — 그 영향은 **별개의 수정 작업** 으로 보고 위 5단계를 다시 거친다.

#### 8.3.3 ⭐⭐ 첨부(SVG·이미지)가 있는 페이지 수정 — html 왕복 필수

Confluence 첨부 파일(SVG, 이미지 등) 은 페이지 본문(storage XHTML) 과 **별도 자산**으로 저장된다. 본문 안에서 그 첨부를 참조하는 마크업은 storage XHTML 기준 다음 형태다:

```html
<figure data-type="media-single" data-layout="center" data-width="702" data-width-type="pixel">
  <div data-type="media" data-media-type="file"
       data-id="e7336ec5-...-623037898b10"
       data-collection="contentId-504594449"
       data-width="720" data-height="760">adc-container-tree.svg</div>
</figure>
```

**핵심 — 어느 형식으로 fetch/send 하느냐가 첨부 보존을 가른다 (2026-05 실측)**:

| 방식 | 첨부 참조 | 결과 |
| --- | --- | --- |
| markdown 으로 fetch → markdown 으로 send | `![](blob:...)` 로 깨져서 들어옴 | ❌ send 시 `<figure>` 참조 소실 → 페이지에서 이미지 사라짐 |
| **html 으로 fetch → html 으로 send** | `<figure data-type="media-single">` 그대로 유지 | ⭐ 첨부 보존됨 |

따라서 **첨부가 하나라도 있는 페이지를 수정할 때는 반드시 `contentFormat: "html"` 로 왕복**한다. markdown 으로 본문을 새로 써서 보내면, 첨부 파일 자체는 attachments 에 남지만 본문의 `<figure>` 참조가 사라져 페이지에서 이미지가 안 보이게 된다.

**첨부 보존 수정 워크플로우 (5단계의 구체화)**:

```
1. 사용자에게 변경 안 제시 → 명시적 허락 (§8.3 그대로)

2. getConfluencePage(contentFormat: "html") 로 최신 storage XHTML fetch
   → response.body 가 <p>, <table>, <figure data-type="media-single"> … 형태의 XHTML
   → version 번호 확인: 메모리의 마지막 version 과 다르면 사용자가 직접 편집한 것

3. fetch 한 XHTML 위에서 텍스트만 교체
   → <figure ...><div data-type="media" data-id="..." ...>파일명</div></figure> 블록은
     data-id·data-collection 까지 한 글자도 바꾸지 말고 그대로 유지
   → 바꿀 문단·표·blockquote 만 XHTML 로 교체

4. updateConfluencePage(contentFormat: "html", body: 교체한 XHTML 전체)

5. versionMessage 에 변경 위치 명시 + response 의 version 이 +1 됐는지 확인
```

> 🚫 **절대 금지 — 첨부 소실 사고**
>
> - 첨부 있는 페이지를 **markdown 으로 fetch/send** 하지 말 것. `<figure>` 가 `![](blob:...)` 로 깨져 들어오고, send 하면 본문 참조가 사라진다.
> - fetch 한 XHTML 의 `<div data-type="media" data-id="...">` 의 **`data-id` / `data-collection` 을 절대 수정하지 말 것.** 이 값이 첨부 자산을 가리키는 키다.
> - Claude 메모리에 있는 (이전 turn 의) 본문으로 send 하지 말 것 — 사용자가 그 사이 SVG 를 첨부했을 수 있다. **매번 직전에 html 로 fetch.**

> 💡 **첨부가 이미 사라진 경우 복구**
>
> 과거에 markdown 으로 잘못 send 해서 본문 참조가 사라졌더라도, **첨부 파일 자체는 페이지 attachments 패널에 남아있다.** 사용자가 편집 모드 → attachments 패널에서 다시 본문에 삽입하면 복구된다. 또는 Claude 가 html 로 fetch 해서 빠진 자리에 `<figure>` 블록을 다시 끼워 넣어도 된다 (단 해당 첨부의 `data-id` 를 알아야 함 — 페이지의 attachments 목록에서 확인).

### 8.4 페이지 작성 시 점검 흐름

각 페이지 작성 시:

1. 페이지 종류 결정 (대표 / 개념 설명 / Q&A / 시스템 구성도 / 튜토리얼)
2. **분야 자동 판단** (13장) → 예시·비유 결정
3. 제목 명명 (6.1 규약 — Part 접두 금지)
4. **다이어그램 필요성 + 형식 결정** ⭐
   - 시스템 구조/연결성/흐름 등장 → 다이어그램 필수
   - 단순 구조 → ASCII 다이어그램 (자동, 즉시 포함)
   - 복잡/색상 중요 → SVG 파일 작성 → blockquote(📌) 안내 자리 표시 (사용자 드래그앤드롭)
5. 구조 골격 작성 (12장 보일러플레이트 활용)
6. 본문 작성 (3.2 개발자 톤 + 3.4 분야별 비유)
7. 컴포넌트 배치 (4장 — 표, 박스 강조 blockquote, 코드 블록)
8. **헤더 위계 점검** (H1, H2 미사용, H3부터 시작)
9. 다이어그램 + 색상 의미 / 화살표 / 핵심 통찰 (7.9)
10. createConfluencePage 호출 (**반드시 `contentFormat: "markdown"`**)
11. 링크 확인 (response의 `webui` 경로)
12. **(다이어그램 첨부 필요 시) 사용자에게 SVG 파일 + 드래그앤드롭 안내 전달**

### 8.5 ⭐⭐ 기존 페이지 수정 시 점검 흐름

§8.3 의 5단계 절차를 매번 다음 순서로 확인하라. 어느 단계라도 건너뛰면 사용자 편집 손실 위험.

1. **변경 대상 식별** — 어느 페이지의 어느 부분을 바꿀 것인가
2. **사용자에게 변경 안 제시** — "어떤 페이지의 어떤 부분을 어떻게 바꿀 것" 인지 자연어로 설명
3. **명시적 허락 대기** — "네", "진행", "OK" 등 명확한 승인 신호. 모호하면 다시 확인
4. **`Atlassian:getConfluencePage` 로 최신 본문 fetch** — 절대 생략 금지. **첨부(SVG·이미지)가 있는 페이지면 `contentFormat: "html"` 로 fetch** (§8.3.3). 첨부 없으면 markdown 도 가능.
5. **fetch 결과 검토** — version 번호로 사용자 직접 편집 여부 확인. `<figure data-type="media-single">` 같은 첨부 참조가 보이면 그대로 보존 대상으로 표시.
6. **본문 수정** — 사용자 편집·첨부 참조 보존하면서 필요한 부분(문단·표)만 교체. `data-id`·`data-collection` 은 절대 손대지 않음.
7. **`updateConfluencePage` 호출** — fetch 한 형식과 동일하게(첨부 있으면 `contentFormat: "html"`), `versionMessage` 에 변경 위치 명시.
8. **응답에서 새 version 번호 확인** — 예상 version (이전 version + 1) 과 일치하는지 확인
9. **사용자에게 변경 완료 + version 보고**

위 9단계 중 1~3 단계는 **§8.4 의 작성 흐름과 가장 큰 차이점**이다. 작성은 사용자 요청 자체가 권한 부여이지만, 수정은 매번 별도 허락이 필요하다.

#### 8.5.1 자주 발생하는 함정

| 함정 | 무엇이 일어나나 | 방지 방법 |
|---|---|---|
| **"작은 변경이니 허락 생략"** | 한 줄 변경에도 본문 전체가 갈아치워짐. fetch 안 했으면 사용자 편집 사라짐 | 변경 규모와 무관하게 §8.3 의 5단계 |
| **"방금 fetch 했으니 또 안 해도 됨"** | 사용자가 그 사이에 페이지를 만졌을 수 있음 | `updateConfluencePage` 직전에 항상 새로 fetch |
| **"링크만 갱신하니 본문 전체를 다시 안 보내도 되겠지"** | `body` 파라미터가 필수. 일부분만 안 됨 | MCP 의 한계로, 항상 전체 body 전송 |
| **"새 페이지 만든 흐름이라 부모 페이지 목차 갱신은 자연스러운 후속 작업"** | 부모 페이지 갱신도 별개의 수정 작업. 별도 허락 필요 | 작업 단위마다 허락 |

## 10. 변형 가능한 것 (커스터마이징 포인트)

## 11. 변형 시 지키는 원칙 (Don't 리스트)

**MCP / 서식 포맷** ⭐⭐⭐
- ❌ `contentFormat: "html"` 으로 호출 시도 금지 — enum 위반으로 거부됨.
- ❌ `contentFormat` 파라미터 생략 금지 (HTML body 보내면 ADF JSON 파싱 실패).
- ❌ markdown body 안에 `<div data-type="panel-info">` / `<span data-type="status">` / `<section data-type="layout-two-equal">` 같은 ADF 전용 HTML 태그 인라인 삽입 시도 금지 — escape 되거나 텍스트로 노출됨.
- ❌ 표·강조·인용을 HTML 태그(`<table>` / `<strong>` / `<blockquote>`)로 작성 금지. GFM markdown syntax (`|...|`, `**...**`, `> ...`) 만 사용.
- ❌ 코드 블록을 `<pre><code class="language-...">` HTML 태그로 작성 금지. fenced code block (``` 언어이름 ... ```) 만 사용.

**구조**
- ❌ **사용자 요청 없이** 3레벨 이상 페이지 트리 만들지 말 것 (사용자 명시 요청 시 §6.3.1 의 예외로 허용).
- ❌ "Part 1 / Part 2" 같은 중간 묶음 페이지 만들지 말 것.
- ❌ 대표 페이지에서 "간단한 개념 / 다이어그램 / 하위 목차" 3블록 중 어느 하나라도 생략 금지.
- ❌ 하위 페이지를 13개 이상 만들지 말 것.
- ❌ 하위 페이지 제목에 "Part X." / "X장." 같은 접두 사용 금지.

**헤더 위계**
- ❌ **본문에 H1 (`# ...`) 사용 금지** (Confluence title 자동 렌더링과 중복).
- ❌ **본문에 H2 (`## ...`) 사용 금지** (시각적으로 너무 큼).
- ❌ 본문의 가장 큰 헤더는 `### ...` (H3) 부터 시작.

**시각화**
- ❌ **본문에 `<svg>...</svg>` 인라인 삽입 시도하지 말 것** (Confluence MCP가 SVG 인라인 미지원, 무시되거나 텍스트로 빠짐).
- ❌ 시스템 구조 / 연결성 / 흐름이 등장하는 페이지에서 시각화 자체를 빼지 말 것 (ASCII / GFM 표 / SVG 파일 첨부 중 적합한 형식 선택).
- ❌ 다이어그램만 있고 색상/화살표/핵심 통찰 블록 없는 상태로 발행 금지.
- ❌ 이미지를 본문에 넣을 자리만 두고 안내 blockquote(📌) 없이 발행 금지 (사용자가 어디에 무엇을 업로드할지 모름).
- ❌ 한 페이지에 6개 이상 색상 사용 금지.
- ❌ 색상만으로 의미 표현 금지 (라벨/형태도 함께).
- ❌ **SVG 파일을 `svg-creation` 스킬 없이 작성 금지**. 그 스킬의 viewBox-only 규칙과 한글 폰트 스택을 건너뛰면 Confluence 일부 매크로에서 SVG 가 0px 로 사라지거나 한글이 시스템 기본체로 폴백된다.
- ❌ SVG 루트에 `width`/`height` 속성 지정 금지 (`viewBox` 만 사용 — svg-creation 규칙).

**독자 톤**
- ❌ "이해하지 못해도 괜찮습니다", "어렵지 않습니다" 같은 위로 톤 사용 금지.
- ❌ "강력한", "혁신적인", "쉬운" 같은 마케팅 톤 사용 금지.
- ❌ 자명한 기술 용어를 굳이 풀어쓰지 말 것 (그 분야 개발자에게 자명하면 그대로 쓴다).

**분야 적응**
- ❌ 페이지 분야와 무관한 비유 사용 금지 (웹 가이드에 차량용 비유 끼워넣지 않기).
- ❌ "한 가지 분야 예시로 모든 분야 설명" 패턴 금지.

**컴포넌트**
- ❌ 박스 강조 (blockquote) 5개 이상 사용 금지.
- ❌ 🚫 절대 금지 헤더를 일상적 주의에 사용 금지 (panel-error 대체 규칙과 동일).
- ❌ 표 컬럼 5개 이상 사용 금지.
- ❌ 색상 직접 지정 시도 금지 (markdown 미지원, 인라인 HTML 도 escape 됨).

**기타**
- ❌ "당신" / "유저" 같은 호칭 사용 금지 (사용자님 / 사용자 사용).
- ❌ Title Case 영문 제목 금지 (한국어 + 괄호 영어 부제).
- ❌ 한 페이지 내에서 같은 개념의 표기 혼용 금지.
- ❌ Q&A 페이지에서 추적 흐름 요약 블록 누락 금지.

## 12. 표준 페이지 보일러플레이트

> **중요**: 모든 보일러플레이트는 **markdown 으로 작성**된다 (Atlassian MCP 가 `contentFormat: "markdown"` 만 받음 — §4.0 참조). 본문은 H3 (`###`) 부터 시작한다. H1 (`#`), H2 (`##`) 를 본문에 쓰지 않는다. 분야별 예시는 13장 분야 매트릭스에 따라 동적으로 선택한다.

> ⚠️ **호출 시 필수 파라미터**
>
> - `contentFormat: "markdown"` ← 항상
> - `body`: 아래 보일러플레이트의 markdown 콘텐츠
> - `title`, `spaceId`, `parentId` (하위 페이지의 경우)

### 12.5 markdown 링크 문법 주의

위 보일러플레이트의 `[[제목]]([URL])` 표기는 본 스킬에서 사용하는 단순 placeholder 다 (이중 대괄호가 아니다). 실제 작성 시에는 표준 GFM 링크 syntax 를 사용한다:

```markdown
[표시될 제목](https://ljindong.atlassian.net/wiki/spaces/TECH/pages/486572035)
```

Confluence 내부 페이지 링크는 절대 URL (`https://ljindong.atlassian.net/wiki/spaces/...`) 형태로 작성한다. tinyui 링크 (`/x/A4AAHQ`) 도 절대 URL 로 풀어서 (`https://ljindong.atlassian.net/wiki/x/A4AAHQ`) 사용 가능.

## 13. ⭐ 분야 판단 + 분야별 콘텐츠 정책

본 스킬의 핵심 운영 원칙. **페이지 작성 전 반드시 분야를 판단하고 그에 맞는 예시·비유·코드를 선택한다.**

### 13.1 분야 자동 판단 기준

페이지 분야는 다음 우선순위로 판단:

```
1. 사용자가 직접 분야를 명시 (예: "Python 가이드 만들어줘") → 즉시 결정
2. 페이지 주제·제목의 키워드 분석
3. 본문에 등장할 기술 스택 / 도구 이름
4. 사용자의 기존 작업 맥락 (이전 대화)
5. 모호하면 사용자에게 한 번 확인
```

키워드 → 분야 매핑:

| 키워드 등장 | 판정 분야 |
| --- | --- |
| AUTOSAR, MISRA, BSW, RTE, ASW, CAN, LIN, FlexRay, ECU, ARXML, MCAL, FuSa | **임베디드 C/C++ (차량용)** |
| MCU, RTOS, FreeRTOS, OSEK, ARM Cortex, NXP, STM32, Infineon, Keil, IAR | **임베디드 C/C++ (일반)** |
| React, Vue, Angular, Svelte, Next.js, CSS, Tailwind, JSX, hook, state | **웹 프론트엔드** |
| Node.js, Express, FastAPI, Django, Spring, ORM, REST, GraphQL, JWT | **백엔드 / API** |
| Python, Pandas, NumPy, ETL, Airflow, Jupyter, scraping, automation | **Python 자동화 / 데이터** |
| Docker, Kubernetes, K8s, Terraform, Ansible, AWS, GCP, Azure, CI/CD | **인프라 / DevOps** |
| LLM, GPT, Claude, embedding, RAG, fine-tuning, prompt, transformer | **AI / ML** |

### 13.2 분야별 콘텐츠 정책 매트릭스

| 분야 | 비유 1순위 | 코드 언어 | 책임 영역 | 자주 쓰는 도구 |
| --- | --- | --- | --- | --- |
| **임베디드 C/C++ (차량용)** | ECU/BSW/ARXML/CAN | C, ARXML, Make | 보안·컴플라이언스(ISO 26262, 21434)·계약 | Keil, IAR, Tessy, CANalyzer, Polyspace |
| **임베디드 C/C++ (일반)** | 레지스터/인터럽트/메모리맵 | C, C++, Make, CMake | 메모리 안전성·실시간성 | GCC, GDB, J-Link, Lauterbach |
| **웹 프론트엔드** | 컴포넌트 트리/렌더링/상태 | JS, TS, JSX, CSS | 접근성·번들 사이즈·SEO | Webpack, Vite, Chrome DevTools |
| **백엔드 / API** | 요청-응답/미들웨어/트랜잭션 | Python, JS, Java, Go | 보안 취약점·확장성·일관성 | Postman, k6, Datadog |
| **Python 자동화 / 데이터** | 파이프라인/스트리밍/배치 | Python, SQL | 데이터 무결성·재현성 | Jupyter, Airflow, dbt |
| **인프라 / DevOps** | 컨테이너/네트워크/오케스트레이션 | YAML, HCL, Bash | 가용성·롤백·보안 | kubectl, Terraform, Helm |
| **AI / ML** | 모델/임베딩/추론 파이프라인 | Python, JSON | 데이터 프라이버시·모델 신뢰성 | HuggingFace, LangChain, MLflow |

### 13.3 분야 판단 후 적용 흐름

```
페이지 작성 요청 도착
       ↓
분야 판단 (13.1)
       ↓
13.2 매트릭스에서 그 분야의 행 선택
       ↓
적용:
  ├── 비유 블록    → 그 분야의 비유 1순위 사용
  ├── 코드 예시     → 그 분야의 코드 언어 사용
  ├── 책임 영역     → 그 분야의 blockquote(⚠️) 콘텐츠
  ├── 도구 이름     → 자주 쓰는 도구를 자연스럽게 인용
  └── 페이지 제목   → 6.1 분야별 명명 예시 따름
```

### 13.4 분야가 모호할 때

분야가 명확히 잡히지 않으면 사용자에게 한 번 확인:

```
"이 가이드의 분야가 [임베디드 C/C++] 인지 [Python 자동화] 인지 
정확히 짚어주실 수 있을까요? 분야에 따라 비유와 예시가 달라집니다."
```

명시 없이 작성하다가 잘못된 분야의 비유를 끼워 넣는 것보다, 한 번 확인하는 것이 안전하다.

### 13.5 본 스킬의 주 운영자 컨텍스트

본 스킬을 사용하는 주요 사용자(스킬 소유자)의 배경:

- **주 도메인**: 자동차 제어기 개발 (AUTOSAR, MISRA-C, CAN, LIN, MCU 펌웨어)
- **보조 도메인**: Python 자동화 (HIL, 로그 파싱, 평가 자동화), 풀스택 웹 개발 (JavaScript)
- **전형 사용 패턴**: 본인 학습 정리 + 팀원 도입 검토 자료 + 보안팀 협의 자료
- **선호 톤**: 결론 먼저, 단정적, 솔직함, 분야 깊이 있는 다룸

따라서 분야가 명확하지 않을 때 **임베디드 C/C++ (차량용)을 디폴트 추정**하되, 페이지 주제가 명백히 다른 분야면 그 분야로 즉시 전환한다.
