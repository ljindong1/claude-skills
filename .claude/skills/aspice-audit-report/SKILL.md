# ASPICE 품질점검 체크리스트 자동 작성(오디팅) 스킬

ASPICE 단계말 품질점검 체크리스트를 AI가 1차로 작성하고, 사람 작성본과 비교해 정확도를 측정하면서 **분류 규칙을 반복 개선**하는 워크플로다. 목표는 "AI가 전부 판정"이 아니라 **AI가 자동 판정 가능한 부분(존재성·형식·미대상 스코핑)을 빠르게 처리하고, 사람이 꼭 봐야 하는 부분(일관성·추적성·승인 등)은 명확히 분리해 넘기는 분업**이다.

## 왜 이 구조인가 — 3계층 점검 모델

체크리스트는 한 장이 아니라 서로 맞물린 3개 시트이고, **산출물명이 셋을 잇는 조인키**다. 이걸 모르면 같은 파일을 시트마다 다시 열고 같은 판단을 반복하게 된다.

| 시트 | 무엇을 보는가 | 점검 단위 |
| --- | --- | --- |
| **Target** | 산출물이 어디 있고 점검 대상인가 | 출력 산출물 ↔ 실제 파일 매핑, 점검대상/미대상 |
| **Process Checklist** | 활동이 절차대로 수행됐는가 | 업무 활동(RACI·실행시점·입력→출력) |
| **WP Checklist** | 산출물 내용이 품질기준을 충족하는가 | 산출물의 ◆관점별 항목 |

자세한 배경·절차는 `references/methodology.md`, 사내 절차↔ASPICE 대응은 `references/process_mapping.md`를 필요할 때 읽어라. 유지보수(어디만 고치면 되나)는 `references/maintenance.md` 참고.

## 입력과 산출물

- **잣대(읽기 전용)**: ASPICE PAM(폴더에서 참조), 사내 시스템 문서 관리대장(적용 절차서·Rev). 점검의 기준이며 채우지 않는다. PAM 원문은 동봉하지 않고 필요한 프로세스 정수는 `references/process_mapping.md`에 둔다.
- **양식**: `assets/blank_template.xlsx`(동봉 빈 양식) 또는 사람 완성본에서 답 칸을 비운 빈 양식.
- **점검 대상**: 오디팅 대상 산출물 폴더(docx/xlsx) + 사내 시스템 URL.
- **산출물**: ① **체크리스트 리포트**(빈 양식에 Target 매핑 + Process/WP 1차 판정 + 사람급 코멘트까지 채운 최종 납품물), ② **매핑 통합 리포트**(매핑 테이블 + 커버리지 한 파일, `build_mapping_report.py`), ③ **AI검증표**(사람 작성본 있을 때, `verify_against_baseline.py` — 스코핑·결과 판정 일치율과 위험오판을 집계하는 **규칙 튜닝용 핵심 분석자료**), ④ **비교분석 검증용**(`build_comparison.py` — 코멘트 텍스트 좌/우 대조, 검증 후 삭제 가능). 그 외 빈 양식·다이제스트·trace_evidence 등은 중간 산출물이다.

> 검증 산출물 3종은 역할이 겹치지 않는다: **AI검증표**=일치율·위험오판 숫자(규칙 튜닝), **비교분석**=코멘트 텍스트 대조(사람급 눈 검토), **verify_comment_coverage**=전 항목 심층 보유 PASS/FAIL 게이트(파일 없음).

## 워크플로 (반복 루프)

순서대로 수행한다. 각 단계는 `scripts/`의 스크립트로 실행한다.

### 0. 기준 고정
적용할 사내 절차서·표준을 어느 Rev로 볼지 관리대장에서 확정한다(작성이 아니라 식별·고정). 이게 "최신 템플릿/절차 준수" 판정의 기준점이 된다.

### 1. 빈 양식 준비
**빠른 시작**: `assets/blank_template.xlsx`(동봉된 기본 빈 양식)를 그대로 결과 파일로 복사해 쓴다.
**양식이 개정됐을 때**: 사람 완성본에서 답 칸(Applicable·Review Result·Review Comments·Non-conformances ID, Target의 파일/판정)만 비우고 문항·구조·드롭다운은 보존해 항상 최신 양식의 빈 양식을 생성한다.
```
python scripts/make_blank_template.py <완성본.xlsx> <빈양식.xlsx>
```
동봉본은 빠른 시작용 기본값이며, 양식 버전이 바뀌면 생성 스크립트 결과를 기준으로 삼는다.

### 2. 산출물 매핑 — 반드시 "내용 기반"으로 (필수)
**원칙: 파일명이 아니라 파일 내용으로 매핑한다.** 파일명 키워드만으로 매핑하면 철자(시험/테스트)·버전·대소문자 차이로 누락·오분류가 반복된다. 그래서 **매번 아래 순서를 그대로 밟는다 — 선택이 아니라 기본 절차다.** 모든 판정에 `날짜 (AI 1차)`를 표기한다.

**2-1) 폴더 다이제스트 생성** — 모든 파일(+1단계 하위 폴더)의 시트명·헤딩·첫 행·검출 ID·버전을 한 장으로.
```
python scripts/summarize_folder.py <대상산출물폴더> --out folder_digest.md
```

**2-2) AI가 다이제스트를 읽고 매핑을 생성** — `folder_digest.md`와 체크리스트 산출물 목록을 함께 읽고, 각 파일이 어느 산출물인지·점검대상/미대상/보조/입력/잣대인지 **내용 근거**로 판단해 `deliverable_map.json`을 만든다(evidence의 파일 필드는 반드시 실제 파일명, note에 시트·헤딩 근거). 시트명·헤딩이 결정적 단서다(예: 'License_Report' 시트 → 오픈소스 관련). 의미가 모호하면(라이선스 검증 vs 취약점 점검처럼) 점검대상으로 **보이게** 두되 최종 동일성 판정은 사람에게 넘긴다 — 임의로 같다고 단정하지 않는다. `deliverable_map.example.json`은 시작점일 뿐이다.

**2-3) 매핑 검토("테이블 먼저")** — AI는 `deliverable_map.json`의 매핑 근거를 사용자에게 제시해 먼저 검토받는다(임의 단정 금지). 매핑을 파일로 남기는 건 2-6의 통합 리포트(매핑 테이블 시트)로 일원화한다 — 더 이상 별도 매핑테이블 파일을 따로 만들지 않는다.

**2-4) 누락 점검**
```
python scripts/coverage_check.py <대상산출물폴더> deliverable_map.json
```
'미매핑'이면 (1) 점검대상인데 누락 → 맵 보강, (2) 보조/입력/잣대(BJ1·타프로젝트·지침서·표준) → 무시. 사람 작성본이 있으면 그 Target 파일명이 추가 근거다.

**2-5) 채우기**
```
python scripts/fill_checklist.py <빈양식.xlsx> <결과.xlsx> --map deliverable_map.json
```

**2-6) 매핑 통합 리포트(한 파일)** — 매핑 테이블 + 커버리지(파일 관점/체크시트 관점)를 **중복 없이 한 파일**로 낸다.
```
python scripts/build_mapping_report.py deliverable_map.json <결과.xlsx> <대상산출물폴더> <매핑_통합리포트.xlsx>
```
시트는 `요약`·`매핑 테이블`·`파일 관점`·`체크시트 관점` 4개. (과거 build_mapping_table.py·mapping_report.py가 '파일 사용여부' 시트를 중복 생성하던 것을 하나로 합친 스크립트다.)
매칭은 가장 구체적인(긴) 키가 이긴다(일반 키 '회의록'이 '검토 회의록'을 가로채지 않음). 하위 폴더도 한 항목으로 인식한다. 리포트 정확도는 맵의 파일 필드가 실제 파일명일 때 보장된다(설명 문구 금지).

판정 원칙(이게 일치율을 좌우하므로 `scripts/classification_rules.py`에서 튜닝):
- **미대상**(테스트 단계말·단계 미도래·해당없음) → `Applicable=No` + 사유. 가장 객관적이므로 먼저 처리.
- **존재성·형식 항목**(정의/포함/작성/구조/템플릿) + 문서 접근 가능 → `Pass` + 근거(문서·절).
- **사람 필수 항목**(승인·배포·협의·고객·추적·일관·정합·동료검토·이력·등록·모니터링) → 판정 보류, `Review Result` 비우고 `확인 필요` 사유. **자동 Pass 금지** — 위험 오판의 주원인이다.
- **사내 시스템 근거 항목**(PMS·레드마인·codebeamer·git URL) → `확인 필요`. 접근 불가.
- **Process Yes/No(스코핑)**: 실행시점이 후행단계(마일스톤·완료·M-1 등 `LATER_PHASE_TIMING`)면 `No`, 미대상이면 `No`, 그 외 활동은 `Yes`(출력 산출물 식별 여부를 코멘트에 표기). 실행시점 칸이 비어 있으면 미대상 패턴에만 의존하므로, 활동 '미수행'으로 인한 No는 사람이 확정한다.

### 3. 정합성 점검
미대상인데 결과가 있음, 코멘트 누락, Fail인데 부적합ID 없음 등 모순을 검사한다(`fill_checklist.py`가 마지막에 자동 수행).

### 4. 검증표 생성 (사람 작성본이 있을 때)
같은 양식의 사람 작성본과 행 단위로 비교해 정확도를 측정한다.
```
python scripts/verify_against_baseline.py <결과.xlsx> <사람작성본.xlsx> <검증표.xlsx>
```
두 축으로 본다: **① 스코핑 정확도**(Applicable Yes/No 일치율)와 **② 결과 판정 정확도**(사람이 Pass/Fail 한 항목 중 일치/미판정/위험오판). **위험 오판(사람 Fail인데 AI Pass)** 을 최우선으로 본다.

### 5. 규칙 튜닝 → 다시 2번 (핵심 개선 루프)
검증표의 불일치를 보고 `classification_rules.py`를 고친다. 전형적 개선:
- **AI 과소(누락)**: 미대상으로 잘못 뺀 산출물 → `NA_KEYWORDS`에서 해당 패턴을 빼거나 매핑을 추가.
- **위험 오판**: 자동 Pass 했는데 사람은 Fail → 그 문항의 키워드를 `HUMAN_REQUIRED`에 추가해 확인 필요로 강제.
- **AI 미판정 과다**: 시스템 접근이 가능해지면 해당 근거를 매핑에 넣어 판정 가능 항목으로 전환.
- **검증/시험 '계획서'를 미대상으로 빠뜨림(과소)**: `NA_KEYWORDS`('통합 검증' 등)가 산출물명에 부분매칭돼 *계획서*까지 No로 떨군다. `classification_rules.is_design_phase_target`(키워드 "계획"·"정적")로 점검대상을 유지한다 — 계획서=설계 단계 점검대상, 명세서/결과서=테스트 단계말 미대상. 또 매핑된 파일이 없어도 미대상이 아니면 `Applicable=Yes`(확인 필요)로 둔다(점검대상≠검증가능).

규칙을 고치면 2번을 다시 돌려 검증표로 일치율 변화를 확인한다. 이 루프를 일치율이 만족스러울 때까지 반복한다.

### 6. 최종 확인 (필수 게이트 — 통과 못 하면 넘기지 않는다)
심층 코멘트 주입 후, 리포트를 사용자에게 넘기기 **전에 반드시** 전수 커버리지를 검사한다. 점검대상(Applicable=Yes) **WP·Process 항목이 하나도 빠짐없이** 사람급(AI 심층) 코멘트를 가졌는지, 사람 작성본이 있으면 사람이 코멘트한 항목 대비 비심층/빠짐이 0인지 확인한다.
```
python scripts/verify_comment_coverage.py <결과.xlsx> [사람작성본.xlsx]
```
- `[PASS]`(WP·Process 전 항목 심층, 사람 대비 0건)가 나올 때까지 반복한다. WP·Process에서 사람이 No로 본 항목(계획서 내 일부 세부 질문, 테스트 단계말 명세서/결과서 등)을 Yes로 잡은 경우는 코멘트가 아니라 `Applicable=No` 스코핑 교정으로 처리한다. `[FAIL]`이면 출력된 '빠짐/비심층' No의 산출물 본문·검토결과서를 다시 정독해 코멘트를 작성·주입하고 재검사한다.
- '확인 필요(문서 부재)'로 정직히 남긴 항목은 심층으로 인정된다(환각 대신 정직한 한계 표기가 정답이다). 단 그 사유가 코멘트에 명시돼 있어야 한다.
- 고위험(누락 시 잘못된 합격) 점검이므로, 가능하면 이 검사를 **별도 에이전트로 한 번 더 교차 확인**(샘플 항목의 코멘트가 실제 본문/검토결과서 근거와 일치하는지)한 뒤 넘긴다.

**비교표는 항상 함께 만든다(검증 산출물).** `[PASS]`를 받은 뒤, 사람이 사람급 도달 여부를 눈으로 대조할 수 있도록 사람 코멘트(좌)/AI 코멘트(우) 비교표를 반드시 생성한다.
```
python scripts/build_comparison.py <결과.xlsx> <사람작성본.xlsx> <비교분석_사람급_전체_검증용.xlsx>
```
- 출력은 요약 시트(산출물별 사람Yes/AI/심층/비고) + 상세 비교 시트(No·항목·사람·AI)로 구성된다.
- **이 비교표는 검증용 임시 산출물이다.** 최종 납품물은 체크리스트 리포트뿐이므로, 파일명에 `_검증용`을 붙여 구분하고 사용자에게는 검증이 끝나면 삭제해도 되는 파일임을 명시한다. 사람 작성본이 없으면(신규 점검) 이 단계는 생략하고 `verify_comment_coverage.py`의 리포트 단독 [PASS]로 갈음한다.

## 심층(필수): 사람급 Review Comment — WP·Process 점검대상 전수 정독

> **이 단계는 선택이 아니라 필수다. 시간·토큰에 관계없이 점검대상(Applicable=Yes) 항목을 WP Checklist(산출물 내용)와 Process Checklist(활동 수행) 둘 다 하나도 빠짐없이 정독해 항목별 코멘트를 작성한다.** 기본 채우기(2-5)의 1차 템플릿 코멘트는 사람 코멘트(절 번호 인용 + 미흡점 + 권고)와 격차가 크므로, 리포트를 넘기기 전에 **반드시 전수 심층화**한다. 결정론적 스크립트로는 불가능 — AI가 본문을 읽고 직접 쓴다. "핵심만"이 아니라 "전부"다.

**원칙: 빠짐없이, 그리고 정직하게.**
- 점검대상 산출물 **본문**과 그 **검토결과서**를 전부 연다. 사람 코멘트의 구체적 결함은 codebeamer가 아니라 폴더의 검토결과서(`*_검토결과서_*.xlsx`, `*_Review_Report_*.xls`, `*_동료검토_*.xls`)의 '검토 결과'·'결함 내용'·Check List(NG) 시트에 들어 있다. 이걸 안 읽으면 사람급이 될 수 없다.
- 근거가 폴더에 **없으면**(codebeamer export의 빈 Description, 폴더에 실문서 부재) 절대 지어내지 말고 `확인 필요 — 문서 부재/codebeamer 접근 필요`로 남긴다. 자동 Pass 금지.

절차(점검대상 산출물마다 반복 — 한 산출물도 건너뛰지 않는다):
1) 대상 + 검토결과서 본문 추출:
```
python scripts/extract_deliverable_text.py <산출물파일> --out text.md
```
xls/xlsx 검토결과서는 시트별로 직접 읽어 결함·NG(예: 동료검토 Check List의 NG 행, '결함 내용' 시트의 검증항목 ID·심각도)를 수집한다.
2) AI가 `text.md`와 검토결과서를 읽고, 그 산출물의 각 WP 문항에 `references/comment_style.md` 형식으로 코멘트를 작성한다 → `comments.json`. **머리는 반드시 AI 정독 마커로 시작**한다: `YYMMDD (AI·본문정독)` / `(AI·결과서 정독)` / `(AI·검토결과서)` / `(AI·추적데이터)` / `(AI·구조확인)` / `(AI·확인필요)`. 이 마커가 있어야 최종 확인 게이트가 '심층'으로 인정한다. 형식: `{절/표/필드 근거} {확인}. 다만 {미흡점}. {권고/확인 필요}`. Pass여도 미흡점을 기록하고, 도메인 적정성·시스템 사실은 단정 말고 "확인 필요"(환각 금지).
3) 주입:
```
python scripts/write_comments.py <결과.xlsx> comments.json
```
`comments.json` 항목은 `{"sheet":"wp","product":"프로젝트 계획서","no":2,"comment":"..."}` — **WP의 No는 산출물 간 중복되므로 반드시 `product`로 구분**(미지정 시 엉뚱한 행·중복 행에 쓰인다).

대상별 깊이:
- **본문형 docx/xlsx**(계획서류·시스템 아키텍처 명세서·단위 테스트 계획서·단위 검증 레포트·QAC 정적분석·모듈리스트·이해관계자 명세서 등) → 본문 정독으로 즉시 사람급. 절 번호·표·필드를 직접 인용한다.
- **codebeamer export docx**(요구사양서·아키텍처/상세 설계서·시스템요구) → 구조/식별/상태는 export로, 추적은 `trace_checker.py`로, **결함은 같은 폴더의 검토결과서**로 사람급 작성. Description 내용 품질은 커넥터 없으면 "확인 필요".
- **폴더에 실문서 없음**(일부 통합/검증 계획서 등) → "확인 필요 — 폴더 외 문서(codebeamer/PMS)"로 정직히 남기고 자동 Pass 금지.

**Process Checklist도 동일하게 전수 심층화한다(빼먹기 쉬운 함정).** Process는 "활동이 절차대로 수행됐는가"라 근거가 산출물 내용이 아니라 수행 사실이다. 근거 위치를 구분해 적는다: (a) 폴더 회의록(개발 점검·SPQA·개발 진행 현황 점검 문서), 검토결과서 개정 반영, 계획서 참조 절, 관리대장 → 정독해 사람급으로, (b) 사내 시스템(PMS WBS 승인·codebeamer 배포·레드마인 등록/상태) → 1차 템플릿이 아니라 "어느 시스템의 무엇을 확인해야 하는지" 구체적으로 적은 `(AI·확인필요)`로. 또한 **사람이 No(미대상)로 본 활동(테스트 단계말·변경 미발생·통합관리 제외·상위 단계 등)을 1차 자동채움이 Yes로 과대 스코핑**하는 일이 잦다 — 최종 확인에서 이를 잡아 `Applicable=No`로 정렬하고 사유를 단다(코멘트 추가가 아니라 스코핑 교정이 정답).

한계: 비트/정의 수준 일관성, 인터뷰·메일, 시스템(PMS/codebeamer) 사실은 사람/커넥터 몫. 자세한 기준은 `references/comment_style.md`.

## 정직성 원칙 (반드시 지킬 것)

자동 점검은 보조이지 대체가 아니다. 다음을 어기면 잘못된 합격이 품질 리스크가 된다.
- 모든 자동 판정에 `(AI 1차)` 표기와 결과 파일에 안내 시트를 남긴다.
- `Pass`는 "문서에 해당 절·항목이 존재함" 수준이며 **내용 적정성은 미검증**임을 명시한다.
- 일관성·추적성·승인처럼 깊은 내용/시스템 근거가 필요한 항목은 절대 자동 Pass 하지 말고 `확인 필요`로 남긴다.
- 임의로 `Fail`을 생성하지 않는다(근거 없는 Fail 금지). 미확인은 `확인 필요`로 둔다.

## 컬럼 레이아웃 주의

스크립트의 `config.py`에 시트명·컬럼 인덱스가 정의되어 있다. 체크리스트 양식이 개정되면(열 추가/삭제) 먼저 `config.py`를 양식에 맞춰 갱신해야 한다. 머리글 행과 실제 데이터 행이 병합 셀로 한 칸 어긋날 수 있으니, 새 양식 적용 시 `make_blank_template.py`가 출력하는 "문항 칸" 검증 줄로 컬럼이 맞는지 확인하라.

## 추적성 자동 점검 (codebeamer export, 네트워크 불필요)

codebeamer Traceability Browser export(`TraceabilityBrowser-*.xlsx`)가 폴더에 있으면, 추적성·일관성·할당 항목을 사내망 접근 없이 판정 근거로 쓸 수 있다. 이 영역이 위험 오판(존재성만 보고 Pass)이 가장 잦은 곳이라 효과가 크다.

1) 미연결 집계:
```
python scripts/trace_checker.py <대상산출물폴더> --out trace_evidence.json
```
활성 상태(Accepted 등)인데 미연결인 "활성 갭"이 진짜 추적 누락이다. Rejected/Information 미연결은 정상으로 분리한다.

2) 결과 체크리스트에 반영:
```
python scripts/apply_trace_evidence.py <결과.xlsx> trace_evidence.json
```
추적/일관/할당 WP 항목에 "활성 미연결 N건(pair별) — Fail 후보" 근거를 붙인다. **자동 Fail/Pass 하지 않고** 데이터 기반 "확인 필요"로 둬, 사람이 사유(의도적 제외 등)를 빠르게 확정하게 한다. 폴더 export로 시스템 점검의 핵심 영역을 네트워크 없이 1차 처리하는 방법이다.

## 선택: 시스템 커넥터 (사내망에서 일치율 끌어올리기)

사내망에서 실행하면 REST API로 시스템 근거를 끌어와 "확인 필요" 항목을 데이터 기반 판정으로 전환할 수 있다. Process는 점검대상의 ~60%가 시스템(PMS·레드마인·codebeamer) 근거라, 여기 접근이 곧 Process 일치율의 핵심 지렛대다.

- `scripts/connectors/redmine_client.py` — 레드마인 이슈 요약·부적합 ID 검증. 설정·사용법은 `scripts/connectors/README.md` 참고.
- 토큰은 환경변수(`REDMINE_URL`/`REDMINE_API_KEY`) 또는 비커밋 `connectors_config.json`에 둔다. 하드코딩 금지.
- API는 사실 데이터(이슈 존재·상태·추적 관계)를 줄 뿐, 절차 적정성·일관성의 최종 Pass는 규칙/사람 몫이다.
- codebeamer·git·PMS는 동일 패턴으로 확장한다. PMS는 커스텀이라 공개 API 확인이 먼저다.
