# ASPICE 품질점검 체크리스트 자동 작성(오디팅) 스킬

ASPICE 단계말 품질점검 체크리스트를 AI가 1차로 작성하고, 사람 작성본과 비교해 정확도를 측정하면서 **분류 규칙을 반복 개선**하는 워크플로다. 목표는 "AI가 전부 판정"이 아니라 **AI가 자동 판정 가능한 부분(존재성·형식·미대상 스코핑)을 빠르게 처리하고, 사람이 꼭 봐야 하는 부분(일관성·추적성·승인 등)은 명확히 분리해 넘기는 분업**이다.

> **[방향 전환 예고 — `references/roadmap.md` 참조]** 이 스킬은 "사람 답안 모사 판정 엔진"에서 **인간 auditor를 위한 1차 evidence 브리핑 + 점검 의도(Intent) 초안 도구**로 재정의 중이다. Pass = 실제로 정독한 산출물에 Intent 충족 evidence가 명시적으로 존재(표준 위치·프로젝트 특성상 대체 파일/회의록/다른 활동 모두 인정, 근거 인용 필수). Not-Pass면 어느 산출물을 읽어야 하는지 지목(불명확하면 `확인 필요`). **시스템 커넥터(Phase D)는 사내망 무결성 리스크로 보류** — 산출물은 export 복사본을 폴더에 모아 읽기 전용 정독. 상세·로드맵은 `references/roadmap.md`.

## 왜 이 구조인가 — 3계층 점검 모델

체크리스트는 한 장이 아니라 서로 맞물린 3개 시트이고, **산출물명이 셋을 잇는 조인키**다. 이걸 모르면 같은 파일을 시트마다 다시 열고 같은 판단을 반복하게 된다.

| 시트 | 무엇을 보는가 | 점검 단위 |
| --- | --- | --- |
| **Target** | 산출물이 어디 있고 점검 대상인가 | 출력 산출물 ↔ 실제 파일 매핑, 점검대상/미대상 |
| **Process Checklist** | 활동이 절차대로 수행됐는가 | 업무 활동(RACI·실행시점·입력→출력) |
| **WP Checklist** | 산출물 내용이 품질기준을 충족하는가 | 산출물의 ◆관점별 항목 |

> **양식 유연성 — 시트는 있는 것만, 있는 만큼.** 세 시트가 모두 있어야 하는 것은 아니다. Process만 있는 양식(예: BJ1 V5.1), 차수별 Process 시트가 여러 개 누적된 양식(`Process Checklist_LP2 1차`, `Process Checklist_소프트웨어 설계 단계말` …)이 모두 정상 입력이다. 시트마다 컬럼 구성이 다를 수 있으므로(구버전 시트는 Applicable 열 없음) 스크립트가 시트별 3행 머리글로 컬럼을 자동 감지한다(`config.detect_cols`). Target 시트가 없어도 산출물 매핑(2단계)은 코멘트 근거 확보를 위해 그대로 수행한다 — 매핑 결과는 Target 시트 대신 매핑 통합 리포트에만 남는다.

자세한 배경·절차는 `references/methodology.md`, 사내 절차↔ASPICE 대응은 `references/process_mapping.md`를 필요할 때 읽어라. 유지보수(어디만 고치면 되나)는 `references/maintenance.md` 참고.

## 입력과 산출물

- **잣대(읽기 전용)**: ASPICE PAM(폴더에서 참조), 사내 시스템 문서 관리대장(적용 절차서·Rev). 점검의 기준이며 채우지 않는다. PAM 원문은 동봉하지 않고 필요한 프로세스 정수는 `references/process_mapping.md`에 둔다.
- **양식**: 스킬 보유 최신 양식 `assets/latest_template.xlsx` — 양식 버전은 **Template History 시트의 개정번호** 기준이며 `assets/template_version.json`에 기록. 현재 **개정번호 7(26.07.07)** — 26.07.07 확정된 JX 표준 문항 세트(Process 127문항, 시스템 요구/아키텍처 검토 122~127 포함)를 명문화한 개정. 구 문항 세트(121문항, CN8 계열)는 이제 SAME·CONTENT_DIFF로 판정된다. Cover의 문서이력(V1.0/V5.1 등)은 프로젝트 결과물의 문서 버전이지 양식 버전이 아니다. 체크리스트가 첨부되면 `template_version.py`로 비교해 기준 양식을 확정한다.
- **점검 대상**: 오디팅 대상 산출물 폴더(docx/xlsx) + 사내 시스템 URL.
- **산출물**: ① **체크리스트 리포트**(빈 양식에 Target 매핑 + Process/WP 1차 판정 + 사람급 코멘트까지 채운 최종 납품물), ② **매핑 통합 리포트**(매핑 테이블 + 커버리지 한 파일, `build_mapping_report.py`), ③ **AI검증표**(사람 작성본 있을 때, `verify_against_baseline.py` — 스코핑·결과 판정 일치율과 위험오판을 집계하는 **규칙 튜닝용 핵심 분석자료**), ④ **비교분석 검증용**(`build_comparison.py` — 코멘트 텍스트 좌/우 대조, 검증 후 삭제 가능). 그 외 빈 양식·다이제스트·trace_evidence 등은 중간 산출물이다.

> 검증 산출물 3종은 역할이 겹치지 않는다: **AI검증표**=일치율·위험오판 숫자(규칙 튜닝), **비교분석**=코멘트 텍스트 대조(사람급 눈 검토), **verify_comment_coverage**=전 항목 심층 보유 PASS/FAIL 게이트(파일 없음).

## 오디트 시작 대화 흐름 (고정 — 이 순서로 확인 후 작업 시작)

오디트 요청을 받으면 작업 전에 아래를 확인한다. **이미 제공된 것은 건너뛰고, 미확인 항목은 가능한 한 한 메시지에 묶어서 질문한다(왕복 최소화).** 선택지가 있는 질문은 객관식으로 제시한다.

1. **산출물 폴더 (유일한 필수)** — 없으면 요청한다. 폴더 없이는 진행 불가.

2. **체크리스트 첨부 여부** — "이번 점검에 쓸 체크리스트가 있으면 첨부해 주세요. 없이 진행하면 스킬 보유 최신 양식(개정번호는 `assets/template_version.json` 참조)으로 시작합니다"라고 질문한다.
   - **미첨부(그냥 진행)** → 보유 최신 양식 `assets/latest_template.xlsx`로 진행.
   - **첨부** → 먼저 `python scripts/template_version.py <첨부.xlsx> --against assets/template_version.json`으로 버전을 확인하고, **결과를 사용자에게 보여준 뒤 "첨부본으로 할까요, 보유 최신본으로 할까요?"를 질문**한다. 판정별 권장 기본값:
     - **NEWER**(첨부가 더 최신) → 첨부본 권장. 채택 시 작업 완료 후 스킬 기준 양식 갱신('양식 버전 관리' 절차, Save skill 1클릭)을 제안한다.
     - **SAME**(내용까지 동일) → 어느 쪽이든 같으므로 질문 생략 가능, 보유본으로 진행.
     - **SAME·CONTENT_DIFF**(같은 개정번호, 실문항 상이 — 프로젝트 테일러링 가능성) → "이 프로젝트 전용 문항 세트로 보입니다"라고 알리고 선택받는다. 그 프로젝트의 점검이면 첨부본 권장(테일러링 존중). 스킬 양식 갱신은 하지 않는다.
     - **OLDER / UNKNOWN** → 보유 최신본 권장(UNKNOWN·LAYOUT_DIFF면 어느 쪽이 최신인지 확인 후 선택).
   - **예외**: 첨부본에 이미 채워진 완성 시트가 있으면(예: BJ1 — 이전 차수 결과 보유) 그 파일 자체가 작업 대상이므로 위 선택 질문은 생략하고 버전 확인 결과는 정보로만 안내한다. 작업 방식은 3번에서 정한다.

3. **점검 차수 + 기존 결과 처리 (한 질문으로 통합)** — 차수(P1·SP1·LP2, 설계/테스트 단계말 등)는 파일에서 읽어낼 수 없으므로 반드시 묻는다. 차수명은 새 시트 이름(`Process Checklist_<차수명>`)이 되고, 점검대상/미대상(Applicable) 판정 기준이 된다. 첨부 파일에 이미 작성된 점검 결과가 있으면 아래 둘 중 하나를 선택받는다:
   ① **기존 결과 유지** (기본 권장) — 이전 점검 결과는 그대로 두고, 이번 점검 결과를 **새 시트에** 작성합니다.
   ② **기존 결과 지우고 새로 작성** — 이전 점검 결과를 지우고 **같은 시트에** 이번 결과를 작성합니다. (재점검이나 양식 재활용 시. 선택되면 전체를 지울지, 특정 차수 시트만 지울지 확인)
   무응답이면 ①로 진행한다. **사용자 선택 없이 기존 결과를 지우지 않는다.** 작성된 결과가 없는 파일(빈 양식)이면 차수만 확인한다.

4. **사람이 작성한 점검 결과(답안지) 유무** — 이렇게 질문한다: "이번 점검을 품질 담당자가 이미 작성한 체크리스트(완성본)가 있나요?"
   ① **있음** — 파일을 지정받는다. AI 점검 후 사람 결과와 자동 비교해 **검증표**(일치율·위험오판 성적표)와 **비교표**(사람/AI 코멘트 좌우 대조)를 추가로 생성하고, 불일치를 근거로 분류 규칙을 개선한다. (AI 학습·정확도 개선 목적 — 현재 단계)
   ② **없음** — AI 단독 점검. 비교 산출물 없이 자체 품질 게이트(`verify_comment_coverage.py` [PASS])만 통과시켜 리포트를 납품하고, 사용자는 결과 중 '확인 필요' 항목만 확정하면 된다. (궁극적 실전 모드 — 빈 양식만 첨부되는 단계)

5. **결과 저장 위치 (선택)** — "결과를 저장할 폴더를 알려주세요. 지정하지 않으면 산출물 폴더 아래 `_오디트결과_YYMMDD` 폴더를 만들어 저장합니다"라고 안내한다. 무응답이면 기본값으로 진행한다.

확인이 끝나면 즉시 아래 워크플로를 시작한다.

## 워크플로 (반복 루프)

순서대로 수행한다. 각 단계는 `scripts/`의 스크립트로 실행한다.

### 0-0. 실행 환경 자가진단 (가장 먼저, 필수)
일부 환경에서 스킬 파일이 VM 마운트에 **잘린 스냅샷**으로 보이는 문제가 실측됐다(줄 중간 절단, stat 크기까지 잘림, cp도 잘린 채 복사). 작업 시작 직후 반드시 실행한다:
```
python scripts/selfcheck.py
```
`[FAIL]`이거나 **작업 도중 한 번이라도 잘림 증상이 나타나면**(selfcheck 통과 후에도 재발 가능 — 마운트가 읽기마다 다른 뷰를 주는 사례 실측) 아래를 따른다:
① **전량 일괄 복원** — 필요한 스크립트를 하나씩이 아니라 **한꺼번에** Read 툴(호스트 경로)로 읽어, **연결된 폴더**(Write 툴은 연결 폴더에만 쓸 수 있음) 아래 `_audit_work/sc/`에 온전한 사본으로 복원한다. 이후 **마운트 경로의 스크립트는 다시 실행하지 않는다**(selfcheck 통과는 이후 읽기의 안전을 보장하지 않음).
② 복원본으로 실행(PYTHONPATH=_audit_work/sc). 복원본 수정은 Edit 금지 — **새 파일명으로 Write**.
③ **양식 xlsx가 손상 뷰인 경우** — 실측상 대부분 "뒷부분 0-패딩" 유형이라 복구 가능하다. 먼저 자동 복구를 시도한다:
```
python repair_padded_xlsx.py <손상양식.xlsx> <복구본.xlsx>
```
복구 성공 시 시트명 검증 후 그대로 사용. 실패(진짜 잘림)면 사용자에게 **최신 양식 원본 첨부**를 요청하고, 받으면 `make_blank_template.py`로 빈 양식을 재생성해 진행(`template_version.py`로 보유 기록과 버전 일치 확인).
④ 사용자에게 스킬 재설치(Save skill)를 권고하고, 발생 사실을 결과 보고에 남긴다.
발생 배경: 새 세션에서도 발생(실측) — 세션 생성 시 스킬 파일 동기화가 끝나기 전에 마운트가 잡히는 경합으로 추정되며, 파일 끝이 0으로 패딩되어 보이는 것이 전형 증상. selfcheck는 실패 시 자동 대기·재시도(3회×10초)로 일시 지연을 흡수한다.

### 0. 기준 고정
적용할 사내 절차서·표준을 어느 Rev로 볼지 관리대장에서 확정한다(작성이 아니라 식별·고정). 이게 "최신 템플릿/절차 준수" 판정의 기준점이 된다.

### 1. 빈 양식 준비 — 기준은 '최신 양식 버전'(Template History 개정번호)
기준 양식은 시작 대화 2번의 버전 비교로 확정한다: 기본은 스킬 보유 `assets/latest_template.xlsx`, 첨부가 NEWER면 첨부 양식. 같은 개정번호라도 프로젝트마다 실문항이 다를 수 있음이 실측으로 확인됐으므로(CN8 vs JX HEV: Process 23문항 문구 상이, WP 351→338) 문항 차이가 보이면 사용자에게 알린다. **구버전 레이아웃(Applicable 열 없음) 시트에는 신규 작성하지 않는다** — 스크립트가 자동 스킵하고 읽기 전용으로 보존한다.

- **완성본이면**: 답 칸(Applicable·Review Result·Review Comments·Non-conformances ID, Target의 파일/판정)만 비운 빈 양식을 생성해 쓴다.
```
python scripts/make_blank_template.py <완성본.xlsx> <빈양식.xlsx>
```
- **완성 시트를 보존하는 증분 모드(차수 추가/재점검 — 시작 대화 3번에서 확인한 대로)**:
```
# 새 차수: 직전 차수 시트를 복사해 새 시트 생성, 그 시트만 빈 칸 (기존 결과 전부 보존)
python scripts/make_blank_template.py <체크리스트.xlsx> <출력.xlsx> --new-sheet "Process Checklist_<차수명>" --copy-from "<직전 차수 시트명>"
# 재점검: 지정 시트만 클리어
python scripts/make_blank_template.py <체크리스트.xlsx> <출력.xlsx> --sheet "<시트명>"
```
  이후 `fill_checklist.py`·`write_comments.py`·`verify_against_baseline.py`·`verify_comment_coverage.py`·`build_comparison.py`도 같은 `--sheet` 옵션으로 이번 차수 시트만 대상으로 한다.
- **빈 양식이면**: 그대로 복사해 결과 파일로 쓴다.
- 생성/사용한 빈 양식을 작업 폴더 `_기준양식/기준양식.xlsx`에 **백업으로 저장**한다. 이 백업은 사용자가 "기존 것으로 해달라"고 명시했을 때만 쓴다.

점검 시점/차수(P1·SP1·LP2 등)는 양식에 전용 필드가 없어 파싱할 수 없으므로 **사용자에게 확인**하고, 다단계 프로젝트면 **직전 차수 체크리스트**를 요청한다(동일 문항의 30~40%가 차수에 따라 Applicable이 뒤집히므로, 없으면 해당 스코핑은 확인 필요로 남긴다).

### 2. 산출물 매핑 — 반드시 "내용 기반"으로 (필수)
**원칙: 파일명이 아니라 파일 내용으로 매핑한다.** 파일명 키워드만으로 매핑하면 철자(시험/테스트)·버전·대소문자 차이로 누락·오분류가 반복된다. 그래서 **매번 아래 순서를 그대로 밟는다 — 선택이 아니라 기본 절차다.** 모든 판정에 `날짜 (AI 1차)`를 표기한다.

**2-1) 폴더 다이제스트 생성** — 모든 파일(+1단계 하위 폴더)의 시트명·헤딩·첫 행·검출 ID·버전을 한 장으로. **zip 압축파일은 자동 해제해 내용까지 포함**한다.
```
python scripts/summarize_folder.py <대상산출물폴더> --out folder_digest.md --extract-zips <결과폴더>/_압축해제
```
- 해제는 **작업 폴더에만** 한다(원본 산출물 폴더 불변). 해제본은 매핑·심층 정독 대상에 포함.
- **재오디트 공존 방지**: `_`로 시작하는 폴더(`_오디트결과_*`·`_압축해제`·`_기준양식` 등)는 다이제스트·커버리지·매핑 리포트가 **자동 제외**하므로, 이전 오디트의 해제본·결과물이 남아 있어도 중복 집계되지 않는다. 사용자가 zip 옆에 직접 푼 동일 이름 폴더가 있으면 해제를 건너뛰고 그 폴더를 쓴다(중복 방지).
- zip만 자동 지원 — 7z/rar/암호 파일은 해제 실패로 표시되니 사용자에게 수동 해제를 요청한다.

**2-2) AI가 다이제스트를 읽고 매핑을 생성** — `folder_digest.md`와 체크리스트 산출물 목록을 함께 읽고, 각 파일이 어느 산출물인지·점검대상/미대상/보조/입력/잣대인지 **내용 근거**로 판단해 `deliverable_map.json`을 만든다(evidence의 파일 필드는 반드시 실제 파일명, note에 시트·헤딩 근거). 시트명·헤딩이 결정적 단서다(예: 'License_Report' 시트 → 오픈소스 관련). 의미가 모호하면(라이선스 검증 vs 취약점 점검처럼) 점검대상으로 **보이게** 두되 최종 동일성 판정은 사람에게 넘긴다 — 임의로 같다고 단정하지 않는다. 매핑의 출발점으로 **`assets/common_mapping_rules.json`을 먼저 적용**한다 — CN8·JX HEV 두 프로젝트 Target 실측에서 일반화한 83개 규칙(자체파일 SELF / 타 문서 포함 IN_FILE — 예: 위험관리·의사소통·개발비용 계획→프로젝트 계획서 / 시스템 URL SYSTEM / 통상 미대상 USUALLY_NA)과 표기 변형(시험↔테스트, 아키텍처↔아키텍쳐, 형상항목관리대장→파일명 '형상관리대장')이 담겨 있다. 이 규칙으로 1차 후보를 잡되 최종 확정은 반드시 내용 근거로 하고, 새 프로젝트에서 확인된 패턴은 이 파일에 추가한다. `deliverable_map.example.json`은 시작점일 뿐이다.

**2-3) 매핑 검토("테이블 먼저")** — AI는 `deliverable_map.json`의 매핑 근거를 사용자에게 제시해 먼저 검토받는다(임의 단정 금지). 매핑을 파일로 남기는 건 2-6의 통합 리포트(매핑 테이블 시트)로 일원화한다 — 더 이상 별도 매핑테이블 파일을 따로 만들지 않는다.

**2-4) 누락 점검**
```
python scripts/coverage_check.py <대상산출물폴더> deliverable_map.json
```
'미매핑'이면 (1) 점검대상인데 누락 → 맵 보강, (2) 보조/입력/잣대(BJ1·타프로젝트·지침서·표준) → 무시. 사람 작성본이 있으면 그 Target 파일명이 추가 근거다.

**2-5) 채우기**
```
python scripts/fill_checklist.py <빈양식.xlsx> <결과.xlsx> --map deliverable_map.json
```

**2-6) 매핑 통합 리포트(한 파일)** — 매핑 테이블 + 커버리지(파일 관점/체크시트 관점)를 **중복 없이 한 파일**로 낸다.
```
python scripts/build_mapping_report.py deliverable_map.json <결과.xlsx> <대상산출물폴더> <매핑_통합리포트.xlsx>
```
시트는 `요약`·`매핑 테이블`·`파일 관점`·`체크시트 관점` 4개. (과거 build_mapping_table.py·mapping_report.py가 '파일 사용여부' 시트를 중복 생성하던 것을 하나로 합친 스크립트다.)
매칭은 가장 구체적인(긴) 키가 이긴다(일반 키 '회의록'이 '검토 회의록'을 가로채지 않음). 하위 폴더도 한 항목으로 인식한다. 리포트 정확도는 맵의 파일 필드가 실제 파일명일 때 보장된다(설명 문구 금지).

판정 원칙(evidence 브리핑 모드 — `references/roadmap.md`, `references/comment_style.md §10~13`):
> **Pass는 실제로 정독한 산출물에 Intent 충족 근거가 명시적으로 존재하고 그것을 인용할 때만** 부여한다(표준 위치·프로젝트 특성상 대체 파일/회의록/다른 활동 모두 인정). Pass가 아니면 **읽어야 할 산출물을 지목**하고, 불명확하면 `확인 필요` 수준으로 남긴다. `judge()`는 자동 Pass를 내리지 않고 스코핑·힌트만 제공한다.
- **판정값 어휘**: `Pass` / `Fail` / `Conditional pass`(조건부 합격 — 합격 계열) / 빈칸(확인 필요). 일치율 비교에서 Pass와 Conditional pass는 합격 계열(`config.PASS_FAMILY`)로 같은 묶음으로 취급한다. 단 AI가 스스로 Conditional pass를 판정하지는 않는다 — 사람 작성본을 읽고 비교·보존하기 위한 어휘다.
- **미대상**(테스트 단계말·단계 미도래·해당없음) → `Applicable=No` + 사유. 가장 객관적이므로 먼저 처리.
- **존재성·형식 항목**(정의/포함/작성/구조/템플릿) → 본문 정독 계층에서 **근거(문서·절) 인용 시 `Pass`**. 키워드만으로 자동 Pass 금지(구 규칙 폐기).
- **사람 필수 항목**(승인·배포·협의·고객·추적·일관·정합·동료검토·이력·등록·모니터링) → 기본은 `확인 필요`(자동 Pass 금지). 단 **표준 위치 또는 대체 산출물/활동에서 Intent 충족 근거를 명시적으로 정독·인용하면 Pass 가능**(comment_style §11). 근거 없으면 읽어야 할 산출물을 지목한다.
- **사내 시스템 근거 항목**(PMS·레드마인·codebeamer·git URL) → `확인 필요`. 접근 불가.
- **Process Yes/No(스코핑)**: 실행시점이 후행단계(마일스톤·완료·M-1 등 `LATER_PHASE_TIMING`)면 `No`, 미대상이면 `No`, **조건부 활동**(실행시점 '주기 도래/발생 시' 또는 사건발생형 활동[형상감사·CCB·변경요청] + 폴더에 수행 흔적 없음)은 Applicable을 비우고 `[조건부 활동] 발생 여부 확인 필요`로 남긴다(일괄 Yes가 과대 스코핑의 주원인 — CN8 실측 27건). **상시 운영 활동**(문제해결·이슈·레드마인·모니터링·의사소통)과 **계획서류**는 시스템 근거·후행 단계 사유로 미대상 처리 금지(게이트가 검사). 그 외 활동은 `Yes`. 실행시점 칸이 비어 있으면 미대상 패턴에만 의존하므로, 활동 '미수행'으로 인한 No는 사람이 확정한다.
- **Process 판정 제안(4단계, 필수)**: 점검대상 코멘트에 `판정: Pass`(폴더 완결 근거) / `Pass 후보`(산출물·본문 확인 완료, 시스템 승인만 잔여 — 단 **일관성·매핑 대사 요구 문항과 특정 검토 활동 요구 문항은 직접 근거 없이 Pass 계열 금지**, `comment_style.md` §7) / `Fail 후보`(파일 부재/추적 활성 갭/NG 미반영 3종 한정 — 내용 요소 미발견은 확인 필요) / `확인 필요`(순수 시스템 근거) 중 하나를 반드시 단다. 게이트가 태그 없는 코멘트를 비심층 처리한다. 부재 확인 시 Fail 후보는 의무다.
- **Result 직접 기록(AI 1차 의견)**: 태그와 함께 Review Result 칸에도 1차 의견을 기록한다 — `판정: Pass`·`Pass 후보`→**Pass**, `Fail 후보`→**Fail**, `확인 필요`→**확인 필요**. 미대상·조건부는 빈칸. **최종 확정은 사람**이며 안내 시트에 이를 명시한다(근거 수준 구분은 코멘트 태그가 담당). 게이트가 태그↔Result 불일치를 검사하고, 검증표는 '확인 필요'를 미판정으로 집계한다.

### 3. 정합성 점검
미대상인데 결과가 있음, 코멘트 누락, Fail인데 부적합ID 없음 등 모순을 검사한다(`fill_checklist.py`가 마지막에 자동 수행).

### 4. 검증표 생성 (사람 작성본이 있을 때)
같은 양식의 사람 작성본과 행 단위로 비교해 정확도를 측정한다.
```
python scripts/verify_against_baseline.py <결과.xlsx> <사람작성본.xlsx> <검증표.xlsx>
```
두 축으로 본다: **① 스코핑 정확도**(Applicable Yes/No 일치율)와 **② 결과 판정 정확도**(사람이 Pass/Fail 한 항목 중 일치/미판정/위험오판). **위험 오판(사람 Fail인데 AI Pass)** 을 최우선으로 본다. 콘솔에 **제안(태그) 정확도**(Pass 후보/Fail 후보 제안 vs 사람 판정 일치율)도 함께 출력된다 — '후보'는 Result를 비우는 정책이므로 Result 일치율 대신 이 지표로 제안 품질을 추적한다(3차 실측: Process 제안 정확도 96%).

**오염 검사(필수)** — 검증표와 함께 반드시 실행한다. AI 코멘트가 사람 코멘트를 근거 없이 복제(참조 오염)하면 일치율이 실력 지표가 아니게 된다(CN8 실측: 복제·사실 전이·명시 참조 46건).
```
python scripts/check_contamination.py <결과.xlsx> <사람작성본.xlsx>
```
[HIGH]=복제 의심, [FACT]=시스템 사실 전이, [REF]=사람 의견 명시 참조. 플래그 항목은 폴더 근거로 재작성하거나 '확인 필요'로 교정한 뒤 재검사한다.

### 5. 규칙 튜닝 → 다시 2번 (핵심 개선 루프)
검증표의 불일치를 보고 `classification_rules.py`를 고친다. 전형적 개선:
- **AI 과소(누락)**: 미대상으로 잘못 뺀 산출물 → `NA_KEYWORDS`에서 해당 패턴을 빼거나 매핑을 추가.
- **위험 오판**: 자동 Pass 했는데 사람은 Fail → 그 문항의 키워드를 `HUMAN_REQUIRED`에 추가해 확인 필요로 강제.
- **AI 미판정 과다**: 시스템 접근이 가능해지면 해당 근거를 매핑에 넣어 판정 가능 항목으로 전환.
- **검증/시험 '계획서'를 미대상으로 빠뜨림(과소)**: `NA_KEYWORDS`('통합 검증' 등)가 산출물명에 부분매칭돼 *계획서*까지 No로 떨군다. `classification_rules.is_design_phase_target`(`DESIGN_PHASE_KEEP`: 계획·정적·설계서·아키텍처·설계 명세·**단위**)로 점검대상을 유지한다 — 계획서·**단위(Unit) 검증 명세서/결과서·정적 분석**=설계 단계 점검대상(3차 실측 과소 21건 교훈), 통합/시스템 검증의 명세서/결과서=테스트 단계말 미대상. 또 매핑된 파일이 없어도 미대상이 아니면 `Applicable=Yes`(확인 필요)로 둔다(점검대상≠검증가능).

규칙을 고치면 2번을 다시 돌려 검증표로 일치율 변화를 확인한다. 이 루프를 일치율이 만족스러울 때까지 반복한다.

### 6. 최종 확인 (필수 게이트 — 통과 못 하면 넘기지 않는다)
**납품 규칙**: 최종 보고에 게이트 결과를 반드시 명시한다("게이트 [PASS] — 점검대상 N건 전수 심층"). `[FAIL]` 상태로는 납품하지 않는 것이 원칙이며, 시간 제약 등으로 부득이 중단할 때는 **파일명에 `_미완`을 붙이고** 보고에 "게이트 미통과 — 비심층 N건, 심층화 계속 필요"를 명시한다(실측: JX 1차본이 비심층 25건 상태로 상태 표기 없이 전달된 사례 재발 방지).
심층 코멘트 주입 후, 리포트를 사용자에게 넘기기 **전에 반드시** 전수 커버리지를 검사한다. 점검대상(Applicable=Yes) **WP·Process 항목이 하나도 빠짐없이** 사람급(AI 심층) 코멘트를 가졌는지, 사람 작성본이 있으면 사람이 코멘트한 항목 대비 비심층/빠짐이 0인지 확인한다.
```
python scripts/verify_comment_coverage.py <결과.xlsx> [사람작성본.xlsx] --require-intent
```
> **evidence 브리핑 게이트(Phase C)**: `--require-intent`를 주면 점검대상 코멘트에 `[의도]` 라인이 있는지도 검사한다. 또한 항상 (a) **Pass/Pass후보인데 인용 근거 없음**, (b) **확인 필요인데 읽어야 할 산출물 미지목**을 `[스코핑 재검토 필요]`로 잡아 `[FAIL]` 처리한다(comment_style §11·§12).

- `[PASS]`(WP·Process 전 항목 심층, 사람 대비 0건)가 나올 때까지 반복한다. 스코핑 교정(Yes→No)은 **사람 작성본·직전 차수 근거가 있을 때만** 허용된다 — 그 근거로 사람이 No로 본 항목을 Yes로 잡은 경우 `Applicable=No`로 정렬한다. `[FAIL]`이면 출력된 '빠짐/비심층' No의 산출물 본문·검토결과서를 다시 정독해 코멘트를 작성·주입하고 재검사한다.
- '확인 필요(문서 부재)'로 정직히 남긴 항목은 심층으로 인정된다(환각 대신 정직한 한계 표기가 정답이다). 단 그 사유가 코멘트에 명시돼 있어야 한다. **심층 인정에는 마커 외에 근거 인용(파일명·절·표·시트·버전 또는 정직한 한계 표기)이 필수**이며, 날짜 마커의 연도 오기도 게이트가 잡는다.
- 고위험(누락 시 잘못된 합격) 점검이므로, 가능하면 이 검사를 **별도 에이전트로 한 번 더 교차 확인**(샘플 항목의 코멘트가 실제 본문/검토결과서 근거와 일치하는지)한 뒤 넘긴다.

**비교표는 항상 함께 만든다(검증 산출물).** `[PASS]`를 받은 뒤, 사람이 사람급 도달 여부를 눈으로 대조할 수 있도록 사람 코멘트(좌)/AI 코멘트(우) 비교표를 반드시 생성한다.
```
python scripts/build_comparison.py <결과.xlsx> <사람작성본.xlsx> <비교분석_사람급_전체_검증용.xlsx>
```
- 출력은 요약 시트(산출물별 사람Yes/AI/심층/비고) + 상세 비교 시트(No·항목·사람·AI)로 구성된다.
- **이 비교표는 검증용 임시 산출물이다.** 최종 납품물은 체크리스트 리포트뿐이므로, 파일명에 `_검증용`을 붙여 구분하고 사용자에게는 검증이 끝나면 삭제해도 되는 파일임을 명시한다. 사람 작성본이 없으면(신규 점검) 이 단계는 생략하고 `verify_comment_coverage.py`의 리포트 단독 [PASS]로 갈음한다.

## Intent 지식베이스 & 대체 근거 탐색 (Phase B·C — references/roadmap.md)

문항 문구가 아니라 **문항이 확인하려는 목적(Intent)** 중심으로 점검하고, 표준 위치에 없어도 **대체 근거**를 폴더에서 찾아 Pass를 인정(인용 필수)하기 위한 도구다. `references/comment_style.md §10~13`과 함께 쓴다.

**B) Intent 지식베이스(`intent_map.json`)** — 문항별 `{intent, pam_ref, focus, depth, primary_location, alt_evidence}`. 스키마는 `assets/intent_map.schema.json`, 예시는 `intent_map.example.json`.
```
# 체크리스트에서 문항을 추출해 초안 생성(빈 칸) → AI가 intent 등을 채운다
python scripts/build_intent_map.py <체크리스트.xlsx> --out intent_map.json
# 양식 개정/재점검 시 기존에 채운 값 보존
python scripts/build_intent_map.py <체크리스트.xlsx> --out intent_map.json --merge <기존 intent_map.json>
```
채운 뒤 코멘트 머리 `[의도]` 라인은 `intent_lib.IntentMap.intent_line(sheet,no,product)`로 만든다(WP는 No가 산출물마다 중복되므로 product로 정확 매칭). 초안→검수 누적 방식은 `common_mapping_rules.json`과 동일 철학이다.

**C) 대체 근거 탐색(`evidence_search.py`)** — 1차 표준 위치에 없을 때, 프로젝트 특성상 대체된 근거(별도 파일·회의록·다른 활동)를 폴더 다이제스트에서 찾는다. **판정하지 않고 후보만 제시** — 최종 확인은 본문 정독.
```
# intent_map의 특정 문항 대체근거 후보(문구+토큰 검색)
python scripts/evidence_search.py folder_digest.md --intent intent_map.json --sheet wp --no 5 --product "프로젝트 계획서"
# 임의 키워드
python scripts/evidence_search.py folder_digest.md --kw 위험관리 --kw 리스크 --kw 회의록
```
후보가 있으면 그 파일을 정독해 Intent 충족 시 **Pass(어느 산출물·활동이 근거인지 인용)**, 없으면 `확인 필요 — 폴더에서 식별되지 않음`으로 남긴다.

## 심층(필수): 사람급 Review Comment — WP·Process 점검대상 전수 정독

> **이 단계는 선택이 아니라 필수다. 시간·토큰에 관계없이 점검대상(Applicable=Yes) 항목을 WP Checklist(산출물 내용)와 Process Checklist(활동 수행) 둘 다 하나도 빠짐없이 정독해 항목별 코멘트를 작성한다.** 기본 채우기(2-5)의 1차 템플릿 코멘트는 사람 코멘트(절 번호 인용 + 미흡점 + 권고)와 격차가 크므로, 리포트를 넘기기 전에 **반드시 전수 심층화**한다. 결정론적 스크립트로는 불가능 — AI가 본문을 읽고 직접 쓴다. "핵심만"이 아니라 "전부"다.

**원칙: 빠짐없이, 정직하게, 그리고 독립적으로.** 필수 규칙(상세·예시는 `references/comment_style.md` 하단):
- **하위 질문 분해**: 문항의 1, 2, 1-1 등 하위 질문 각각에 대해 확인/미확인/확인불가를 기술.
- **Process 3단 판정**: ① 수행 필요했는가 → ② 수행됐는가 → ③ 증거 품질 충분한가.
- **Fail 후보 명시**: 근거로 미수행이 확인되면 코멘트에 "Fail 후보"를 명시(판정 회피 금지, Result 확정은 사람).
- **NA 개별 사유**: No 판정마다 산출물명·실행시점 인용한 개별 사유. 산출물이 폴더에 있으면 존재 사실 병기.
- **사람 코멘트 참조 금지**: 같은 차수 사람 작성본은 점검 중 근거로 쓰지 않는다. 폴더에 없는 시스템 사실 전이 금지.
- **날짜 마커**: 당일 YYMMDD(게이트가 연도 자동 검사).
- 점검대상 산출물 **본문**과 그 **검토결과서**를 전부 연다. 사람 코멘트의 구체적 결함은 codebeamer가 아니라 폴더의 검토결과서(`*_검토결과서_*.xlsx`, `*_Review_Report_*.xls`, `*_동료검토_*.xls`)의 '검토 결과'·'결함 내용'·Check List(NG) 시트에 들어 있다. 이걸 안 읽으면 사람급이 될 수 없다.
- 근거가 폴더에 **없으면**(codebeamer export의 빈 Description, 폴더에 실문서 부재) 절대 지어내지 말고 `확인 필요 — 문서 부재/codebeamer 접근 필요`로 남긴다. 자동 Pass 금지.

절차(점검대상 산출물마다 반복 — 한 산출물도 건너뛰지 않는다):
1) 대상 + 검토결과서 본문 추출:
```
python scripts/extract_deliverable_text.py <산출물파일> --out text.md
```
xls/xlsx 검토결과서는 시트별로 직접 읽어 결함·NG(예: 동료검토 Check List의 NG 행, '결함 내용' 시트의 검증항목 ID·심각도)를 수집한다.
2) AI가 `text.md`와 검토결과서를 읽고, 그 산출물의 각 WP 문항에 `references/comment_style.md` 형식으로 코멘트를 작성한다 → `comments.json`. **머리는 반드시 AI 정독 마커로 시작**한다: `YYMMDD (AI·본문정독)` / `(AI·결과서 정독)` / `(AI·검토결과서)` / `(AI·추적데이터)` / `(AI·구조확인)` / `(AI·확인필요)`. 이 마커가 있어야 최종 확인 게이트가 '심층'으로 인정한다. 형식: `{절/표/필드 근거} {확인}. 다만 {미흡점}. {권고/확인 필요}`. Pass여도 미흡점을 기록하고, 도메인 적정성·시스템 사실은 단정 말고 "확인 필요"(환각 금지).
3) 주입:
```
python scripts/write_comments.py <결과.xlsx> comments.json
```
`comments.json` 항목은 `{"sheet":"wp","product":"프로젝트 계획서","no":2,"comment":"..."}` — **WP의 No는 산출물 간 중복되므로 반드시 `product`로 구분**(미지정 시 엉뚱한 행·중복 행에 쓰인다).

대상별 깊이:
- **본문형 docx/xlsx**(계획서류·시스템 아키텍처 명세서·단위 테스트 계획서·단위 검증 레포트·QAC 정적분석·모듈리스트·이해관계자 명세서 등) → 본문 정독으로 즉시 사람급. 절 번호·표·필드를 직접 인용한다.
- **codebeamer export docx**(요구사양서·아키텍처/상세 설계서·시스템요구) → 구조/식별/상태는 export로, 추적은 `trace_checker.py`로, **결함은 같은 폴더의 검토결과서**로 사람급 작성. Description 내용 품질은 커넥터 없으면 "확인 필요".
- **폴더에 실문서 없음**(일부 통합/검증 계획서 등) → "확인 필요 — 폴더 외 문서(codebeamer/PMS)"로 정직히 남기고 자동 Pass 금지.

**Process Checklist도 동일하게 전수 심층화한다(빼먹기 쉬운 함정).** 각 코멘트에는 판정 제안 태그(판정: Pass / Pass 후보 / Fail 후보 / 확인 필요)를 반드시 포함한다 — `references/comment_style.md` §7. Process는 "활동이 절차대로 수행됐는가"라 근거가 산출물 내용이 아니라 수행 사실이다. 근거 위치를 구분해 적는다: (a) 폴더 회의록(개발 점검·SPQA·개발 진행 현황 점검 문서), 검토결과서 개정 반영, 계획서 참조 절, 관리대장 → 정독해 사람급으로, (b) 사내 시스템(PMS WBS 승인·codebeamer 배포·레드마인 등록/상태) → 1차 템플릿이 아니라 "어느 시스템의 무엇을 확인해야 하는지" 구체적으로 적은 `(AI·확인필요)`로. **심층 단계에서 스코핑 강등(Yes→No)은 원칙적으로 금지다.** 실측(2차 계획서류 17건, 3차 단위 검증류 21건)에서 AI가 매번 새로운 '후행 단계' 일반화를 만들어 점검대상을 잘못 뺐다. 규칙: (a) **블라인드 점검(사람 작성본 없음)에서는 강등 금지** — 미대상이 의심되면 Applicable은 Yes로 두고 코멘트에 `[스코핑 검토 제안] ○○ 사유로 미대상 가능성 — 사람 확정 필요`를 덧붙인다. (b) 강등은 fill 규칙(실행시점 후행·NA 키워드·조건부)과 **사람 작성본/직전 차수 근거가 있을 때만** 한다. (c) 단위 검증·정적 분석·계획서류·상시 운영 활동은 어떤 경우에도 설계 단계말 점검대상이다(게이트가 검사).

한계: 비트/정의 수준 일관성, 인터뷰·메일, 시스템(PMS/codebeamer) 사실은 사람/커넥터 몫. 자세한 기준은 `references/comment_style.md`.

## 정직성 원칙 (반드시 지킬 것)

자동 점검은 보조이지 대체가 아니다. 다음을 어기면 잘못된 합격이 품질 리스크가 된다.
- 모든 자동 판정에 `(AI 1차)` 표기와 결과 파일에 안내 시트를 남긴다.
- `Pass`는 "문서에 해당 절·항목이 존재함" 수준이며 **내용 적정성은 미검증**임을 명시한다.
- 일관성·추적성·승인처럼 깊은 내용/시스템 근거가 필요한 항목은 절대 자동 Pass 하지 말고 `확인 필요`로 남긴다.
- 임의로 `Fail`을 생성하지 않는다(근거 없는 Fail 금지). 미확인은 `확인 필요`로 두되, 근거로 미수행이 확인되면 코멘트에 "Fail 후보"를 명시한다.
- **같은 차수 사람 작성본은 점검 중 근거로 쓰지 않는다(참조 오염 금지).** 사람 작성본은 검증 단계(검증표·비교표·오염 검사)에서만 연다. 폴더에 없는 시스템 사실을 사람 코멘트에서 옮겨 적는 것을 금지한다.

## 양식 버전 관리 (최신 하나만 보유)

- **양식 버전 = Template History 시트의 최대 개정번호.** Cover 문서이력은 쓰지 않는다.
- 스킬은 최신 양식 **하나만** 보유한다: `assets/latest_template.xlsx` + `assets/template_version.json`(개정번호·일자·레이아웃 지문). 과거 양식으로 신규 점검은 하지 않는다.
- **갱신 절차(새 최신 양식 접수 시) — 자동화됨**: 사용자가 "이게 최신 양식으로 확정됐다"고 전달하면 한 명령으로 처리한다:
```
python scripts/adopt_template.py <새양식.xlsx>
```
  자동으로 ① 빈 양식 생성(답 칸 클리어 + 증적 이미지 제거 + Cover의 프로젝트명·문서이력 정리) ② **이전 보유 양식과 문항 자동 대조**(Process/WP 추가·삭제 건수, Target 산출물 증감) ③ **Template History에 개정 행 자동 등재** — 첨부본이 더 높은 개정번호를 정식 기재했으면 그대로 채택, 내용이 바뀌었는데 번호가 그대로면(이력 누락) 다음 번호로 등재(작성/검토/승인 공란 — 공식 승인 후 기입 안내) ④ `latest_template.xlsx`·`template_version.json` 갱신. 변경 요약을 사용자에게 보여주고, 마지막으로 `.skill` 재패키징 후 Save skill 클릭을 안내한다. 스킬 폴더는 실행 중 읽기 전용이라 최종 반영은 **재패키징+재설치**로만 가능하다.

## 컬럼 레이아웃 주의

컬럼 위치는 `config.detect_cols()`가 각 시트의 3행 머리글을 읽어 **시트별로 자동 감지**한다. 같은 워크북 안에서도 시트마다 레이아웃이 다를 수 있다 — 예: BJ1 V5.1의 구버전 `Process Checklist` 시트는 Applicable 열이 없다(이 경우 `cols["applicable"]=None`이 되며 **신규 작성(클리어·채우기)은 자동 스킵**, 읽기(검증표·커버리지·비교표)에서는 Result 존재로 Yes/No를 추정). 머리글이 읽히지 않는 특이 양식은 `config.py`의 고정값 폴백을 쓰므로 그때만 고정값을 갱신하면 된다. 새 양식 적용 시 `make_blank_template.py`가 출력하는 "문항 칸" 검증 줄로 컬럼이 맞는지 확인하라.

## 추적성 자동 점검 (codebeamer export, 네트워크 불필요)

codebeamer Traceability Browser export(`TraceabilityBrowser-*.xlsx`)가 폴더에 있으면, 추적성·일관성·할당 항목을 사내망 접근 없이 판정 근거로 쓸 수 있다. 이 영역이 위험 오판(존재성만 보고 Pass)이 가장 잦은 곳이라 효과가 크다.

1) 미연결 집계:
```
python scripts/trace_checker.py <대상산출물폴더> --out trace_evidence.json
```
활성 상태(Accepted 등)인데 미연결인 "활성 갭"이 진짜 추적 누락이다. Rejected/Information 미연결은 정상으로 분리한다.

2) 결과 체크리스트에 반영:
```
python scripts/apply_trace_evidence.py <결과.xlsx> trace_evidence.json
```
추적/일관/할당 WP 항목에 "활성 미연결 N건(pair별) — Fail 후보" 근거를 붙인다. **자동 Fail/Pass 하지 않고** 데이터 기반 "확인 필요"로 둬, 사람이 사유(의도적 제외 등)를 빠르게 확정하게 한다. 폴더 export로 시스템 점검의 핵심 영역을 네트워크 없이 1차 처리하는 방법이다.

## 선택: 시스템 커넥터 (사내망에서 일치율 끌어올리기)

사내망에서 실행하면 REST API로 시스템 근거를 끌어와 "확인 필요" 항목을 데이터 기반 판정으로 전환할 수 있다. Process는 점검대상의 ~60%가 시스템(PMS·레드마인·codebeamer) 근거라, 여기 접근이 곧 Process 일치율의 핵심 지렛대다.

- `scripts/connectors/redmine_client.py` — 레드마인 이슈 요약·부적합 ID 검증. 설정·사용법은 `scripts/connectors/README.md` 참고.
- 토큰은 환경변수(`REDMINE_URL`/`REDMINE_API_KEY`) 또는 비커밋 `connectors_config.json`에 둔다. 하드코딩 금지.
- API는 사실 데이터(이슈 존재·상태·추적 관계)를 줄 뿐, 절차 적정성·일관성의 최종 Pass는 규칙/사람 몫이다.
- codebeamer·git·PMS는 동일 패턴으로 확장한다. PMS는 커스텀이라 공개 API 확인이 먼저다.
