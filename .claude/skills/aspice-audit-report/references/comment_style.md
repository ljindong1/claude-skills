# Review Comment 작성 스타일 — 사람급 (본문 정독 심층 모드)

> ## ⚠ 가장 중요 — 검토결과서를 반드시 함께 읽어라
> 사람 코멘트의 **구체적 결함**(예: "인터페이스 테이블 ISLA 입력명 누락", "컴포넌트별 DFD/CFD 정합성 불일치")은 codebeamer가 아니라 **폴더에 있는 동료검토 검토결과서**(`*_검토결과서_*.xlsx`, `*_Review_Report_*.xls`, `*_동료검토_*.xls`)의 **'검토 결과'·'결함 내용' 시트**에 들어 있다. 산출물 본문(특히 codebeamer export로 Description이 빈 경우)만 읽고 "확인 필요"로 넘기면 사람 수준에 절대 도달하지 못한다.
> **각 점검대상 산출물마다, 짝이 되는 검토결과서를 찾아 결함(검증항목 ID·결함 내용·심각도·상태)을 읽고 해당 WP 문항에 인용하라.** 이게 사람급의 핵심 출처다.


사람 감사자의 코멘트를 모사하려면 **산출물 본문을 정독**하고 항목마다 아래 형식으로 쓴다.
결정론적 스크립트(judge())로는 불가능하다 — AI가 `extract_deliverable_text.py` 결과를 읽고 직접 작성한다.
토큰·시간이 많이 들므로, 정밀 점검이 필요한 핵심 산출물에 적용한다.

## 형식
`YYYY-MM-DD : {근거 위치} {확인 내용}. 다만 {미흡점/한계}. {권고/확인 필요}.`

## 필수 요소 (사람 코멘트에서 추출한 패턴)
1. **날짜 prefix**. 재검토하면 날짜를 누적한다("2026-05-08, 2026-05-24 : …").
2. **근거 위치를 정확히**: 절 번호('절 2.3.3'), 표/필드명('표 회의 체계', 'WBS 선후행 필드', '이해관계자 요구사항 Priority 필드'), URL, 교차문서('품질보증계획서 절 3.1').
3. **확인 동사**: "~을 확인함", "~로 정의됨".
4. **Pass여도 미흡점·한계를 기록**(Pass ≠ 완벽): "다만 AUTOSAR/MCAL 역할 범위가 명확히 문서화되어 있지 않음".
5. **권고/확인 필요**: "~ 정의 권고", "~ 직접 확인 필요".
6. **본문에 근거가 없으면 솔직히**: "상세 WBS는 PMS로 관리되어 계획서 본문 미수록 → PMS 직접 확인 필요".

## 금지
- 근거 위치 없는 일반 문구("문서 내 해당 항목 존재·충족 확인") 금지.
- 도메인 적정성("이 정도면 충분한가")을 단정하지 말 것 — 미흡점은 기록하되 최종 합격(Pass 확정)은 사람.
- 본문에 없는 내용을 있다고 쓰지 말 것(환각 금지).

## 사람 몫으로 남는 한계 (정직하게)
- 비트/정의 수준 내용 일관성 비교(추적성은 `trace_checker.py`로 일부 데이터화 가능).
- 인터뷰·메일 등 비문서 근거.
- 시스템(PMS·codebeamer·레드마인) 사실 — 커넥터 없으면 "확인 필요".

## 절차
1. `extract_deliverable_text.py <파일> --out text.md` 로 본문 전체(절 구조) 추출.
2. AI가 text.md를 읽고, 해당 산출물의 각 WP 문항에 위 형식으로 코멘트 작성 → `comments.json`.
3. `write_comments.py <결과.xlsx> comments.json` 로 결과 체크리스트에 주입.

## 산출물 두 종류 — 본문형 vs codebeamer export (중요)

사람급 코멘트 가능성은 산출물 형태로 갈린다.
- **본문형 docx**(프로젝트/형상/문제해결/변경/품질보증 계획서, 단위 테스트 계획서, 시스템 아키텍처 명세서 v1.0): 절·표에 실제 내용 있음 → **본문 정독으로 즉시 사람급 가능.**
- **codebeamer export**(요구사양서·아키텍처·시스템요구·상세설계 = "Export of ..."): 추출하면 모든 `Description:`이 비어 있음. export엔 **절 구조 + ID(SWR-/SWA-/SysRS-) + Status + Associations**만 남음. 실제 내용은 codebeamer에 있다.

### export 문서에서 export만으로 점검 가능한 것
1. 구조화/분류(절 구조), 2. 고유 식별(ID), 3. 추적 관계(Associations + TraceabilityBrowser → `trace_checker.py`), 4. 상태(Status; 대부분 Draft/In Review = 미승인).

### export만으로 불가 → codebeamer 필요
명확성·검증가능성·모순·일관성·HSI 반영·할당 적절성 등 내용 품질. → codebeamer 커넥터(`/api/v3/items`로 Description 수집, redmine 커넥터와 동일 패턴)로 내용을 끌어와야 사람급 작성 가능. 접근 없으면 "확인 필요"가 정직한 한계.

즉 사람급의 병목은 AI 작성력이 아니라 **내용 접근**이다. 내용에 닿으면 사람급, 닿지 못하면(export 껍데기) 누구도 불가.

## 전수·마커 원칙 (최종 게이트 연동)
- 점검대상(Applicable=Yes) 산출물은 **하나도 빠짐없이** 본문+검토결과서를 정독해 코멘트를 단다.
- 모든 코멘트 머리에 정독 마커를 붙인다: `(AI·본문정독)`/`(AI·결과서 정독)`/`(AI·검토결과서)`/`(AI·추적데이터)`/`(AI·구조확인)`/`(AI·확인필요)`. 이 마커가 있어야 `verify_comment_coverage.py`가 '심층'으로 인정한다.
- 근거가 폴더에 없으면 환각하지 말고 `(AI·확인필요)`로 사유를 적는다.

## 판정·코멘트 필수 규칙 (26.07.06 추가 — 3개 프로젝트 실측 개선)

### 1. 하위 질문 분해
문항에 하위 질문(1, 2, 1-1, 2-1 …)이 있으면 **각각에 대해** 확인/미확인/확인불가를 기술한다.
사람 코멘트가 다룬 부분만 따라가지 말고 문항 전체를 해석하라.
예: "1) 검토 회의 소집 — 회의록 260410에서 확인. 2) 변경요청 시 재검토 — 변경 미발생으로 해당 없음."

### 2. Process 3단 판정 프레임
각 Process 문항은 이 순서로 판단하고 코멘트에 반영한다:
① 이 프로젝트 상황에서 이 활동이 수행될 필요가 있었는가 (차수·변경 발생 여부·범위)
② 수행되었는가 (수행 증거)
③ 증거 품질은 충분한가 (승인·배포·기록 완전성)

### 3. Fail 후보 명시 (판정 회피 금지)
근거로 미수행/미충족이 확인되면 코멘트 머리에 **"Fail 후보"** 를 명시해 제안한다
(Result는 비워 두어 최종 판정은 사람이 확정). 근거를 수집하고도 아무 신호를 주지 않는 것을 금지한다.
예: "Fail 후보 — 아키텍처 시나리오 평가서·시스템 FMEA가 작성·승인·배포되지 않음(폴더·추적데이터 기준)."

### 4. NA(미대상) 개별 사유
No 판정마다 그 문항의 산출물명·실행시점·단계를 인용한 개별 사유를 쓴다. 일괄 문구 금지.
해당 산출물이 폴더에 존재하면 "산출물 존재 — 범위 제외가 맞는지 확인 필요"를 덧붙인다.

### 5. 사람 코멘트 참조 금지 (오염 방지)
점검·코멘트 작성 중에는 같은 차수 사람 작성본을 근거로 쓰지 않는다.
"(사람 의견과 동일)" 등의 표현 금지. 폴더에 없는 시스템 사실(PMS 승인, 변경 상태, Finding 유무)은
사람 코멘트에 있어도 옮겨 적지 말고 "확인 필요"로 남긴다. 검증 단계의 check_contamination.py가 이를 검사한다.

### 6. 날짜 마커
코멘트 머리 날짜는 반드시 당일 YYMMDD (연도 오타 주의 — 게이트가 자동 검사).

### 7. Process 판정 제안 태그 (필수 — 게이트가 검사)
Process 점검대상(Yes) 코멘트는 아래 중 하나를 반드시 포함하고, **Review Result 칸에도 대응 값을 기록**한다
(판정: Pass·Pass 후보→Pass / Fail 후보→Fail / 확인 필요→확인 필요. 최종 확정은 사람 — 게이트가 태그↔Result 일치를 검사):
- **"판정: Pass"** — 수행 증거가 폴더에서 완결 확인(회의록·검토결과서·개정이력 등 시스템 근거 불요 활동).
  
  **금지: 문항이 명시적으로 요구하는 산출물이 하나라도 부재하면, 다른 증거가 있어도 판정: Pass 금지 → Fail 후보.**
  (실측 위험오판: No.32 — 동료검토 결과서 존재를 근거로 Pass했으나 문항 요구물인 시나리오 평가서·FMEA가 부재, 사람은 Fail.
  코멘트에 "다만 ○○ 부재"를 쓰면서 Pass를 주는 것은 모순 — 게이트가 검사한다.)
- **"Pass 후보"** — 폴더 근거(산출물 존재·본문 확인)는 충족, 시스템 승인·등록만 미확인(Result에는 Pass 기록, 사람이 승인 확인 후 확정).
  **Pass 계열(판정: Pass·Pass 후보) 제안 금지 조건** (JX 실측: 사람 Fail 9건 중 Pass 계열 오제안 6건의 원인):
  ⒜ 문항이 **두 문서 간 일관성·매핑·추적 대사**를 요구하면(예: "분석서의 요구가 사양서에 반영됐는가")
     산출물 존재·작성 흐름만으로 Pass 계열 금지 — 직접 대사를 수행했을 때만. 못 했으면 "확인 필요 — 문서 간 대사 필요".
  ⒝ 문항이 **특정 검토 활동**(타당성 검토·동료검토·형상감사 등)의 수행 증거를 요구하면 **그 활동의 직접 기록**만 인정.
     유사 활동으로 대체 판단 금지 — 동료검토 회의록은 타당성 검토의 증거가 아니다(JX No.123 실측 오판).
- **"Fail 후보"** — 다음 **3종에 한해서만** 단다(2차 실측: 기준 없이 달아 31건 중 20건이 위양성):
  ⑴ 요구되는 산출물·검토결과서 **파일 자체가 폴더에 부재**, ⑵ **추적데이터 활성 갭**(trace_checker 기준),
  ⑶ **검토결과서 NG가 후속 버전에 미반영**. Result에는 Fail 기록(사람 확정).
  문서 '내용 요소가 안 보임'(참조 미명시, 절 미발견 등)은 Fail 후보가 아니라 "확인 필요"다 — 사람은 다른 경로(체크리스트 기반 검토 기록 등)로 확인했을 수 있다.
  ※ 실측: 사람 Fail 5건 중 4건에서 AI가 '폴더 부재'를 이미 인지하고도 태그를 안 달았다 — 부재를 확인했으면 반드시 단다.
- **"확인 필요"** — 근거가 전부 사내 시스템(PMS 등)에 있는 활동.
조건부 활동(실행시점이 '주기 도래 시'·'발생 시')은 "[조건부 활동]"으로 발생 여부 확인을 요청한다(일괄 Yes 금지).


### 8. 스코핑 보호 규칙 (미대상 교정 금지 대상 — 게이트가 검사)
- **상시 운영 활동**(문제해결·이슈관리·레드마인·모니터링·의사소통): 근거가 시스템에 있어도 모든 단계에서 점검대상 Yes + "확인 필요". "시스템 근거 = 범위 밖"으로 일반화하지 마라 (2차 실측: 과소 12건 전원 이 유형).
- **계획서류**("…계획서"): 검증/통합/시험 계획서 포함 전부 설계 단계 점검대상. 심층 단계에서 후행 단계 사유로 No로 뒤집기 금지 (2차 실측: WP 과소 21건 중 17건).
- **단위(Unit) 검증류**(단위 검증 명세서·결과서·QAC 정적분석): 설계 단계말 점검대상 — 후행 테스트 아님 (3차 실측: WP 과소 28건 중 21건).
- **심층 단계 스코핑 강등(Yes→No) 금지**: 블라인드 점검에서는 강등하지 말고 `[스코핑 검토 제안]`으로 사람에게 제안만 한다. 강등은 사람 작성본/직전 차수 근거가 있을 때만.
- **태그와 Result는 항상 짝**이다(write_comments가 자동 동기화, 게이트가 불일치 검사).

### 9. 동일 근거 — 동일 처리 (일관성)
같은 근거 상황에는 같은 태그를 단다 (JX 실측: 같은 '회의록 폴더 부재'에 No.14는 Fail 후보, No.15·17은 단정 불가 — 비일관):
- **그룹웨어/전산 관리 기록**(회의록·승인·배포 기록): 폴더 부재 시 "폴더 외 문서일 수 있어 부재 단정 불가 — 시스템 확인 필요"로 **통일**. Fail 후보 금지.
- **폴더가 정본 위치인 산출물**(계획서·설계서·명세서·검토결과서 등 문서 산출물): 부재 확인 시 Fail 후보.
- **조건부/사건발생형 활동**(형상감사·CCB 등 발생 여부 미확인): Fail 후보 금지 — [조건부 활동]으로 발생 확인 요청 (게이트가 검사).
