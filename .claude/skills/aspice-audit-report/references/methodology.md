# 작성 방법론 (배경)

## 산출물 중심으로 작성하는 이유
시트를 위→아래 행 순서로 채우면 같은 산출물이 Process 1~2회·WP 10여 항목·Target 1회로 흩어져 같은 파일을 반복해서 열고, "절차 준수"와 "내용 품질" 판단 모드를 수십 번 오간다. 그래서 **산출물 1건을 열면 그 산출물에 걸린 모든 시트의 모든 항목을 한 번에 처리**한다(조인키 = 산출물명).

## 단계(Phase)는 항목이 아니라 스코핑으로 구분
P1/P2는 별도 항목 세트가 아니라 같은 마스터 항목에 `실행시점`·`재수행 조건`을 적용한 것이다.
- 단계 미도래 항목 → `Applicable=No` ("테스트 단계말 점검 범위").
- 한 번 본 항목도 `재수행 조건`(예: P2 변경 이행)이 걸리면 다음 단계에서 다시 본다.

## 점검 전 게이트 (존재성 → 형식 → 내용)
1. **존재성**: 산출물이 실제로 있는가(Target 매핑·미대상 분리).
2. **형식**: 골격(목차·필수 절·최신 템플릿)을 갖췄는가 — WP 각 산출물 첫 항목 "공통: 최신 템플릿을 사용하였는가?"가 이 게이트다.
3. **내용**: 골격을 통과한 산출물만 깊은 품질 점검으로.

자동 점검은 1·2단계와 미대상 스코핑을 잘 처리하지만, 3단계(특히 일관성·추적성)와 시스템 근거는 사람이 봐야 한다.

## 결과 기입 규칙
- `Applicable`: 대상 Yes / 해당없음·단계미도래 No.
- `Review Result`: 충족 Pass / 미충족 Fail / 보류는 비움.
- `Review Comments`: `YYYY-MM-DD : 근거 위치 + 판단`. 문서는 절·필드, 시스템은 URL.
- `Non-conformances ID`: Fail일 때 레드마인 부적합 이슈 연계. 근거 없는 Pass·ID 없는 Fail 금지.

## ASPICE와의 관계
체크리스트는 ASPICE를 사내 절차서로 현지화한 뒤 그 준수를 Pass/Fail로 보는 것. 정식 ASPICE 평가(N/P/L/F·역량수준·BP/GP)는 아니다. 잣대는 PAM이 아니라 사내 절차서(관리대장의 적용 Rev)다.
