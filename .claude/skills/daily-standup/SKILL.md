# 데일리 스탠드업 작성·등록

이진동(`@이진동`, `U0B8UFCATRU`)이 매일 아침 슬랙 **#daily-standup** 채널에 올리는 스탠드업을 **복붙 없이 대화로** 작성하고, 본인 이름으로 채널에 등록해주는 스킬.

핵심 가치는 매일 아침 **복붙 없이 대화로** 스탠드업을 채우고, **반드시 미리보기·확인을 거쳐서만** 본인 이름으로 게시한다는 것이다(게시는 되돌리기 어려우므로). 지난 글은 참고용으로 보여주되, 내용을 끌어와 미리 채우지는 않는다 — 항목은 매번 사용자에게 새로 묻는다. 또한 각 항목마다 **문구를 리뷰해 수정본을 제안하되, 원본을 쓸지 수정본을 쓸지는 항상 사용자가 결정한다**(Claude는 제안만, 임의 채택 금지).

## 대상 정보 (고정값)

- 채널: **#daily-standup** (ID `C0BCY4F9E86`, 워크스페이스 mobaseasec). ID가 안 맞으면 `slack_search_channels`로 `standup`을 검색해 다시 찾는다.
- 작성자: 이진동 `U0B8UFCATRU`. 슬랙 커넥터가 본인 계정이므로 `slack_send_message`로 보내면 **본인 이름·아바타로** 게시된다.
- 양식(4구획, 슬랙 mrkdwn):

```
:white_check_mark: **어제 한 일**
• …

:dart: **오늘 할 일**
• …

:warning: **이슈 / 막힌 점**
• 없음

:bulb: **제안 / 공유할 내용**
• 없음
```

이모지 shortcode(`:white_check_mark:` 등)는 슬랙이 전송 시 ✅ 등으로 렌더하므로 위 텍스트 그대로 보낸다.

**중요 — 제목 볼드는 별표 2개(`**제목**`).** `slack_send_message`는 **표준 마크다운**을 쓰므로 볼드는 `**텍스트**`(별표 2개)다. 별표 1개 `*텍스트*`는 볼드가 아니라 *이탤릭*으로 렌더되니 쓰지 않는다(예전에 `*어제 한 일*`로 보내 볼드가 안 됐던 원인). 4개 구획 제목(`어제 한 일`, `오늘 할 일`, `이슈 / 막힌 점`, `제안 / 공유할 내용`)은 반드시 `:white_check_mark: **어제 한 일**`처럼 별표 2개로 감싼다.

## 전체 흐름

1. 도구 로드 → 2. 마지막 스탠드업 조회·표시(참고용) → 3. 4개 항목을 **한 번에 하나씩** 문답(항목별 질문→답→되비춤→다음 질문) → 4. 미리보기·확인 → 5. 등록

## Step 1. 도구 로드

`tool_search`로 슬랙 도구를 불러온다(deferred라 먼저 로드).

- `tool_search(query="slack search messages")` → `slack_search_public_and_private`
- `tool_search(query="slack send message channel")` → `slack_send_message`
- (채널 ID 재확인이 필요할 때만) `tool_search(query="slack search channels")` → `slack_search_channels`

## Step 2. 마지막 스탠드업 조회·표시 (참고용)

내가 #daily-standup에 마지막으로 올린 글을 찾아 **참고용으로만** 보여준다(형식·톤 참고). 내용을 끌어와 이번 항목을 미리 채우지는 않는다.

- 검색: `slack_search_public_and_private(query="in:#daily-standup from:<@U0B8UFCATRU>", sort="timestamp", sort_dir="desc", limit=3)`
- 가장 최근 1건을 고른다(요일 무관 — 금요일 글이면 그걸 쓴다). 사용자에게 "마지막 스탠드업(○/○)은 이거였어요" 하고 간단히 보여준다.
- **없으면**(첫 사용 등) 그냥 건너뛰고 Step 3으로 가서 4항목을 묻는다.

## Step 3. 4개 항목 문답 (한 번에 하나씩)

**기본 흐름은 "한 번에 한 항목만" 묻는 대화형이다.** 4개 질문을 한꺼번에 나열하지 말 것 — 사용자가 이 방식을 명시적으로 원한다(오늘도 "왜 항목별로 하나씩 묻고 내가 답하는 게 아니냐"는 지적이 있었다). 순서는 아래와 같다.

1. 먼저 **어제 한 일**만 묻고, 사용자의 답을 기다린다.
2. 답이 오면 **그 답을 어떻게 불릿으로 담았는지 그 자리에서 되비춰 보여주고**(예: `✅ **어제 한 일** / • …`), 이어서 **오늘 할 일**만 묻는다.
3. 같은 식으로 **이슈 / 막힌 점**을 묻는다 — 답이 없거나 "없음"이면 `• 없음`.
4. 마지막으로 **제안 / 공유할 내용**을 묻는다 — 없으면 `• 없음`.

각 항목마다 (a) 그 항목 하나만 질문 → (b) 답 수신 → (c) 담긴 불릿을 되비춤 → (d) **리뷰·수정 제안(아래 규칙)** → (e) 사용자가 원본/수정본을 고르면 확정 → (f) 다음 항목 질문, 을 반복한다. 되비추기는 사용자가 표현·의도가 맞는지 즉시 확인하고 그 자리에서 고칠 수 있게 하는 장치다.

**항목별 리뷰·수정 제안 (사용자가 결정, Claude는 제안만):**
사용자가 한 항목을 답하면, 원문을 그대로 담은 뒤 **더 나은 문구가 있으면 수정본을 함께 제시**한다. 그리고 사용자가 **원본 유지 / 수정본 사용**을 고르게 한다. 규칙은 다음과 같다.
- 원문 불릿을 먼저 그대로 보여준다(원본 보존이 기본값).
- 다듬을 여지가 있으면 "수정 제안:" 으로 대안 문구를 제시하고, **무엇을 왜 바꿨는지 한 줄로** 밝힌다(예: 표현 간결화, 오타, 용어 표기 통일, 능동형). 스탠드업 문맥에 맞는 간결·명료함을 기준으로 한다.
- **내용을 창작·추가하지 않는다.** 없는 성과를 만들지 말고, 사용자가 쓴 사실 범위 안에서 표현만 다듬는다. 도메인 용어(AUTOSAR, CAN, LIN, MCU, WPC, PR, Gitea 등)와 사용자의 의도된 뉘앙스는 보존한다.
- 고칠 게 없으면 억지 제안을 만들지 말고 "문구 손볼 곳 없어 보입니다"라고 밝힌 뒤 다음 항목으로 넘어간다.
- 사용자가 "원본", "그대로", "1번" 등으로 고르면 원문을, "수정본", "고친 걸로", "2번" 등이면 대안을 확정한다. 명시적 선택 전에는 임의로 수정본을 채택하지 않는다.

**예외(사용자가 앞서가면 맞춰준다):** 사용자가 한 답변에 여러 항목을 한꺼번에 적거나 "그냥 다 물어봐"라고 하면, 그때는 받은 만큼 그대로 반영하고 남은 항목만 이어서 묻는다. 이 경우에도 항목별 수정 제안은 붙이되, 최종 미리보기(Step 4)에서 한 번에 원본/수정본을 확인받아도 된다. 즉 하나씩 묻기가 기본이되, 사용자가 몰아서 답하는 것을 막지는 않는다.

작성 규칙:
- **제목 볼드는 별표 2개(`**제목**`).** 4개 구획 제목은 `**...**`(별표 2개)로 감싼다(예: `:dart: **오늘 할 일**`). 별표 1개는 이탤릭이라 볼드가 안 된다.
- **구획 사이엔 빈 줄 한 줄.** 각 구획(제목+불릿) 뒤, 다음 구획 제목 앞에 빈 줄을 한 줄 넣는다. 빈 줄 없이 붙여 보내면 슬랙에서 두 번째 구획부터 들여쓰기가 어긋나 보인다. 한 구획 안의 불릿끼리는 빈 줄을 넣지 않는다.
- **줄바꿈 = 불릿.** 한 항목의 답에서 사용자가 엔터로 나눈 각 줄을 `• ` 불릿 하나로 변환한다. 이미 `•`나 `-`로 시작하는 줄은 그대로 두고(중복으로 `• •` 만들지 않음), 없으면 `• `를 붙인다. 빈 줄은 무시한다.
- 한 항목에 여러 불릿이 올 수 있다. 사용자가 한 줄로만 답하면 불릿 하나가 된다.
- 사용자가 자연어로 길게 말하면 핵심만 간결한 불릿으로 다듬되, **내용을 임의로 추가·창작하지 않는다**. 모호하면 되묻는다. 표현을 다듬을 때는 위 "항목별 리뷰·수정 제안"에 따라 원본과 수정본을 함께 제시해 사용자가 고르게 한다(임의 채택 금지).
- 사용자가 쓴 표현·용어(예: AUTOSAR, CAN, WPC, PR, Gitea 등)는 그대로 살린다.
- **링크는 슬랙 링크 서식으로.** 사용자가 URL(예: Confluence 페이지)을 공유하면 `<URL|라벨>` 형태로 넣어 클릭 가능한 링크로 만든다(예: `• [GIT] 포크 vs 클론: <https://.../GIT+Fork+vs+Clone|GIT Fork vs Clone 차이>`). 라벨은 사용자가 준 제목을 쓰고, 없으면 문서명을 간결히 쓴다.

## Step 4. 미리보기·확인 (필수)

완성된 메시지를 위 양식 그대로(이모지 shortcode·굵게 포함한 **실제 전송될 원문**) 코드블록으로 보여주고, **"이대로 #daily-standup에 등록할까요?"** 라고 명시적으로 묻는다. 사용자가 분명히 동의(예: "응", "등록", "좋아", "올려줘")하기 전에는 **절대 보내지 않는다.** 수정 요청이 오면 반영 후 다시 미리보기를 보여준다.

사용자가 "고칠 데 없나?" 같이 점검을 요청하면, 형식(볼드·빈 줄·링크 서식·오탈자)만 가볍게 확인해 주고, 어제↔오늘 항목의 연결이 어색해 보이면 그 사실만 짚어 되묻는다. **단, 내용을 임의로 고치거나 창작하지 않는다** — 문구 개선은 원본/수정본을 함께 제시해 사용자가 고르게 하고, 판단이 필요한 내용 수정은 확인만 한다.

미리보기에는 Step 3에서 사용자가 항목마다 고른 최종 문구(원본 또는 채택한 수정본)를 반영한다. Step 3에서 수정 제안에 대한 선택이 아직 안 끝난 항목이 있으면, 미리보기 단계에서 원본/수정본을 한 번에 확인받은 뒤 최종본을 확정한다.

## Step 5. 등록

확인을 받으면 `slack_send_message(channel_id="C0BCY4F9E86", message=<원문>)`로 보낸다. 보낸 뒤 "등록 완료" 한 줄과, 가능하면 메시지 링크(permalink)를 알려준다. 채널을 잘못 지정해 다른 곳(다른 채널·DM)으로 보내지 않도록 채널 ID를 반드시 확인한다.

## 경계 / 주의

- **확인 없이는 게시 금지.** 미리보기 → 명시적 동의 → 그때만 전송. 이 순서를 건너뛰지 않는다.
- **내용 창작 금지, 수정은 제안까지만.** 사용자가 말한 것만 정리한다. 비어 있으면 비운 채로 두거나 `• 없음`으로 둔다. 문구 개선이 보이면 수정본을 제안하되 채택 여부는 사용자가 정한다 — 원본 유지가 기본값이고, 명시적 선택 없이 수정본을 반영하지 않는다.
- **본인 이름으로만.** 다른 사람 대신 쓰거나 다른 채널에 올리지 않는다. 대상은 항상 본인(`U0B8UFCATRU`)과 `#daily-standup`.
- **한 번에 한 건.** 이미 오늘 올린 글이 있어 보이면 그 사실을 알리고, 새로 더 올릴지 사용자에게 확인한다(중복 게시 방지).
