# Slack → 구글 캘린더 등록

슬랙 공지/일정 메시지를 찾아 구글 캘린더에 등록하는 워크플로우. 핵심 원칙은 **자연어 날짜를 정확히 파싱하고, 등록 전에 반드시 미리보기로 확인받는 것**이다. 잘못 해석한 일정이 캘린더에 쌓이는 것을 막기 위함이다.

## 전체 흐름

1. 도구 로드 → 2. 슬랙에서 대상 메시지 검색 → 3. 일정 정보 파싱 → 4. **미리보기 제시 + 확인 대기** → 5. 확인 후 캘린더 등록 → 6. 결과 보고(캘린더 링크 포함)

4번 확인 단계를 건너뛰지 마라. 사용자가 "바로 등록해" 라고 명시하지 않는 한 항상 미리보기를 먼저 보여준다.

## Step 1. 도구 로드

`tool_search`로 Slack 검색 도구와 Google Calendar 도구를 불러온다. 이 도구들은 deferred 상태라 먼저 로드해야 호출할 수 있다.

- 슬랙 검색: `tool_search(query="slack search messages")` → `slack_search_public_and_private` 등
- 캘린더: `tool_search(query="google calendar create event")` → `create_event`, `list_calendars` 등

## Step 2. 슬랙에서 대상 메시지 검색

사용자가 준 키워드(예: "AI Agent 구축 사례 세미나")로 `slack_search_public_and_private`를 호출한다. 결과가 여러 개면 가장 관련도 높고 최신인 **원본 공지 메시지**를 고른다(스탠드업 요약·DM 언급 등 파생 메시지가 아니라 공지 본문).

대상이 모호하면 후보를 1~2개 제시하고 어느 것인지 확인한다. 메시지를 특정하면 본문 텍스트와 작성자·채널·시각을 확보한다.

### 본문 밖 정보까지 확인 (중요)

일정 정보가 메시지 **본문에 없고 첨부파일명이나 스레드 답글에 묻혀 있는 경우가 흔하다.** 본문만 보고 "정보 없음"으로 판단하지 말고 아래를 함께 확인한다.

- **첨부파일명**: 슬랙 검색 결과의 `Files:` 항목에 붙은 파일명을 읽는다. 면접·지원자 자료는 `금_1400_고용재_AI_면접제안_...zip`처럼 **요일·시간·이름이 파일명에 인코딩**돼 있는 경우가 많다. (`금_1400` = 금요일 14:00.) 본문에 시간이 없으면 파일명에서 보강한다.
- **스레드 답글**: 부모 메시지에 `Reply count`가 1 이상이면 `slack_read_thread`로 답글까지 읽는다. 일정 변경·확정·취소가 답글에 달리는 경우가 많다.
- **여러 메시지 종합**: 면접 일정처럼 여러 슬롯이 시간순으로 나뉘어 올라오면(예: Meet 링크를 시간대별로 개별 메시지로 공유) 채널을 시간순으로 훑어 누락 없이 종합한다. "화/수/목" 같은 상대 요일은 메시지 작성일을 기준으로 실제 날짜로 환산한다.

요약하면, **본문 → 첨부파일명 → 스레드 → 인접 메시지** 순으로 훑어 일정 정보를 빠짐없이 모은 뒤 파싱한다.

## Step 3. 일정 정보 파싱

메시지 본문에서 다음을 추출한다. 한국어 자연어 날짜 해석이 핵심이다.

- **summary(제목)**: 공지 제목을 그대로. 보통 `[공지] ...`, `[세미나] ...` 형태.
- **start / end**: 본문의 일시. 본문에 없으면 첨부파일명·스레드에서 보강(Step 2 참고). 아래 규칙을 적용.
- **location**: "장소 : 회의실" 등.
- **description**: 본문의 상세(내용·강사·대상 등) + 출처 표기. 정보를 첨부파일명에서 가져왔다면 그 사실도 짧게 남긴다(예: `시간 출처: 첨부파일명 '금_1400_...'`).

### 날짜·시간 규칙 (자주 막히는 부분이니 주의)

- **연도 누락**: "7월 10일 금요일"처럼 연도가 없으면 현재 날짜를 기준으로 가장 가까운 미래의 해당 날짜로 해석한다. (오늘이 6월이고 "7월 10일"이면 올해 7월 10일.)
- **요일 검증**: 본문에 요일이 적혀 있으면(예: "금요일") 추론한 날짜의 실제 요일과 일치하는지 확인한다. 어긋나면 추측하지 말고 미리보기에서 "본문 요일과 날짜가 불일치합니다" 라고 짚어준다.
- **종료 시간 누락**: 시작만 있고 끝이 없으면 **기본 1시간 30분**으로 잡는다. (세미나·회의 기본값.) 미리보기에 "종료 시간이 공지에 없어 1.5시간으로 설정" 이라고 명시한다.
- **"(예정)" / "잠정" 표기**: 시작 시각에 "(예정)" 같은 단서가 있으면 description에 `※ 시간은 공지상 '예정'이며 변동될 수 있음` 주석을 넣고, 미리보기에서도 잠정 일정임을 알린다.
- **타임존**: 항상 `Asia/Seoul` 로 고정한다. `timeZone="Asia/Seoul"` 를 명시한다.

### description 작성

본문 상세를 정리해 넣고, 마지막에 출처를 남긴다. 예:

```
내용: 제어 SW 개발 프로세스에서 AI Agent 구축 및 운용 사례
강사: 사외 개발자

※ 시간은 공지상 '예정'이며 변동될 수 있음
출처: Slack #채널명 공지 (작성자)
```

## Step 4. 미리보기 제시 + 확인 대기 (필수)

등록하기 전에 아래 형식으로 미리보기를 보여주고 멈춘다. 사용자의 확인("등록해", "OK", "좋아")을 받은 뒤에만 Step 5로 간다.

```
아래 내용으로 등록할게요. 확인해 주세요.

• 캘린더 : 모베이스ASEC-이진동
• 제목   : [공지] AI Agent 구축 사례 세미나
• 일시   : 2026년 7월 10일(금) 15:30 ~ 17:00 (KST)
• 장소   : 회의실
• 내용   : 제어 SW 개발 프로세스에서 AI Agent 구축 및 운용 사례 / 강사: 사외 개발자

참고: 종료 시간은 공지에 없어 1.5시간으로 잡았고, 시작 시각이 '(예정)'이라 변동될 수 있습니다.
이대로 등록할까요?
```

날짜·요일 불일치나 정보 누락이 있으면 이 단계에서 반드시 짚어준다. 사용자가 수정을 요청하면 반영 후 미리보기를 다시 보여준다.

## Step 5. 캘린더 등록

확인을 받으면 `create_event`로 등록한다. **기본 캘린더는 모베이스다.** 사용자가 다른 캘린더를 지정하지 않는 한 아래 ID를 쓴다.

- 모베이스 캘린더 ID: `e12621c1ae350b5c2f414c7149f86a8f97072748be3b3515a30e4fcbb960441e@group.calendar.google.com`
  (표시명: "모베이스ASEC-이진동")

이 ID가 더 이상 맞지 않거나 환경이 바뀐 것 같으면 `list_calendars`로 이름에 **"모베"** 가 들어간 캘린더를 찾아 그 ID를 쓴다("모베스", "모베이스", "모베이스ASEC" 등 사용자가 어떤 줄임말로 불러도 이 캘린더를 가리키는 것으로 해석한다). 개인 일정으로 넣어달라고 하면 primary 캘린더(기본값)에 등록한다.

호출 예:

```
create_event(
  calendarId="e12621c1ae350b5c2f414c7149f86a8f97072748be3b3515a30e4fcbb960441e@group.calendar.google.com",
  summary="[공지] AI Agent 구축 사례 세미나",
  startTime="2026-07-10T15:30:00",
  endTime="2026-07-10T17:00:00",
  timeZone="Asia/Seoul",
  location="회의실",
  description="...(Step 3에서 만든 내용)..."
)
```

### 중복 방지

같은 공지를 두 번 등록하지 않도록 주의한다. 직전에 이미 같은 일정을 만들었거나, 사용자가 "아까 그거"라고 하면 새로 만들기 전에 기존 등록 여부를 확인한다. (이번 세션에서 막 만든 이벤트 ID를 기억하고 있으면 그것을 참조한다.)

## Step 6. 결과 보고

등록 성공 후 간결하게 보고한다. 캘린더명·제목·일시·장소와 `htmlLink`(캘린더에서 보기)를 포함한다. 길게 늘어놓지 말고 핵심만.

## 자주 발생하는 케이스

- **잘못된 캘린더에 등록함**: 사용자가 "모베이스 캘린더에 넣어야 하는데"처럼 정정하면, 기존 이벤트를 `delete_event`로 지우고 올바른 `calendarId`로 다시 `create_event` 한다.
- **여러 일정이 한 메시지에**: 회의 링크 모음처럼 여러 시간이 한 메시지에 있으면 어느 것을 등록할지 먼저 확인한다.
- **참석자 추가 요청**: `attendees`에 이메일을 넣되, 외부인에게 초대 메일이 가지 않도록 기본은 `notificationLevel="NONE"`을 고려한다(사용자 의사 확인).
