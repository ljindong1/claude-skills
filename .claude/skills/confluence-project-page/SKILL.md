# Confluence 신규 프로젝트 페이지 생성 스킬

mobaseasec Confluence 의 **내부 프로젝트** 영역에 신규 프로젝트가 생길 때, 표준 페이지 세트를
**고정된 구조 + 매크로**로 한 번에 만들어 준다. 콘텐츠(프로젝트 정보)는 매번 다르지만
**레이아웃·매크로·필드 구성은 고정**한다.

생성되는 페이지는 항상 2개다:

```
[메인] {프로젝트명}                       ← 부모 페이지 하위에 생성
   └── [자식] 진행 현황                    ← 메인 페이지 하위에 자동 생성
```

메인 페이지 본문 구조(2단 레이아웃):

```
┌─────────────────────────────┬─────────────────────────────┐
│ ■ 개요                       │ ■ Jira 프로젝트 (URL 있을 때)│
│   (Page Properties 매크로)   │   (인라인 스마트 링크)       │
│   프로젝트명 / 코드 /        │ ■ 과제 참여 인원 구성        │
│   개발 배경 / 요청 내용 /    │   - 이름 1, 2, …             │
│   요청 부서·담당자·일자 /    │                              │
│   과제 책임자 /              │ ■ 하위 페이지                │
│   시작·종료일 / 진행 상태    │   (Children 매크로)          │
└─────────────────────────────┴─────────────────────────────┘
───────────────────────────────── (구분선)
■ 개발 로드맵
  (Roadmap Planner 매크로 — 인터랙티브 타임라인)
```

자식 "진행 현황" 페이지: **번호 / 날짜 / 진행 현황 / 비고** 4열 테이블.

---

## 0. 핵심 제약 — 반드시 먼저 읽기

- **contentFormat 은 `html`** 로 생성한다. 메인 페이지는 Page Properties·Children·Roadmap 매크로와
  2단 레이아웃을 쓰는데, 이건 markdown 으로 표현되지 않는다. markdown 으로 보내면 매크로가
  텍스트로 깨진다. (참고: 이 MCP 는 create/update 모두 `contentFormat: "html"` 정상 동작한다.)
- **부모 페이지는 매번 사용자에게 확인**한다. 임의로 정하지 않는다. 기본 후보는 `내부 프로젝트`
  (pageId `249200664`) 이며, 이걸 디폴트로 **제안**하되 확정은 사용자에게 받는다.
- **본문 템플릿은 직접 손으로 쓰지 말고 `assets/` 의 템플릿 파일을 읽어 플레이스홀더만 치환**한다.
  매크로 HTML 은 한 글자만 틀려도 렌더링이 깨지므로 통째로 재작성하지 않는다.
- **개요 테이블에 `<thead>` 를 쓰지 않는다.** `<thead>` 안의 행은 변환기가 모든 셀을 `<th>` 로
  바꿔, 프로젝트명 **값** 셀까지 헤더(굵게/음영)가 된다. 전 행을 `<tbody>` 에 두고 라벨만 `<th>`,
  값은 `<td>` 로 둔다 (템플릿은 이미 이렇게 되어 있다).
- **제목은 스페이스 내 유일해야 한다.** 메인(프로젝트명)·자식("진행 현황") 둘 다. 중복이면 400 으로
  거부되므로, 자식은 기본 `{프로젝트명} 진행현황` 으로 한정한다 (Step 5).
- **너비는 full-width breakout 으로 최대화한다.** 2단 레이아웃 섹션에 `data-breakout="full-width"`
  를 주면 개요·하위 페이지 영역이 화면 끝까지 펼쳐진다(검증됨, 템플릿에 반영). 단, 페이지 **표시
  설정 자체**(편집기의 Fixed width ↔ Full width 토글)는 content-property 라 이 MCP 로는 설정할 수
  없다. breakout 으로 콘텐츠 폭은 최대가 되지만, 페이지 좌우 여백까지 완전히 없애려면 사용자가
  편집기에서 한 번 토글해야 한다. (로드맵 매크로는 자체 고정 폭이라 breakout 영향이 작다.)

> ✅ **검증됨 (2026-06-25, mobaseasec)**: 위 템플릿을 `contentFormat:"html"` 로 생성하면 Page
> Properties·Children·Roadmap·2단 레이아웃 4개 매크로가 모두 보존되고, 로드맵 `source`/`hash`,
> 열 너비, HTML 이스케이프(`&·<·>`), 그리고 2단 섹션의 `data-breakout="full-width"` 가 정상
> 적용됨을 실제 페이지 생성으로 확인했다. (단일 컬럼 full-width 섹션 `layout-one` 은 미지원.)
> 진행 현황 표는 참조 페이지(250216464)와 **동일하게** `data-width="1305"` + 열 폭 번호 58 /
> 날짜 113 / 진행 현황 797 / 비고 333, 헤더 가운데 정렬, 번호 칸 가운데 정렬로 맞춤(검증됨).

## 2. 워크플로우

### Step 1 — 프로젝트 정보 수집

아래 필드를 사용자에게 받는다. 대화에 이미 나온 값은 다시 묻지 않고 채운다.
한 번에 다 묻기보다, **빠진 필수 항목만** 모아서 묻는다.

| 필드 | 필수 | 설명 / 기본값 |
| --- | --- | --- |
| 프로젝트명 | ✅ | 메인 페이지 **제목**으로도 사용. ⚠️ Confluence 는 **스페이스 내 제목이 유일**해야 하므로 같은 이름이 이미 있으면 생성이 400 으로 거부된다. Step 2 에서 부모 자식 목록을 확인해 중복이면 사용자에게 알린다 |
| 프로젝트 코드 | ⬜ | 없으면 공란 |
| Jira 프로젝트 URL | ⬜ | 있으면 우측 단(하위 페이지 목록 단) **최상단**에 `■ Jira 프로젝트` + 인라인 스마트 링크로 표시. 없으면 블록 자체를 생략 |
| 개발 배경 | ✅ | 왜 시작하는지 |
| 개발 요청 내용 | ✅ | 무엇을 해달라는지 |
| 요청 부서 | ✅ | |
| 요청 담당자 | ✅ | "이름 직책" 형식 권장 (예: 김심수 연구기획팀장) |
| 요청 일자 | ✅ | `YYYY-MM-DD` |
| 과제 책임자 | ✅ | |
| 과제 시작일 | ⬜ | 없으면 공란 |
| 과제 종료일 | ⬜ | 없으면 공란 |
| 진행 상태 | ✅ | 기본값 `검토/협의 중` |
| 과제 참여 인원 | ✅ | 이름 목록 (1명 이상) |

진행 현황 첫 행(선택): 요청 접수 이력을 시드로 1행 넣을지 사용자에게 확인한다.
기본 제안: `1 / {요청 일자} / {개발 요청 내용 요약} / (비고)`. 원치 않으면 빈 표(헤더만)로 생성한다.

### Step 2 — 부모 페이지 확정

> "이 프로젝트 페이지를 어느 페이지 하위에 만들까요? (기본: `내부 프로젝트`, pageId 249200664)"

사용자가 다른 페이지를 지정하면 그 pageId 를 쓴다. 페이지 제목만 알고 pageId 를 모르면
`Atlassian:getConfluencePageDescendants` 나 `Atlassian:search` 로 찾는다.

### Step 3 — 메인 페이지 본문 빌드

1. `assets/main_page_body.html` 을 읽는다.
2. 매크로 ID 플레이스홀더 3개를 **새 UUID** 로 치환한다 (각각 서로 다른 값):
   `{{MACRO_ID_DETAILS}}`, `{{MACRO_ID_CHILDREN}}`, `{{MACRO_ID_ROADMAP}}`.
   UUID 는 `python3 -c "import uuid;print(uuid.uuid4())"` 로 생성하거나 임의의 UUIDv4 를 쓴다.
3. 개요 필드 플레이스홀더를 사용자 값으로 치환한다:
   `{{PROJECT_NAME}}`, `{{PROJECT_CODE}}`, `{{BACKGROUND}}`, `{{REQUEST_CONTENT}}`,
   `{{DEPT}}`, `{{REQUESTER}}`, `{{REQUEST_DATE}}`, `{{LEAD}}`, `{{START_DATE}}`,
   `{{END_DATE}}`, `{{STATUS}}`. 값이 없으면 빈 문자열로 둔다.
4. `{{MEMBERS}}` 를 참여 인원 목록으로 치환한다. 각 인원은 `<li><p>이름</p></li>` 형식:
   예) `<li><p>한무희</p></li><li><p>서진우</p></li>`
4-b. `{{JIRA_BLOCK}}` 를 치환한다. 이 자리는 우측 단(하위 페이지 목록 단) **최상단**이다.
   - Jira 프로젝트 URL 이 **있으면**:
     `<p>■ Jira 프로젝트</p><p><a href="{url}" data-card-appearance="inline">{url}</a></p>`
     (URL 은 HTML 이스케이프. `data-card-appearance="inline"` 면 Jira 프로젝트 칩으로 깔끔히 렌더됨)
   - **없으면**: `{{JIRA_BLOCK}}` 를 **빈 문자열**로 치환(블록 생략).
5. **치환 값은 HTML 이스케이프**한다 (`&`→`&amp;`, `<`→`&lt;`, `>`→`&gt;`). 특히 개발 요청 내용에
   `&`, `<`, `>` 가 들어가면 깨지므로 주의.
6. **개발 로드맵 매크로(`data-extension-key="roadmap"`)의 `data-parameters` 는 손대지 않는다.**
   원본 그대로 두면 기본 골격(행 1/행 2, 막대 1·2·3, 표식 1)이 들어간 인터랙티브 로드맵이
   생성된다. 실제 일정은 사용자가 Confluence 편집기에서 채운다. (날짜·막대를 코드로 바꾸려
   하면 매크로 hash 가 어긋나 깨질 수 있으므로 시도하지 않는다.)

치환은 손편집보다 스크립트가 안전하다. 필요하면 `assets/` 파일을 읽어 Python `str.replace` 로
처리한 본문 문자열을 만든다.

### Step 4 — 메인 페이지 생성

`Atlassian:createConfluencePage` 를 호출한다 (파라미터명은 `tool_search` 로 확인). 일반적으로:

```
cloudId:        37df4bda-5774-4df5-b131-5e67ab03a156
spaceId:        43843599
parentId:       {Step 2 에서 확정한 부모 pageId}
title:          {프로젝트명}
body:           {Step 3 에서 완성한 HTML}
contentFormat:  html
```

응답에서 생성된 메인 페이지의 `id` 를 확보한다 → 자식 페이지의 parentId 로 쓴다.

### Step 4.5 — 메인(개요) 페이지에 레이블 추가

메인 프로젝트 페이지(개요가 있는 페이지)에는 **`내부프로젝트`** 레이블을 붙인다. 레이블은
페이지 본문이 아니라 메타데이터라 `body` 로는 넣을 수 없고, 전용 도구나 REST 가 필요하다.

1. 먼저 `tool_search` 로 Confluence **레이블 추가 도구**(예: content label / add label) 가 이번
   세션에 있는지 확인한다. 있으면 그 도구로 pageId(=Step 4 의 메인 페이지 id)에 `내부프로젝트` 를 추가.
2. 레이블 도구가 없으면(현재 기본 Atlassian MCP 에는 없음) **자동 추가가 불가**하다. 이때는
   사용자에게 수동 추가를 안내한다: 페이지 하단 레이블 아이콘 또는 `•••` → "레이블 추가" →
   `내부프로젝트` 입력. (REST `/wiki/rest/api/content/{id}/label` 엔드포인트 자체는 닿지만,
   인증 토큰을 MCP 가 쥐고 있어 bash 에서는 401/403 이라 직접 호출 불가.)

> 레이블이 붙어 있으면 나중에 레이블 기준 검색·필터·Page Properties Report 로 내부 프로젝트들을
> 한데 모아 보기 좋다. 자식 "진행 현황" 페이지에는 기본적으로 붙이지 않는다(메인에만).

### Step 5 — 진행 현황 자식 페이지 생성

> ⚠️ **제목 유일성**: Confluence 는 스페이스 내 페이지 제목이 유일해야 한다. 이미 다른
> 프로젝트에 "진행 현황" 페이지가 있으면 plain "진행 현황" 으로는 생성이 거부된다(검증됨).
> 따라서 자식 제목은 기본적으로 **`{프로젝트명} 진행현황`** 으로 한정한다.
> (예: 프로젝트명이 "AI 에이전트 활용" 이면 → `AI 에이전트 활용 진행현황`. 프로젝트명이 매우 길면
> 프로젝트 코드를 써서 `{프로젝트 코드} 진행현황` 도 가능. 사용자 선호에 맞춘다.)

1. `assets/progress_child_body.html` 을 읽는다.
2. `{{ROWS}}` 를 채운다. 각 행은 참조 페이지(250216464)와 동일하게 **열 폭·정렬을 맞춘다**.
   번호 칸은 가운데 정렬, 나머지는 좌측 정렬:
   ```
   <tr><td data-colwidth="58"><p style="text-align: center">{번호}</p></td><td data-colwidth="113"><p>{날짜}</p></td><td data-colwidth="797"><p>{진행 현황}</p></td><td data-colwidth="333"><p>{비고}</p></td></tr>
   ```
   - 시드 1행을 넣기로 했으면 그 행을, 빈 표면 `{{ROWS}}` 를 빈 문자열로 치환한다.
   - 값은 Step 3 과 동일하게 HTML 이스케이프.
3. `createConfluencePage` 호출:
   ```
   cloudId:        37df4bda-5774-4df5-b131-5e67ab03a156
   spaceId:        43843599
   parentId:       {Step 4 에서 받은 메인 페이지 id}
   title:          {프로젝트명} 진행현황
   body:           {완성한 HTML}
   contentFormat:  html
   ```

### Step 6 — 결과 보고

생성된 두 페이지의 제목과 URL(webUrl)을 사용자에게 보고한다. 로드맵·진행 현황 표는
편집기에서 추가로 채우면 된다는 점을 한 줄로 안내한다.

## 2.5 기존 페이지에 내용 추가·수정 (사후 편집 모드)

페이지 생성 이후, 사용자가 **특정 페이지를 지정하고 내용을 주면** 그 페이지 양식에 맞게 깔끔히
정리해 반영한다. 트리거 예: "이 진행현황에 오늘 내용 추가해줘", "[URL] 이 페이지에 이거 정리해서
넣어줘", "개요의 진행 상태를 완료로 바꿔줘", "참여 인원에 OOO 추가".

### 공통 워크플로우 (부분 수정 미지원 → fetch 먼저)

1. **대상 페이지 확정**: 사용자가 URL·제목·pageId 중 무엇을 주든 pageId 로 확정한다. 제목만 주면
   `Atlassian:search` 또는 부모의 `getConfluencePageDescendants` 로 찾는다.
2. **현재 본문 fetch (필수)**: `getConfluencePage(cloudId, pageId, contentFormat="html")`.
   ⚠️ MCP 의 update 는 **본문 전체 교체**다(부분 수정 없음). 반드시 현재 HTML 을 받아 그 위에서
   고친다. fetch 를 생략하고 일부만 보내면 기존 표·매크로·다른 내용이 전부 사라진다.
   - `getConfluencePage(html)` 는 편집용 HTML 방언(`<table data-width>`, `data-colwidth`,
     `data-type="extension"` 등)으로 돌려준다 — 이게 update 가 받는 형식과 같아 그대로 왕복된다.
3. **페이지 유형 판단**:
   - 번호/날짜/진행 현황/비고 4열 표 → **진행현황 페이지** (아래 규칙)
   - Page Properties(개요) 테이블 + 매크로 → **개요 메인 페이지** (아래 규칙)
   - 그 외 → 본문 맥락에 맞춰 자연스럽게 통합
4. **내용 정리**: 사용자가 준 거친 메모를 그 페이지 문체·셀 구조에 맞게 **핵심만 한두 문장으로**
   깔끔히 다듬는다(구어체·군더더기 제거). 값은 HTML 이스케이프(`&·<·>`).
5. **전체 본문 update**: `updateConfluencePage(cloudId, pageId, contentFormat="html", body=수정본 전체)`.
   매크로 블록과 `data-parameters`(특히 roadmap source/hash)는 절대 손대지 않는다.

### 진행현황 페이지에 일정/진행 추가 — ⭐ 최신이 항상 맨 위

- **새 항목은 항상 헤더 바로 아래(첫 데이터 행)에 삽입**한다. 최신이 위로, 과거가 아래로(내림차순).
- **번호** = 기존 표의 최대 번호 + 1 (따라서 맨 위 행이 가장 큰 번호). 표가 비어 있으면 1 부터.
- **날짜** = 사용자가 명시 안 하면 오늘 날짜(`YYYY-MM-DD`).
- **진행 현황** = 사용자가 준 내용을 한두 문장으로 정리.
- **비고** = 부가 정보/첨부 안내가 있으면, 없으면 공란.
- 행 형식·열 폭은 기존 표와 동일하게 맞춘다(번호 58 / 날짜 113 / 진행 현황 797 / 비고 333,
  번호 칸 가운데 정렬). 여러 건을 한 번에 주면 시간 순으로 **가장 최신이 가장 위**가 되도록 쌓는다.

새 행 형식(맨 위 삽입):
```
<tr><td data-colwidth="58"><p style="text-align: center">{새 번호}</p></td><td data-colwidth="113"><p>{날짜}</p></td><td data-colwidth="797"><p>{정리한 진행 내용}</p></td><td data-colwidth="333"><p>{비고}</p></td></tr>
```

예시 — 기존 표의 최대 번호가 4이고 사용자가 "오늘 킥오프 미팅 했고 다음 주까지 요구사항 정리하기로
함" 이라고 주면:
- 번호 5, 날짜 오늘, 진행 현황 "킥오프 미팅 진행. 차주까지 요구사항 정리 예정", 비고 공란
- 이 행을 **헤더 바로 다음**(기존 번호 4 행보다 위)에 넣는다.

### 개요 메인 페이지 필드 수정

- Page Properties 테이블에서 **해당 라벨 행의 `<td>` 값만 교체**한다. 라벨(`<th>`)·표 구조·열 폭·
  매크로는 그대로 둔다. (예: 진행 상태 `검토/협의 중` → `진행 중`, 과제 종료일 채우기 등)
- 참여 인원 변경은 `■ 과제 참여 인원 구성` 아래 `<ul>` 의 `<li><p>이름</p></li>` 를 추가/삭제.
- details / children / roadmap 매크로 div 와 그 `data-parameters` 는 **건드리지 않는다**.
- 수정 후 `getConfluencePage(html)` 로 다시 읽어 매크로 4종이 살아있는지 확인하면 안전하다.

## 3. 매크로 참고

| 매크로 | data-extension-key | 역할 | 비고 |
| --- | --- | --- | --- |
| Page Properties | `details` (bodied-extension) | ■개요 테이블을 감싸 라벨/포털 집계 대상화 | thead 미사용, 전 행 tbody (라벨 th / 값 td) |
| Child pages | `children` | ■하위 페이지 자동 목록 | `allChildren: true` |
| Roadmap Planner | `roadmap` | 인터랙티브 타임라인 | data-parameters 원본 유지 |
| 2단 레이아웃 | `<section data-type="layout-two-equal">` | 개요 ↔ 하위/인원 좌우 분할 | full-width breakout |

매크로 HTML 의 세부 구조나 hash 동작이 궁금하면 `references/macros.md` 를 읽는다.

## 4. 주의사항

- **생성 전 부모 확인 필수.** 잘못된 위치에 만들면 이동이 번거롭다.
- **매크로 data-parameters 통째 보존.** 특히 roadmap 의 `source`/`hash` 는 건드리지 않는다.
- **HTML 이스케이프 누락 주의.** 개요/진행현황 값에 `&·<·>` 가 있으면 반드시 엔티티로 변환.
- **이미 같은 이름의 프로젝트 페이지가 있는지** 먼저 확인하면 중복 생성을 피할 수 있다
  (`Atlassian:getConfluencePageDescendants` 로 부모의 자식 목록 확인).
- 생성 결과가 깨져 보이면 `getConfluencePage(contentFormat="html")` 로 다시 읽어 어떤 매크로가
  유실됐는지 진단하고, 해당 매크로 블록만 템플릿과 대조한다.
