# 보고서 도입부 (개념 정리 4블록)

본격 세부 해석에 들어가기 전, 보고서 첫머리는 아래 4블록으로 구성한다. "추상 →
시각 → 구조 → 용어" 순서로 점점 구체화되며 들어가게 한다.

**철칙**: 모든 블록의 내용은 전송된 ARXML 파일을 파싱한 결과만을 근거로 한다.
일반론·표준 데이터플로 추정을 넣지 않는다(SKILL.md 최상단 원칙 참조).

**시각화 형식 규칙 (전 보고서 공통)**:
- 구조·계층·관계·흐름(트리, 연결, 배선, 생성 흐름) → **SVG**. `svg-creation` 스킬
  사용. Confluence 첨부 흐름을 따른다.
- **트리(AR-PATH·패키지·컨테이너 계층)는 항상 `svg-creation`의 「폴더트리 패턴」**으로
  그린다 — 파일 탐색기식 폴더/파일 글리프, 고정 간격, 우측 개수 칩. 모양이 보고서마다
  동일해야 한다. 기준 예시: `references/arpath_tree_example.svg`.
- 단순 속성·수치 나열(태그 목록, 타입 분포, 파라미터 값) → **표**.
- **ASCII 트리/다이어그램은 쓰지 않는다** — Confluence에서 정렬이 깨진다.

**작성 시 주의**: "이 문서는 개념 정리 파트다 / N개 블록으로…" 같은 메타 안내
문구는 넣지 않는다. 곧바로 내용으로 시작한다.

### 서문 (블록 1 위) — 고정 규칙

도입부 맨 앞 서문은 **대상 모듈(예: CanIf, PduR, Dem)이 무슨 역할을 하는 BSW
모듈인지**를 1~2문장으로 설명하고, 마지막 한 문장으로 "`<업로드 파일명 원문>`은
그 `<모듈>`의 설정을 다루는 ARXML이다"로 끝낸다. 형식은 다음을 그대로 따른다:

```
<모듈>(<영문 풀네임>)은 <상위/하위 어디 사이에 위치하는> BSW <분류> 모듈이다.
<핵심 역할 1~2가지>를 담당한다. `<파일명.arxml>`은 그 <모듈>의 설정을 다루는 ARXML이다.
```

- 서문에는 **"정의용/값용" 판정이나 그 근거(루트 element·스키마·refine·variant)를
  쓰지 않는다.** 그건 블록 1에서만 다룬다(서문↔블록1 중복 금지).
- 모듈 역할 설명은 그 모듈의 일반 기능이며 파일 파싱값이 아니어도 된다(서문 한정 예외).
  단 단정적 수치·이 파일 고유 사실은 넣지 않는다.
- 첫머리부터 끝까지 **"이 파일" 같은 대명사 대신 업로드된 파일명 원문**을 쓴다
  (예: `이 파일은` ❌ → `Ecud_CanIf.arxml`은 ✅). 파일명은 `/mnt/user-data/uploads/`의
  basename을 글자 그대로 복사한다.
- 참조형 파일이면 서문 끝에 "단독으로는 (RTE 등) 생성이 불가능하고 가리키는 정의
  파일들과 함께 묶여야 완성된다"를 한 줄 덧붙인다.

## 정의용 vs 값용 — 블록별 분기

블록 1 골격은 공유하되, 블록 2·3·4는 파일 종류(정의용/값용)에 따라 내용이 갈린다.

| 블록 | 정의용 (ECU Extract, System Desc 등) | 값용 (ECU Configuration, MCAL 등) |
|---|---|---|
| 1 종류 | `### 1. 정의용 (definition)` | `### 1. 값용 (values)` |
| 2 흐름 | 생성 - 변환 - 출력 (입력 소스 → 스택 개발사 refine → 이 파일) | 생성 - 변환 - 출력 (정의·연결모듈 → 컨피규레이터 → 이 파일 → 코드 생성기) |
| 3 트리 | `path_tree` (SYSTEM/ECU-INSTANCE/COMPOSITION) | `ecuc_tree` (MODULE→CONTAINER→SUB) |
| 4 태그 | `*-REF`, `*-PORT`, `*-MAPPING` 위주 | `ECUC-*-PARAM-VALUE`, `CONTAINER-VALUE` 위주 |

판정은 `arxml_inspect.py`의 `element_counts`에 `ECUC-MODULE-CONFIGURATION-VALUES`가
있으면 값용, `SYSTEM`이 있으면 정의용. (자세한 판정표는 file_types.md)

## 블록 1. 정의용 / 값용 (고정)

이 블록은 **파일이 정의용인지 값용인지만** 1~2줄로 박는다. 길게 풀지 않는다.

- H3 제목은 파일 종류에 따라 **정확히** 다음 중 하나로 한다:
  - 정의용이면  `### 1. 정의용 (definition)`
  - 값용이면    `### 1. 값용 (values)`
- 본문은 **1~2줄**. 형식을 그대로 따른다:
  - 정의용: "루트가 `ECUC-MODULE-DEF` 이므로 **정의용**이다. 즉 값이 아니라 양식
    (스키마)을 담는다. <한 줄 근거: refine 출처·variant 등 파일에서 읽힌 사실>."
  - 값용: "루트가 `ECUC-MODULE-CONFIGURATION-VALUES` 이므로 **값용**이다. 즉 양식이
    아니라 그 양식(`<정의 파일명>`)에 채운 실제 값을 담는다. <한 줄 근거: 인스턴스 수·
    variant 등>."
- **2행 표(이 파일의 위치 / 의미)는 만들지 않는다.** 분류 축 표·"담는 방식" 행 금지.
- 판정 근거: `arxml_inspect.py`의 `element_counts`에 `ECUC-MODULE-CONFIGURATION-VALUES`
  가 있으면 값용, `ECUC-MODULE-DEF`(값 없이)면 정의용, `SYSTEM`이면 시스템 구조 정의.
  (자세한 판정표는 file_types.md)

## 블록 2. 생성 - 변환 - 출력 흐름 (SVG · 고정)

H3 제목은 **반드시 정확히** `### 2. 생성 - 변환 - 출력 흐름` 으로 한다. "무엇을
가리키나", "참조 대상", "어떻게 만들어지나" 등 다른 제목으로 바꾸지 않는다.

이 블록은 **이 파일이 어떻게 만들어지는지**를 좌→우 흐름으로 보인다. 칸 구성과
내용은 아래로 고정하며, 매번 자유롭게 바꾸지 않는다.

**칸 구성 (좌→우, SVG)**:
```
[생성: 입력 소스] → [변환: 주체/도구] → [출력: 이 파일] → ([전달처: 다음 소비자])
```

1. **생성(입력 소스)** — 이 파일을 만들 때 입력이 된 원천. 파싱으로 읽히는 것만.
   - 정의용: refine 원본(예: 표준 `/AUTOSAR/EcucDefs/<Module>`).
   - 값용: 따르는 정의(`DEFINITION-REF` 대상) + 연결 모듈(`VALUE-REF` 대상, 예 Can 드라이버·EcuC Pdu).
2. **변환(주체/도구)** — 입력을 이 파일로 바꾼 행위자. **그 작업을 실제로 한 주체만** 적는다.
   - 정의용의 refine: 주체는 **AUTOSAR BSW 스택 개발사**(네임스페이스로 드러난 벤더, 예 AUTRON).
     이는 스택 개발사가 제품 배포 시 미리 하는 것이며 **컨피규레이터 도구(Mobilgene C Studio
     등)가 하는 일이 아니다 — 도구명을 넣지 않는다.**
   - 값용의 값 채움: 주체는 **컨피규레이터 도구**다. 이때만 `Mobilgene C Studio`(미지원
     작업이면 실제 지원 도구)를 변환 칸에 넣는다.
   - **규칙: Mobilgene C Studio는 그 도구가 실제로 수행하는 작업(값 채움 등)에만 넣는다.
     아무 근거 없이 변환 칸에 자동으로 넣지 않는다.**
3. **출력(이 파일)** — 산출된 파일. 박스 라벨은 **"이 파일"이 아니라 업로드 파일명 원문**을
   쓴다(길면 앞·뒤를 살린 축약 표기 가능, 예 `AUTRON_..._PDF.arxml`). 아래에 루트 element를
   부기한다(예 `ECUC-MODULE-DEF /AUTRON/CanIf`).
4. **전달처(다음 소비자)** — **파일 안에 단서가 있을 때만** 그린다.
   - 값용: `MODULE-DESCRIPTION-REF` 대상(BSWMD)과 함께 코드 생성기로 넘어가는 정도까지.
     그 이후(빌드 등)는 파일에 없으므로 칸 아래 캡션으로 "이후 단계는 파일에 단서 없음(생략)".
   - 정의용: 전달처 단서가 없으면 칸을 만들지 말고 캡션으로 "전달처: 파일 내 단서 없음(생략)".
   - **파일에 없는 다음 단계를 일반론으로 추정해 칸으로 그리지 않는다.**

**금지/주의**:
- ASCII 다이어그램 금지. 이 흐름은 `svg-creation` 스킬의 박스+화살표로 그린다.
- "이 파일"·대명사 대신 파일명. 본문·SVG 라벨 모두 동일.
- 일반론 INPUT 추정 금지 — `arxml_inspect.py`의 `outgoing_refs`가 가리키는 실제 대상만.
  파일이 안 가리키는 것은 그리지 않는다(예 MCAL Can 컨피그가 ECU Extract를 참조하지
  않으면 ECU Extract를 넣지 않는다).

## 블록 3. AR-PATH 폴더트리 (SVG)

AR-PATH 트리는 **항상 동일한 폴더트리(파일 탐색기) 스타일**로 그린다. 종류·파일이
달라도 모양은 같아야 한다. ASCII 트리는 Confluence에서 정렬이 깨지므로 쓰지 않는다.

**규격은 `svg-creation` 스킬의 「폴더트리(Folder Tree) 패턴」을 그대로 따른다** —
폴더/파일 글리프, 행 간격 32 / depth당 들여쓰기 36, 가이드선, 우측 개수 칩, 하단 범례.
글리프 모양·간격·색 RGB는 거기서 단일 정의되므로 여기서 다시 만들지 말고 그 규격을 쓴다.
기준 모양은 **`references/arpath_tree_example.svg`** (Fls 정의 트리)이며, 새 트리도 이
예시와 똑같은 톤으로 복제한다.

재료는 정의용 `path_tree`, 값용 `ecuc_tree`, 정의(.epd) `ecuc_def_tree`.

**노드 색 의미 (AR-PATH 고정 매핑)** — 폴더트리 패턴의 역할별 색을 다음과 같이 배정한다.
색은 **자식 유무로만** 정한다(강조·판단 없음):

| 노드 | 글리프·색 |
| --- | --- |
| 트리 루트 (MODULE-DEF / SYSTEM / 최상위 패키지) | mint 폴더 |
| 자식(sub·ref·하위컨테이너)을 가진 컨테이너 | blue 폴더 (펼쳤든 개수만 표기했든 무조건) |
| 자식 없이 member만 있는 평면 노드(리프) | grey 파일 |

- 정의용: 패키지 → SYSTEM/ECU-INSTANCE/COMPOSITION 기둥. 개수 칩엔 자식 종류 요약(TAG×N).
- 값용: MODULE → CONTAINER → SUB. 개수 칩엔 param/ref/sub 수.
- 정의(.epd): MODULE-DEF → CONTAINER-DEF → SUB(구조체 계층). 개수 칩엔 member/ref/sub 수.
- 빈 패키지·없는 자식은 사실만 담담히 표기. 추정으로 채우지 않는다.
- 트리 아래 1~2문장으로 "무엇이 기둥인지"를 짚는다. depth 기본 3, 조정 가능.

## 블록 4. 이 파일 특유 태그 (표)

`arxml_inspect.py`의 `specific_tags`(공통 태그 제외)를 **표**로 옮긴다(속성 나열이라
표가 적합). 추출 방식은 종류 공통이나 실제 나오는 태그가 다르다.
- 정의용: `*-REF`, `*-PORT`, `*-MAPPING` 위주.
- 값용: `ECUC-*-PARAM-VALUE`, `ECUC-CONTAINER-VALUE`, `REFERENCE-VALUE` 위주.
- 설명은 `element_glossary.md`를 재활용한다.
- 표 끝에 "이 태그 분포가 곧 파일의 정체"라는 한 줄 해석을 단다(정의용=가리키고
  연결하는 파일 / 값용=값을 채워 넣은 파일).
- 태그가 많아 부담되면 줄일 수 있으나, 세부 해석에서 쓰일 수 있으므로 기본은 넉넉히.
